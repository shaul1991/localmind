# Goal: 프로젝트별 계약 저장소 — DDD 차용(컨텍스트 맵·유비쿼터스 언어) (why)

spec: [spec.md](spec.md) · plan: [plan.md](plan.md)

## Background

028이 바이브 코딩 무대에 도메인 스페셜리스트 9종과 **도메인 가이드**(`guides/{domain}.md` —
스택·컨벤션·금지사항, 완화 게이트)를 세웠다. 026·027은 디자인 무대에 **design.md 정본
위계**(텍스트 정본이 최상위, Figma·tokens.json은 파생 — "다르면 design.md가 이긴다"는
드리프트 규칙)를 세웠다. architect 정본은 이미 "bounded context·도메인 모델·**유비쿼터스
언어**"를 소유로 명시한다(templates/agents/architect.md 소유 절) — 그러나 그 유비쿼터스
언어가 적히는 **파일**은 없다.

028의 실전 검증(2026-07-05)에서 **크로스 도메인 계약의 공백**이 드러났다:

- **① API 계약이 휘발한다** — backend-dev가 API를 구현하고 요청/응답 형태·상태 코드를
  세션 보고로만 남기면, 다음 세션의 frontend-dev(또는 ios/android-dev)가 그 계약을 참조할
  정본이 없다. design.md처럼 축적되는 정본이 없다.
- **② 인프라 환경 정보가 미기획** — 서비스 URL·환경변수 이름 같은 크로스 도메인 좌표가
  적히는 곳이 없다.
- **③ 용어(유비쿼터스 언어)가 흩어진다** — 같은 개념을 도메인마다 다른 이름으로 부르고,
  architect의 소유 문구는 있으나 실체(파일·형식·갱신 규율)가 없다.

028이 도메인 축의 "일반적으로 어떻게 짓는가"(guide)를 세웠다면, 029는 프로젝트 축의
"이 프로젝트에선 무엇이 계약인가"(contract)를 세운다.

## Problem

- 바이브 코딩은 촘촘한 명세 없이 진행되므로(028 무대 정의), 크로스 도메인 좌표(API 형태·
  환경·용어)가 세션 밖으로 나가면 **다음 세션·다른 도메인 페르소나가 재구성 불가**하다 —
  매번 되묻거나 코드를 역추적한다.
- 디자인 토큰은 design.md 정본(026)으로 완결됐으나, **API 계약·환경·용어는 정본이 없다** —
  design.md가 푼 문제의 나머지 절반이 비어 있다.
- 용어 충돌(같은 단어가 컨텍스트마다 다른 의미)이 표면화·판정될 자리가 없다 — DDD가
  bounded context·유비쿼터스 언어로 푸는 고전적 문제가 방치된다.

## Objective

프로젝트별 **계약 저장소**를 규약으로 세운다 — DDD를 **실용 수준으로 차용**한다(bounded
context·컨텍스트 맵·유비쿼터스 언어 — 전면 DDD 강제 아님). 저장 위치는 028 `guides/` 관례를
재사용해 **노트 폴더 하위 `projects/{project}/`**(백업·인덱스 무플러그인 계승), 문서 4종
(컨텍스트 맵·용어집·API 계약·환경)을 둔다. **신규 페르소나·신규 코드 없이** 기존 페르소나
본문·AGENTS.md·템플릿만 최소 수술한다(028 완화 게이트에 "계약 확인"을 얹는 방식). 게이트는
두 규칙의 조합이다: **문서 부재는 완화**(028 — 없으면 진행 + 명시), **문서 존재 시 코드와의
불일치는 드리프트 결함**(026/027 design.md 규칙 계승 — 계약이 정본). architect의 기존 유비쿼터스
언어 소유를 **파일로 실체화**한다. localmind 자체 개발은 SDD(specs/)가 이미 계약 역할을
하므로 계약 저장소를 강제하지 않는다.

## Success metrics

- `templates/contracts/`에 계약 문서 템플릿 4종(context-map·ubiquitous-language·
  api-contract·environments)이 존재하고, AGENTS.md에 "프로젝트 계약 저장소" 절(구조·프로젝트
  식별·게이트 조합·시크릿 금지·`cp` 안내·localmind 예외)이 있으며, owner/consumer 페르소나
  본문 편집이 전부 repo grep AC로 고정된다.
- `environments.md` 템플릿이 **시크릿 값 금지 경고**를 담고, docs가 백업 커밋 경고(docs/agents.md
  "주의" 절 계승)를 계승한다 — grep은 보조 가드이고 **1차 방어는 사람 전수 검토**(plan 단계 2,
  거짓 안심 방지).
- loadRegistry가 **19종 problems 0 불변**(본문만 편집, frontmatter 불변). 편집된 페르소나
  description에 계약 트리거 어휘(계약·컨텍스트 맵·용어집·환경 정보)가 부재해 **라우팅 tie
  0**을 유지한다(028 어휘 서로소 계승 — 신규 트리거 미도입).
- docs/personas.md(SSoT)는 신규 페르소나가 없어 **로스터 불변**이되, 계약 소유 매트릭스를
  각주·참조로 반영한다(구성 원칙 재조정 아님 — 페르소나 수 불변).

## Non-goals

- **신규 페르소나·신규 코드** — 콘텐츠(템플릿)·규약(AGENTS.md)·기존 페르소나 본문 편집만.
  seedAgents·loadRegistry·렌더·brain.ts 무변경(028과 동일 스코프).
- **계약 자동 로딩(017 연계)** — 세션 시작 시 계약을 자동 주입하지 않는다. 명시 참조로
  시작(Open q).
- **기계 가독 형식 이행** — API 계약의 OpenAPI화, 환경의 `.env.example`화, 용어집의 구조화
  포맷은 027 tokens.json 패턴처럼 opt-in 후속(Open q). 시작은 마크다운.
- **계약 전용 백업·인덱스 배관** — `projects/`는 노트 폴더 하위라 백업·인덱스가 공짜다
  (brain.ts 제외 목록 미추가, 019 asset-mirror 미확장 — 028 정합).
- **실제 프로젝트의 계약 내용 작성 대행** — 문서 스키마·소유 매트릭스·워크플로우만.
- **런타임 자동 라우팅으로 계약 강제** — 명시 위임 + 페르소나 규율(028 계승).
- **프로젝트 아카이빙 자동화** — 완료 프로젝트 정리 규약은 Open q.
- **dba 스키마·data-platform 파이프라인 계약(5종째)** — 1차는 서비스 공개 경계(API·환경·
  용어·컨텍스트)만. DB 물리 스키마·파이프라인 출력은 각 도메인 내부 계약으로 간주 —
  크로스 도메인 참조가 반복되면 재론(Open q).

## Constraints

- 오픈소스: 개인 절대경로·특정 스택·**실제 시크릿 값** 금지(전부 플레이스홀더). 비개발자
  포함 사용자가 이해할 평이한 한국어.
- **028 guides 관례 계승**: 위치는 firstNotesDir 하위, 인덱스 포함(agents/·skills/처럼 제외
  아님 — 계약은 진짜 축적 지식), 백업 포함(git 노트 저장소). 신규 배관 없음.
- **026/027 design.md 정본 불변**: 디자인 토큰은 계약 저장소에 **복제하지 않는다** — design.md가
  정본, 계약 저장소는 그 위치를 가리키는 **포인터만** 둔다.
- **페르소나 description 불변**: 신규 트리거 어휘를 만들지 않는다(부분 문자열 함정·라우팅
  tie 회피 — 026 AC-2·028 AC-3 실증 클래스). 편집은 본문·AGENTS.md·템플릿에 한정.
- 모델(AGENTS.md 실패 파장 × 난이도): 이 스펙의 **저술**(규약·경계·소유 매트릭스) =
  최상위 티어 🔖(라우팅·소유가 여기서 갈림). 계약 문서의 **런타임 작성**은 각 owner 페르소나
  티어(architect opus, backend-dev·infra opus — 028 배정).

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 특히 여러 기술 도메인을 넘나들며 바이브
코딩하는 개발자 — 프로젝트별 계약이 세션 간·도메인 간 재설명을 없애 준다(028 도메인
가이드가 스택 재설명을 없앤 것의 프로젝트 축 대응).

## Risks

- **계약 staleness(owner가 코드 바꾸고 계약 안 고침)** → 계약의 존재 이유("다음 세션 참조
  가능")가 무너진다. 완화: 드리프트 결함 규칙(owner는 계약 변경을 코드 변경과 동시에) +
  owner 소유 규율. 코드 강제 불가(정직한 한계 — 페르소나 규율 의존).
- **프로젝트 식별 오류(cwd 매칭 실패·오매칭)** → 잘못된 프로젝트 계약 참조. 완화: 모호 시
  사용자 질문 폴백(추측 금지). 식별은 Claude 내부라 CI 미검증(정직한 한계).
- **시크릿 유출(environments.md 백업 커밋)** → 노트 폴더는 백업 저장소에 그대로 커밋된다
  (docs/agents.md "주의" 절). 완화: 템플릿 상단 경고 + AC로 고정(값 아닌 이름만). 그래도 사용자
  규율 의존 — 코드가 시크릿을 막지 못한다(정직한 한계).
- **문서 증식·유지 부담** → 4종 × 프로젝트 수. 완화: 부재=완화 게이트(강제 아님 — 실제
  필요할 때만 `cp`), 최소 4종(design.md는 포인터).
- **guides/ ↔ projects/ 혼동** → 도메인 가이드와 프로젝트 계약의 경계가 흐려짐. 완화: 2축
  (도메인 축 vs 프로젝트 축) 명문화 — 도메인 dev는 둘 다 참조("일반적으로 어떻게" + "이
  프로젝트에선 무엇이 계약").
- **계약 인덱스 오염** → environments의 env 이름·api-contract 문자열이 노트 검색에 섞임.
  완화: 028 guides 인덱스 정책 계승(포함 — 진짜 축적 지식). 검색을 오염시키면 제외 재론(Open q).
- **탁상 설계** → 계약 템플릿·소유 매트릭스·식별 규약은 첫 실전 바이브 코딩에서 개정 전제
  (028·026 Risks 계승).
