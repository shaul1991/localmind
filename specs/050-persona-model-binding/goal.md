# Goal: 페르소나/모델 바인딩 온보딩

> 모델 이력 — 작성: Fable 5 · 검토: Fable 5(critic) · 구현(예상): 미정

## Background — 배경

localmind 워크플로 스킬(goal-ready·sdd-implement·sdd-self-review 등)은 역할(아키텍트·크리틱·
워커·인터뷰어 …)과 실행 품질 등급이 조합된 절차를 오케스트레이션한다. 저장소 규약(AGENTS.md
"실행 등급 배치")은 이미 provider/model 중립 추상 — `critical-reasoning` / `standard` /
`economy` 3등급 — 을 정의하고, "구체 provider·model 매핑·가격·alias는 이 규약의 필수 계약이
아니다 — 필요하면 별도 local binding 문서(optional adapter)로 둔다"고 예고했다. 그러나 그
adapter는 아직 존재하지 않는다.

한편 localmind는 비개발자를 포함해 누구나 설치하는 오픈소스 도구이고, 설치마다 사용하는 AI
런타임(Claude Code·Codex·Gemini CLI 등)과 그 런타임에서 가용한 모델·격리 위임 능력이 전부
다르다. 지금은 이 차이를 흡수할 설정 지점이 없어, 일부 워크플로 정본에 특정 모델명과 특정
페르소나가 직접 박혀 있다.

## Problem — 문제

1. 워크플로 스킬이 특정 모델명·페르소나를 하드코딩하고 있어, 그 모델·페르소나가 없는
   설치(다른 런타임·다른 구독·비개발자 환경)에서는 지침이 그대로 적용되지 않거나 오도한다.
2. 사용자가 "내 환경에서는 이 등급에 이 모델, 이 역할에 이 페르소나"라고 정할 방법이 없다 —
   선택을 기록할 장소도, 그 선택을 스킬이 읽는 규약도 없다.
3. 하드코딩된 모델명은 시간이 지나면 낡는다(모델 세대 교체). 정본 스킬을 매번 고치는 방식은
   유지보수 비용이 크고, 비개발자 사용자는 스킬 본문을 고칠 수 없다.
4. 페르소나(격리 서브에이전트) 위임을 지원하지 않는 런타임에서 역할 지시가 실행 불가능한
   전제로 남아, 워크플로가 어색하게 끊기거나 조용히 무시된다.

## Objective — 목표

설치(런타임)마다 **최초 1회 온보딩 인터뷰**로 (a) 실행 등급 → 구체 모델, (b) 역할 → 페르소나
바인딩을 사용자가 확정하게 하고, 워크플로 스킬이 하드코딩 대신 **그 바인딩을 참조**할 수 있는
토대를 만든다. 바인딩이 없거나 런타임 능력이 부족해도 워크플로는 중단되지 않는다 — 평이한
한국어 안내 또는 명시적 fallback으로 이어진다. 이는 후속 slice(051 — 워크플로 스킬 본문의
바인딩 참조 정합)의 기반이다.

## Expected outcome — 기대 결과

- 신규 설치자가 온보딩 1회로 자기 환경(가용 모델·페르소나)에 맞는 바인딩을 갖는다 —
  AI가 등급별 추천 초안을 제시하고 사용자가 확정·수정한다.
- 워크플로 스킬이 "어떤 모델·페르소나로"를 설치별 바인딩에서 얻는다. 바인딩 미설정 상태의
  실행은 side-effect 전에 안내되고, 페르소나 미지원 런타임은 비독립(fallback)임을 밝히고
  현재 세션이 역할을 대행한다.
- 런타임(Claude·Codex·Gemini)마다 바인딩이 독립적으로 존재한다 — 서로 동기화하지 않는다.
- 모델 세대가 바뀌면 사용자가 재설정으로 바인딩만 갱신한다 — 워크플로 정본은 그대로다.

## Success metrics — 성공 지표

- [x] 바인딩이 없는 설치에서 온보딩 1회 실행으로 실행 등급 3종 → 모델, 핵심 역할 → 페르소나
      바인딩이 완성된다(추천 초안 제시 → 사용자 확정 흐름 — 시나리오 테스트 + 도그푸드로 확인).
      — **근거**: 온보딩 도그푸드(T5.1, `_bindings/<id>.json` 생성 관찰).
- [x] 바인딩 미설정 상태에서 워크플로 스킬을 실행하면 side-effect 발생 전에 평이한 한국어
      안내가 100% 표시된다(무단 진행 0건).
      — **근거**: 미설정 안내 도그푸드(T5.2, AC-3) + `resolveTier`/`resolveRole` 부재 사유 유닛.
- [x] 페르소나 위임 능력이 없는 런타임에서 워크플로가 중단 없이 완주하고, 보고에
      "비독립(fallback)"이 명시된다.
      — **근거**: fallback 정적 계약(`binding-contract.md` FR-5) + 도그푸드(T5.2, 가능 환경 한정 —
      지침수준이라 실소비는 051).
- [x] 온보딩 재실행으로 기존 바인딩을 수정할 수 있다(기존 값 표시 → 부분 수정 저장).
      — **근거**: `mergeBinding` 유닛(재설정 보존) + 재실행 도그푸드(T5.1, AC-2).
- [x] 한 런타임의 바인딩 설정·수정이 다른 런타임의 바인딩에 영향을 주지 않는다(동기화·백업
      파이프라인에 포함되지 않음).
      — **근거**: `loadBinding` 정확 일치 + traversal 가드 유닛(런타임 격리) + 백업 격리 셸 31/0.

## Non-goals — 비목표

- **워크플로 스킬 본문의 정합 개편** — goal-impl(sdd-implement) 등 스킬이 실제로 바인딩을
  참조하도록 본문을 중립화하는 작업은 후속 slice(051)의 몫이다. 050은 바인딩과 소비 규약
  (계약)까지만 만든다.
- **모델 가용성 자동 감지·프로빙** — 어떤 모델이 실제로 쓸 수 있는지는 사용자가 확정한다.
  런타임 API를 조회해 자동 검증하지 않는다.
- **모델 가격·성능 벤치마크 관리** — 추천 초안은 AI의 일반 지식 기반 참고일 뿐, 가격표·
  벤치마크를 유지하지 않는다.
- **페르소나 신규 제작·레지스트리 개편** — 기존 레지스트리를 나열·추천할 뿐이다.
- **프로젝트(저장소)별 바인딩** — 이번 범위는 설치(런타임)별 전역 바인딩만.
- **기기 간·런타임 간 동기화** — 의도적으로 하지 않는다(아래 Constraints).

## Constraints — 제약

- **실행 등급 추상 재사용**: AGENTS.md "실행 등급 배치"의 3등급(critical-reasoning /
  standard / economy)을 그대로 쓴다 — 새 등급 체계를 발명하지 않는다. 이 작업은 그 규약이
  예고한 "별도 local binding 문서(optional adapter)"의 구체화다.
- **로컬·비동기화**: 바인딩은 설치(런타임)별 로컬 설정이다. 런타임마다 가용 모델이 다르므로
  기기 간·런타임 간 동기화 대상이 아니다.
- **중단 금지**: 바인딩 부재·페르소나 미지원이 워크플로를 중단시키면 안 된다 — 안내 또는
  명시적 fallback(비독립 보고)으로 잇는다(044 capability fallback 방식과 정합).
- **오픈소스 대상**: 사용자는 설치한 개인 누구나(비개발자 포함)다. 온보딩 질문·에러·안내는
  평이한 한국어로 쓰고, 특정 인물·개인 경로를 예시로 쓰지 않는다.
- **모델명은 낡는다**: 추천 초안에 담기는 모델명은 시간에 따라 낡을 수 있는 사실이다 —
  정본은 언제나 사용자가 확정한 값이며, 추천은 참고임을 사용자에게 밝힌다.
- **배포 호환**: 기존 localmind 배포 체계(데이터 정본 + 템플릿 시드 + 3개 런타임 타깃)와
  충돌하지 않아야 한다.

## Stakeholders — 이해관계자

- **단일 사용자(설치한 개인 누구나 — 비개발자 포함)** — 온보딩으로 자기 환경을 선언하고,
  워크플로 품질(올바른 등급의 모델·역할 위임)을 얻는다.
- **localmind 유지보수자** — 모델 세대 교체 때 스킬 정본을 고치는 대신 바인딩 계약을 유지한다.
- **후속 slice 051** — 이 바인딩을 참조해 워크플로 스킬 본문을 중립화한다.

## Risks — 리스크

- **추천 초안의 낡음**: AI 지식 기반 추천 모델명이 이미 은퇴·개명됐을 수 있다 → 사용자 확정
  절차를 필수로 하고, 추천이 참고임을 고지해 완화한다.
- **온보딩 마찰**: 설정을 요구하는 문턱이 생겨 사용자가 포기할 수 있다 → 미설정 상태의
  안내·fallback 경로를 보장해 "설정 없이는 아무것도 못 하는" 상태를 만들지 않는다.
- **이중 정본 기간**: 050(바인딩)과 051(스킬 정합) 사이에 하드코딩과 바인딩이 공존해 드리프트
  위험이 있다 → 050에서 소비 규약(계약)을 명문화해 051이 기계적으로 치환하게 한다.
- **격리 개념의 오해**: "왜 런타임마다 다시 설정하나"를 사용자가 이해하지 못할 수 있다 →
  온보딩·안내 문구에서 이유(런타임마다 가용 모델이 다름)를 밝힌다.
