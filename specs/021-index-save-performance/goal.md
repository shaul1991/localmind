# 021 — 색인 저장 성능: 진행 저장 비용을 색인 크기에서 분리 (why)

## Background

재색인은 중단 안전을 위해 임베딩 배치(기본 8청크)가 끝날 때마다 색인 전체를 JSON으로
직렬화해 원자적으로 재기록한다(`saveIndex` — temp 쓰기 + rename). 이 설계는 색인이
작을 때는 문제가 없지만, 저장 비용이 **색인 크기에 비례**하므로 색인이 클수록 배치당
오버헤드가 커진다 — 배치 수 × 색인 크기 = 총 쓰기량이 O(n²)로 자란다.

020 사고 복구 중 실측(2026-07-04, Apple Silicon + 호스트 Ollama Metal):

- 1,065파일 · 8,262청크 전체 재색인에 **66분**. 임베딩 백엔드는 Metal(청크당 수십 ms)
  인데도 배치 처리율이 분당 45~49요청에 머묾 — 병목은 GPU가 아니라 저장.
- 색인 파일 113MB. 배치마다 최대 113MB를 직렬화·재기록 → 총 JSON 쓰기량 수십 GB.
  색인이 자랄수록 후반 배치가 느려지는 전형적 O(n²) 곡선.
- 배치·동시성 기본값(`BRAIN_BATCH=8`, `BRAIN_CONCURRENCY=2`)은 "Docker CPU 임베딩의
  타임아웃 cascade 방지"용 보수값 — 호스트 GPU 라우팅에서는 요청 고정 오버헤드(litellm
  프록시 홉 포함)만 키운다.

## Problem

- 진행 저장(중단-이어감)의 안전장치가 대량 색인에서 성능 병목이 된다 — 특히 기기
  전환·복구 직후의 전체 재색인(019·020의 핵심 시나리오)에서 체감이 가장 크다.
- 임베딩 라우팅(호스트 GPU vs Docker CPU)과 무관하게 CPU 보수 파라미터가 기본이라,
  빠른 백엔드를 살리지 못한다(doctor가 "10~50배 빨라진다"고 안내하는 그 구성에서).

## Objective

진행 저장의 빈도를 **경과 시간 기준으로 스로틀**해 저장 비용을 색인 크기·배치 수에서
분리한다. 중단-이어감 의미론(파일 단위 커밋, 오류 중단 시에도 완료분 보존)은 그대로
유지한다. 임베딩 라우팅이 호스트(GPU)면 배치 기본값을 진입점에서 상향한다.

## Success metrics

- **재색인의 진행 저장에 한하여**, 색인 파일 쓰기 횟수가 배치 수가 아니라 경과 시간에
  비례한다(기본 간격 + 최종 1회). 1,000파일급 전체 재색인에서 저장 쓰기량이 O(n²) →
  O(n)으로. (검색 경로의 말미 저장은 범위 밖 — spec "적용 범위" 참조.)
- 중단(오류 포함) 후 재실행이 완료·저장된 파일을 재임베딩하지 않는다 — 유실 상한이
  기존(마지막 배치)보다 나빠지지 않는다.
- 호스트 GPU 라우팅의 전체 재색인 소요가 실측으로 단축된다(도그푸드 — 66분 대비).
  *(2026-07-04 구현 시점 기준 실측 대기 — 색인 무효화 비용이 있어 다음 자연스러운
  전체 재색인 기회에 측정한다. 스텁 계측으로는 저장 1회 vs 10회 확인됨.)*

## Non-goals

- 색인 파일 포맷 변경(벡터 바이너리 사이드카, append-only 저널, 스트리밍 JSON) —
  하위호환을 깨는 큰 개편은 후속 스펙. 이번엔 저장 "빈도"만 다룬다.
- loadIndex(전체 파싱) 비용, 검색 경로 성능, Ollama 서버 자체 튜닝.
- `BRAIN_CONCURRENCY` 기본 변경 — NUM_PARALLEL=1 Ollama 큐 적체 근거가 여전히 유효.
- removeFromIndex·capture의 단건 즉시 저장 — 스로틀 대상이 아니다.

## Constraints

- 중단 안전이 성능보다 우선 — 스로틀은 "유실 상한 = 마지막 저장 이후"를 만들므로,
  오류 중단 경로에서 완료분을 저장하는 보강이 반드시 함께 간다(기존 유실 상한 유지).
- 원자적 쓰기·락·reload-merge(다중 프로세스 정합)는 그대로 둔다.
- 비개발자 사용자 — 파라미터를 몰라도 기본값으로 체감되게(설정은 선택 사항).

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 특히 대량 노트 보유자와 기기 전환
직후 사용자.

## Risks

- 스로틀 간격 동안 crash(SIGKILL·정전 등 try/finally가 못 잡는 중단)면 그 구간
  작업이 유실된다 — 재임베딩으로 자가 치유되는 파생물이므로 수용(간격 기본 10초).
- 호스트 감지 오판(잘못된 배치 상향) → 감지 규칙을 doctor와 동일 판정으로 고정하고,
  명시 env가 항상 우선.
- 큰 배치가 임베딩 요청 타임아웃을 넘길 가능성 → GPU 속도로 상쇄되지만, 문제가 되면
  BRAIN_BATCH 명시 지정으로 즉시 복귀 가능.
