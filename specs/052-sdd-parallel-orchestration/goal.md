# Goal: SDD 병렬 오케스트레이션 규약

> 모델 이력 — 작성: Fable 5 · 검토: Fable 5(critic)+codex 교차 · 구현: Opus 4.8(fan-out worker 위임)

## Background — 배경

localmind의 SDD 워크플로 스킬 두 개 — 요구를 goal/spec/plan/tasks 문서로 만드는 스킬
(goal-ready)과 문서를 구현으로 옮기는 SDD 구현 스킬(goal-impl) — 는 다단계 작업을 격리
서브에이전트로 오케스트레이션한다. 구현 스킬 규약에는 이미 "phase별 서브에이전트 위임(각자 새
컨텍스트), 메인은 조율·검증만"이 있고, tasks 설계 규약은 독립 task에 `[P]` 표기를 둔다.

이 위에서 병렬 실행은 이미 실전 검증됐다: 2026-07-17 specs/050 구현에서 서로 다른 파일만
건드리는 두 phase(에이전트 바인딩 소스 ∥ 백업 스크립트)를 메인이 동시에 spawn해 실제로 병렬
실행했다. 그러나 이것은 메인이 그때그때 눈으로 판정한 ad-hoc 관행이었다 — 052는 이 관행을
규약으로 승격한다.

## Problem — 문제

1. **무엇이 동시 실행 가능한지에 대한 명시 규약이 없다.** 병렬화 판단이 메인 세션의 즉흥
   판단에 의존해 재현 불가능하고, 새 설치·다른 세션에 교육(전수)되지 않는다.
2. **병렬 worker 간 파일 충돌이 무방비다.** 두 worker가 같은 파일을 동시에 고치면 서로의
   변경을 덮어쓸 수 있는데, 이를 막는 선언·판정 장치가 없다.
3. **tasks 산출물에 의존·파일 정보가 없다.** 현행 `[P]` 표기는 "독립"이라는 결론만 있고 그
   근거(무엇에 의존하고 어떤 파일을 건드리는지)가 없어, 메인이 병렬 가능성을 기계적으로
   판정할 수 없다.
4. **과병렬 방지 기준이 없다.** 병렬화는 공짜가 아니다 — worker 결과가 메인 컨텍스트로
   돌아오는 비용이 있어, 잔task 여러 개를 병렬화하면 오히려 손해인데 이를 제어할 규약이 없다.

## Objective — 목표

SDD 워크플로 스킬 두 개(goal-ready·goal-impl)에 **병렬 오케스트레이션 규약**을 명문화한다:
tasks 산출물이 각 task/phase의 의존(depends-on)과 건드리는 파일(files)을 선언하고, 메인은 그
선언이 이루는 의존 DAG를 읽어 "의존 충족 + 파일 disjoint + 유의미한 크기"인 노드만 동시에
spawn하며, 배리어에서 통합 검증 후 다음 레이어를 해금한다. 문서 작성 단계(goal-ready)는 하드
체인은 직렬로 유지하되 곁가지(사실수집·디자인 등)를 병렬화한다. 판단이 즉흥이 아니라 규약이
되게 하여, 어떤 세션·어떤 설치에서도 같은 병렬화가 재현되게 한다.

## Expected outcome — 기대 결과

- SDD 구현 스킬 실행 시, 파일이 겹치지 않고 의존이 풀린 phase/task들이 규약에 따라 한 메시지에
  동시 spawn되어 벽시계 시간이 줄어든다(050에서 실증된 이득의 규약화).
- tasks 산출물만 읽으면 "무엇이 동시에 돌 수 있는가"가 선언적으로 보인다 — 메인의 눈짐작이
  아니라 선언된 메타데이터가 판정 근거가 된다.
- 파일이 겹치는 노드는 규약에 의해 직렬로 떨어져, 병렬 worker 간 덮어쓰기 충돌이 구조적으로
  차단된다.
- 문서 작성 단계에서 사실수집·디자인 같은 곁가지가 본류와 병렬로 진행되되, 최종 크리틱 게이트는
  항상 모든 산출물이 모인 뒤에 온다.
- 병렬화가 손해인 상황(잔task·곁가지 없는 단순 슬라이스)에서는 규약이 병렬화를 막아, 단순한
  작업은 단순하게 남는다.

## Success metrics — 성공 지표

- [x] SDD 구현 스킬 정본에 fan-out 규칙(의존 충족 + 파일 disjoint → 동시 spawn, 배리어 통합
      검증·커밋 후 해금)이, 문서 작성 스킬 정본에 곁가지 병렬 규칙(크리틱 최종 배리어 포함)이
      명문으로 존재한다(정적 점검으로 확인).
- [x] tasks 설계 규약이 각 task/phase에 depends-on·files 선언을 요구하고, 규약 개정 후 새로
      산출되는 tasks 문서가 이 선언을 포함한다.
- [x] 도그푸드 1회: depends-on·files가 선언된 실제 slice에서, 파일 disjoint하고 의존이 풀린
      노드 2개 이상이 규약에 근거해 동시 spawn되고, 배리어에서 메인 통합 검증이 수행됨이
      관찰된다.
- [x] 같은 도그푸드에서 파일이 겹치거나 의존이 안 풀린 노드가 병렬로 spawn된 사례 0건.
- [x] 곁가지·병렬 여지가 없는 단순 작업에서 규약이 불필요한 병렬화를 유발한 사례 0건(직렬
      그대로 완주).

## Non-goals — 비목표

- **Workflow 도구 기반 결정적 오케스트레이션** — 외부 워크플로 엔진·전용 도구로 DAG를 기계
  실행하는 방식은 대규모 전용이라 이번 범위에서 기각한다. 052는 스킬 산문 규약 + tasks
  메타데이터까지다.
- **건드리는 파일의 자동 감지** — 코드 분석·정적 추적으로 files를 자동 산출하지 않는다. tasks
  설계 시점의 **선언**이 근거이고, 선언의 품질은 설계자(및 검증 배리어)가 책임진다.
- **SDD 흐름 자체의 의미 변경** — goal→spec→plan→tasks 하드 체인, TDD, self-review, 커밋
  규약 등 기존 SDD 흐름의 단계·의미는 바꾸지 않는다. 052는 그 흐름 안에서 "무엇을 동시에
  돌려도 되는가"만 규약화한다.
- **worktree 격리의 표준화·자동화** — 각 worker에게 저장소 복제본을 주고 병합하는 방식은
  옵션으로 언급만 하고, 절차 표준화·자동화는 하지 않는다.

## Constraints — 제약

- **형식화 수준은 산문 + 메타데이터**: 규칙은 스킬 정본의 산문 규약으로, 병렬화 근거는 tasks의
  depends-on·files 선언으로 둔다 — 그 이상의 기계 장치를 도입하지 않는다(사용자 인터뷰 확정).
- **양 스킬 모두 적용**: 구현 스킬(phase/task DAG fan-out)과 문서 작성 스킬(곁가지 병렬 —
  하드 체인은 직렬 유지, 크리틱은 항상 마지막 배리어) 둘 다 개정한다.
- **두 병렬 체제를 구분**: goal-impl은 **공유 코드트리**라 파일 disjoint 판정·직렬/worktree가
  필요하다. 반면 goal-ready는 각 spec이 **자기 폴더(`specs/NNN/`)** 이고 저작 페르소나가 전부
  **Read 전용(마크다운만 반환, 파일 쓰기는 메인이 직렬화)** 이라 **동시 쓰기 충돌 자체가 없다** —
  독립 슬라이스/spec 간 goal-ready는 기본 병렬 안전이며(예: 052 문서 준비 ∥ 050 구현), 한계는
  파일이 아니라 **내용·결정 의존**뿐이다(한 슬라이스의 goal→spec 체인, 슬라이스 간 결정 의존).
  따라서 파일 충돌 정책(직렬/worktree)은 주로 goal-impl 체제의 제약이다.
- **파일 충돌은 직렬화가 기본**: files가 겹치면 그냥 순차 실행한다. worktree 격리는 큰 이득이
  명확할 때만 명시적으로 택하는 옵션이다 — 조율 복잡도 최소화가 우선.
- **과병렬 금지**: genuinely 독립 + 파일 disjoint + 각 노드가 유의미한 크기일 때만 병렬화한다.
  잔task 여러 개의 병렬화는 worker 결과가 메인 컨텍스트로 돌아오는 비용 때문에 손해다.
- **선행 slice 순서**: 순서는 plan D-4에서 확정 — **051을 대기하지 않고** 현행 정본에 반영하되,
  착수 시점 051 완료 시 편집 대상만 치환한다. 문서는 스킬을 이름 무관하게 서술한다.
- **오픈소스 대상**: 사용자는 설치한 개인 누구나(비개발자 포함)다. 규약 문구는 평이하게 쓰고,
  특정 인물·개인 절대경로를 예시로 쓰지 않는다.

## Stakeholders — 이해관계자

- **단일 사용자(설치한 개인 누구나 — 비개발자 포함)** — SDD 작업의 벽시계 시간 단축과, 병렬
  실행에서도 충돌 없는 안정적 결과를 얻는다.
- **localmind 유지보수자** — 병렬화 판단이 규약·선언으로 남아 세션 간 재현·개선이 가능해진다.
- **SDD 워크플로 스킬 정본과 후속 slice** — tasks 메타데이터가 이후 오케스트레이션 개선의
  공통 어휘가 된다.

## Risks — 리스크

- **과병렬로 인한 손해**: 규약이 있어도 "유의미한 크기" 판정이 느슨하면 잔task 병렬화로 오히려
  총비용이 늘 수 있다 → 비용 가드를 규약에 명시하고, 성공 지표에 "불필요한 병렬화 0건"을 둔다.
- **병렬 worker 간 파일 충돌**: files 선언이 부정확하면(누락·과소 선언) disjoint 판정이 틀려
  덮어쓰기가 발생할 수 있다 → 직렬화 기본 원칙 + 배리어에서 메인 통합 검증으로 이중 방어한다.
- **규약 미준수(형해화)**: 산문 규약은 강제력이 약해 메인이 여전히 눈짐작으로 spawn하거나
  선언을 생략할 수 있다 → 정적 점검 가능한 명문 규칙 + tasks 선언 요구 + 도그푸드 지표로
  준수를 관찰 가능하게 만든다.
- **선행 slice 지연**: 예고된 051이 늦어지면 052 구현이 표류할 수 있다 → 문서를 스킬 이름
  무관하게 서술해 규약 내용은 051 결과와 독립이 되게 한다(순서만 의존).
