# 자주 묻는 질문 (FAQ) — 기획자·실사용자 관점

> 실제 사용자(기획자)가 localmind를 처음 검토하며 던진 질문들을 정리했습니다.
> 개념이 처음이면 [비유로 이해하기](concepts.md)부터 보시면 더 쉽습니다.

---

## 1. 개념 — 무엇이고, 왜, 누구에게

### Q. localmind가 정확히 뭔가요?
지금 쓰는 **Claude Desktop / Cursor에 "기억력"과 "내 자료를 읽는 눈"을 달아주는 부품**입니다.
새 앱을 배우는 게 아니라, 쓰던 도구에 부품 하나 끼우는 것. 대화 방식은 그대로이고, AI가 *내 일*에 대해 똑똑해집니다.

### Q. 저는 AI를 가끔만 쓰는데 꼭 필요할까요?
**사용 빈도보다 "들고 있어야 할 맥락의 양"이 기준입니다.**
- 가끔 대화만 하고 그 자리에서 끝난다 → **아직 필요 없음** (Claude Desktop으로 충분)
- **프로젝트를 여러 개** 굴리고, 요구사항·검토 등 *"전에 우리가 뭐라고 했더라"* 가 잦다 → **가치가 큼**

### Q. 어떤 사람에게 효과가 큰가요?
프로젝트 3~4개를 동시에 관리하는 기획자처럼 **맥락 전환·결정 추적·내 문서 검색**이 많은 사람.
한 번의 대화가 아니라 **"파고든 결과가 매번 증발하는"** 손실을 메워 줍니다.

---

## 2. 노트·기억 — 무엇이고 어떻게 쓰나

### Q. "노트"가 뭔가요?
**컴퓨터에 저장된 글(텍스트 파일)** 일 뿐입니다. localmind는 상자 두 개를 들고 있습니다:

| 상자 | 내용 | 예시 |
|---|---|---|
| 📒 **기억 메모**(mem0) | AI가 적어두는 짧은 사실 | "A프로젝트 결제는 OAuth로 결정" |
| 📁 **문서함**(.md 노트) | 내가 쓴 기획 문서 파일 | `요구사항.md`, `회의록.md` |

> 차이: **기억 메모**는 일하며 쌓이는 *짧은 사실*, **문서함**은 내가 쓴 *긴 문서*.

### Q. 노트는 어떻게 쌓이나요?
- **말해서**: 대화 중 "이거 기억해둬: …" → 기억 메모에 한 줄 저장
- **연결해서**: 노트 폴더(`NOTES_DIR`)를 한 번 가리키면 → 문서함에 통째로 흡수, 이후 자동

### Q. 쌓인 걸 어떻게 쓰나요?
**그냥 평소처럼 물어보면** 됩니다. localmind가 알아서 두 상자를 뒤져 근거 있는 답을 줍니다. 별도 조작 없음.

---

## 3. 신뢰·투명성 — AI가 적은 걸 어떻게 확인하나

### Q. AI가 알아서 기록한다는데, 뭘 적었는지 어떻게 믿죠?
**몰래 쌓는 구조가 아닙니다.** 적히는 모든 건 ① 내 컴퓨터 안에, ② 열어볼 수 있는 평문으로 남습니다.

- 📁 **문서함** → 그냥 폴더 열어 파일로 확인·수정·삭제
- 📒 **기억 메모** → 대화창에서 바로:
  1. **적는 순간 화면에 보임** (`remember` 호출이 대화창에 표시)
  2. **"내 기억 다 보여줘"** → 전체 목록 (`list_memories`)
  3. **"이거 지워줘"** → 항목 삭제 (`delete_memory`)
  4. **읽는 파일로 export** (`make memory-export`)

> 신뢰를 "믿어주세요"가 아니라 **"직접 확인하세요"** 로 드리는 구조. 전부 로컬, 외부 전송 없음.

## 4. 시작하기 — 설치와 연동

### Q. 처음에 뭘 하면 되나요? (1회, 약 30분)
> 아직 "버튼 하나" 앱은 아니지만, `make` 명령이 **한 단계씩 안내**해줍니다.

1. **준비물** — Docker 설치. (claude를 쓰면 토큰 1회 발급, codex만 쓰면 호스트 로그인)
2. **받아서 켜기**
   ```bash
   git clone https://github.com/shaul1991/localmind && cd localmind
   make install build
   make claude-token   # claude 구독 토큰 발급(브라우저 1회) → 안내대로 .env에 입력 (codex만 쓰면 생략)
   make up             # 스택 기동
   ```
3. **앱에 연결** — Claude Code면 **원클릭** `make mcp-install`. Cursor/Claude Desktop이면 `make mcp-config`가 출력한 JSON을 설정에 붙여넣기.
   ```bash
   make mcp-install                       # 기본 노트 폴더 ~/.localmind
   make mcp-install NOTES_DIR=/내/노트폴더  # 내 .md 노트로 바로 RAG
   ```
4. 앱 재시작 → `ask`·`remember`/`recall`·`capture_note`/`search_notes`/`ask_brain`·`list_*`/`delete_*` 도구가 생김.

이후 매일 하는 것: **그냥 대화.** 설치는 한 번뿐 — 자세한 사용은 👉 [사용법](usage.md).

### Q. Cursor에서도 되나요?
**네.** 방식은 Claude Desktop과 동일, 설정 파일만 `.cursor/mcp.json`(또는 Settings → MCP)로 다를 뿐입니다.

### Q. Cursor·Claude·Codex를 오가도 배경 설명을 다시 안 해도 되나요?
**"저장된" 맥락에 한해 그렇습니다.**

| 종류 | 도구 바꿔도 따라오나 |
|---|---|
| 기억해둔 결정 / 문서함 노트 | ✅ 따라옴 (재설명 불필요) |
| 방금 그 창에서 한 *저장 안 한* 대화 | ❌ 자동 이동 아님 |

→ 비결: **중요한 결론은 "기억해둬" 한마디나 노트로 박아두기.** 그러면 어느 도구든 살아있음.
전제: 도구마다 **같은 `NOTES_DIR` + `OPENMEMORY_USER`** 로 연결(= 같은 두뇌 공유).

## 5. 운영 — 성능과 백업

### Q. 노트가 아주 커지면(예: 100GB) 느려지나요?
**아니요. 질문마다 전부 읽지 않습니다.** 미리 만든 **색인**으로 관련된 몇 조각에만 곧장 점프합니다(도서관 카탈로그처럼).
- 꺼내오는 양이 노트 크기에 비례해 늘지 않음 → **질문 속도 거의 일정**
- 커지면 늘어나는 건 *질문 속도*가 아니라 **저장 공간 + 쌓을 때 1회 정리 시간**
- 개인 노트로 100GB는 사실상 불가(보통 수백 MB~몇 GB). 순수 텍스트 100GB ≈ 수천만 페이지
- 진짜 한계는 속도보다 **정확도**(많아지면 엉뚱한 조각이 섞일 수 있음) → 튜닝·프로젝트 분리로 관리

### Q. 컴퓨터가 바뀌거나 고장나면 백업은요?
**백업할 건 두 개(노트 + 기억), 나머지(색인·앱)는 다시 만들면 됨.**

| 대상 | 방법 |
|---|---|
| 📁 문서함(.md) | 동기화 폴더(Dropbox 등)나 git에 두기 → 자동/이력 |
| 📒 기억 | `make memory-export` → 파일로 추출 후 보관, `make memory-import`로 복원 |
| 🗂️ 색인 / ⚙️ 앱 | 백업 불필요(노트에서 재생성 / 다시 clone) |

복구: 새 컴퓨터에 설치 → 노트 복원 + `memory-import` → 색인 재생성 → 끝.

> **한 줄 명령으로**: 백업은 `make backup`(메모리+노트를 내 private repo로 커밋·푸시, 자동은 `make backup-cron`), 복구는 새 기기에서 `make recover RESTORE_REPO=<내 백업 repo>`.

### Q. GitHub만으로 백업되나요?
**네.** 노트·기억 둘 다 텍스트 파일이라 git에 딱 맞고, **변경 이력**까지 남습니다. 단:
- ⚠️ **반드시 Private 저장소** (기획 내용이 공개되면 안 됨)
- ⚠️ GitHub는 **자동이 아님** → `git push` 필요 (주기적으로 하거나 자동 예약)

### Q. 노트/기억 말고 개인 설정 파일도 같이 백업하고 싶어요(예: 다른 도구의 전역 설정)
**`BACKUP_EXTRA_FILES`로 지정하면 됩니다.** 기본은 아무 파일도 다루지 않습니다(선택 사항).

```bash
make backup BACKUP_EXTRA_FILES="~/.claude/CLAUDE.md,~/.codex/config.toml"
```

지정한 파일만 백업 저장소(`extras/` 폴더)에 함께 커밋되고, 새 기기에서 `make restore`/
`make recover` 시 원래 위치로 복원됩니다(기존 파일과 다르면 `.bak-날짜시간`으로 보존 후 교체).

- ⚠️ **백업 저장소가 Private이어도 파일 내용은 그대로 커밋됩니다** — 비밀번호·API 키 등
  민감정보가 담긴 파일은 지정하지 마세요.
- `$HOME`(내 홈 폴더) 하위 파일만 지원합니다.

## 6. 리스크 — 약관·계정 안전

### Q. 이걸 쓰면 Claude/Codex/Cursor 계정이 정지되거나 약관 위반이 되나요?
**"절대 안전"이라고 단언할 수 없습니다 — API로 재노출하는 형태는 회색지대가 맞습니다. 다만 *개인 사용*은 공식 약관이 명시적으로 허용하는 쪽입니다.**
Anthropic 공식 [Claude Code Legal & compliance](https://code.claude.com/docs/en/legal-and-compliance)는 경계를 *"어떤 도구로 감쌌나"* 가 아니라 **사용 패턴**으로 긋습니다:
- ✅ 허용: *"ordinary, individual usage of Claude Code"* — 본인 구독으로 내 머신에서 나 혼자 쓰는 것
- ❌ 금지: third-party가 *"on behalf of their users"*(타인을 대신해) 구독 자격증명으로 요청을 라우팅하는 것 → 이 경우 **API 키 인증 필수**

localmind는 구독형 CLI를 로컬 API처럼 재노출하는데, 약관과 닿는 건 이 *재노출 형태* 하나입니다.
- 단, **혼자·본인 계정·합리적 사용량**이면 "ordinary, individual usage" 범주라 명백한 위반(공유·재판매·대량 자동화·타인 라우팅)에 해당하지 않아 **위험은 낮은 편**
- Cursor는 접속 창구일 뿐이라 위험 거의 없음 — 위험은 주로 **Claude/Codex 구독 쪽**

### Q. 다운받아서 쓰는 것 자체도 문제인가요?
**아니요.** localmind 소프트웨어는 **MIT 오픈소스** — 받고 설치하는 건 문제없습니다.
위험은 *다운로드*가 아니라 *"내 구독을 API처럼 쓴다"는 사용 방식*에 있습니다.

### Q. 약관 위험을 낮추는 안전한 사용 패턴은?

**✅ DO** (= 공식 약관의 *"ordinary, individual usage"*)
- 본인 계정으로 본인만 / 팀이라도 각자 자기 구독
- 사람 속도로 대화형 사용 · localhost·비공개 유지 · 개인 생산성 용도

**❌ DON'T** (= *"on behalf of their users"* 라우팅 → API 키로 전환해야 함)
- 계정 공유 · 접근 재판매 · 봇/스케줄러 대량 호출 · rate limit 우회 · 공개 서비스 백엔드 · 타인 요청을 내 구독으로 처리

**💡 핵심 인사이트** — 약관과 닿는 건 **"모델 호출(claude/codex 추론)" 하나뿐**입니다.
기억·노트·임베딩은 전부 **로컬이라 약관 위험 0**.
→ 가장 보수적인 길: **백엔드만 정식 유료 API로 교체**(`base_url`/키만 변경)하면 *종량 요금은 내지만 회색지대는 0*, 두뇌 기능(기억·노트·하루 일과)은 그대로.

**🛡️ 안전망**: 각 제공사 약관 주기적 확인([Claude Code Legal & compliance](https://code.claude.com/docs/en/legal-and-compliance) — 정책은 변경 가능) · 노트/기억 백업 유지 · 정식 API로 갈아탈 길 열어두기
(데이터는 계정과 분리된 로컬 평문이라, 계정에 문제가 생겨도 **지식은 안전**)
