# Goal: Index Durability & Performance (인덱스 원자성·캐싱·동시성)

## Background — 배경

2026-07-01 품질·보안 평가에서 **높음(H2)** 및 **보통(M4)** 으로 지목된 발견들이다.
second-brain 인덱스(`.brain-index.json`)는 "파일이 정본, 인덱스는 파생" 철학의 파생물이지만,
실사용 vault에서 76MB까지 커진 것이 관찰됐다. 현재 구조는 이 크기에서 두 가지 문제를 만든다.

## Problem — 문제

1. **매 조회마다 전체 파싱**: `ensureIndexed()`가 호출될 때마다(모든 `search_notes`/
   `ask_brain`/`note_links`/watcher 이벤트) 인덱스 전체를 `readFileSync`+`JSON.parse`한다.
   76MB 기준 조회마다 수백 ms급 낭비이며 vault가 클수록 악화된다.
2. **비원자적 쓰기**: `saveIndex()`가 temp 파일+rename 없이 직접 덮어쓴다. 쓰는 도중
   프로세스가 죽으면 인덱스가 깨지고, 자가복구(버전 체크 실패 → 빈 인덱스)는 되지만
   **전체 vault 재임베딩**이라는 큰 비용이 든다(수천 청크 × CPU 임베딩 1~4s/청크).
3. **동시 실행 경합**: watcher 이벤트와 MCP 도구 호출이 동시에 `ensureIndexed()`를 부르면
   각자 인덱스 사본을 만들어 마지막 저장이 상대의 변경을 덮어쓴다. 다음 실행에서
   자가치유되지만 그 사이 임베딩 중복 수행(낭비)과 순간적 불일치가 발생한다.

## Objective — 목표

인덱스 조회를 인메모리 캐시로 상수화하고, 쓰기를 원자적으로 만들며, 동시 인덱싱을
직렬화해 — vault가 커져도 검색 지연·재임베딩 사고·중복 작업이 발생하지 않게 한다.

## Expected outcome — 기대 결과

- 인덱스 파일이 변경되지 않았으면 두 번째 조회부터 디스크 읽기·JSON 파싱이 발생하지 않는다.
- 인덱스 저장 중 프로세스가 죽어도 기존 인덱스 파일은 온전하다(전체 재임베딩 사고 없음).
- `ensureIndexed()`가 동시에 여러 번 호출되면 실제 인덱싱은 한 번만 수행되고 나머지는
  그 결과를 공유한다.

## Success metrics — 성공 지표

- 변경 없는 vault에서 연속 `search_notes` 호출 시 2회차부터 인덱스 로드 시간 ≈ 0
  (파일 stat 확인만).
- 저장 중단 시나리오(쓰기 도중 kill)에서 기존 인덱스가 파싱 가능 상태로 유지됨.
- 동시 `ensureIndexed()` N회 호출 시 실제 파일 스캔·임베딩 파이프라인 실행 횟수 = 1.

## Non-goals — 비목표

- 인덱스 포맷 변경(JSON → Float32 바이너리)은 범위 밖 — BACKLOG B2의 별도 트리거
  ("수만 청크 초과 시") 유지. 이번엔 접근 패턴만 고친다.
- 전용 벡터 DB(pgvector 등)로의 이전은 범위 밖.
- 다중 프로세스 간 잠금(MCP 서버 + reindex 스크립트가 동시에 도는 경우의 파일 락)은
  범위 밖 — 인프로세스 직렬화만 다룬다(다중 프로세스 케이스는 Open questions로 명시).

## Constraints — 제약

- 기존 인덱스 파일 포맷·`INDEX_VERSION`(3)을 유지한다 — 사용자 재인덱싱 유발 금지.
- `brain.ts`의 기존 export 시그니처(searchNotes/capture/noteLinks 등)를 바꾸지 않는다.
- stdout 미사용 등 기존 MCP 규약 유지.
- 캐시는 인메모리만 — 새 파일·의존성 추가 금지.

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함) — 대형 vault에서도 빠르고 안전한 검색

## Risks — 리스크

- 캐시 무효화 버그: 외부 프로세스(reindex 스크립트, 다른 MCP 인스턴스)가 인덱스 파일을
  바꿨는데 캐시가 낡은 데이터를 반환할 위험 — 파일 mtime/size 확인으로 완화.
- rename 원자성은 같은 파일시스템 내에서만 보장 — temp 파일을 인덱스와 같은 디렉토리에
  만들어야 함(다른 볼륨이면 rename이 copy가 됨).
- 직렬화(mutex)로 인한 지연: 대형 재인덱싱 중 검색이 대기하게 됨 — 기존에도 임베딩이
  끝나야 검색이 정확했으므로 실질 악화는 아니나, 행 방지를 위한 타임아웃 고려 필요.
