# 023 — 색인 포맷 확장: 벡터 바이너리 사이드카 (why)

spec: [spec.md](spec.md) · plan: [plan.md](plan.md)

> 초안은 라벨↔경로 바인딩 트랙을 함께 담았으나, 적대 리뷰에서 "바인딩은 선택 필드
> 추가라 버전 파괴가 불필요(결합은 편의지 필연이 아님)"가 실증되어 **024로 분리**했다
> (2026-07-04 사용자 확정). 이 스펙은 버전 파괴를 실제로 강제하는 사이드카 +
> 마이그레이션만 다룬다. 바인딩: [specs/024](../024-label-path-binding/goal.md).

## Background

색인(`src/brain.ts`)은 파생물이다 — 노트(.md)가 정본이고 색인은 언제든 재유도 가능하다.
현재 색인 포맷(`INDEX_VERSION = 4`)은 단일 JSON 파일 `.brain-index.json`이며 구조는:

- `BrainIndex { version, embeddingModel?, dims?, files: Record<key, FileEntry> }`
  (key = `label/relpath`)
- `FileEntry { hash, folder, chunks: IndexedChunk[], linksOut }`
- `IndexedChunk { path, text, vector: number[] }`
  — `vector`는 임베딩(bge-m3 1024차원)을 **JSON 숫자 배열**로 인라인 저장한다.

2026-07-04 실측(Apple Silicon + 호스트 Ollama Metal): 1,065파일·8,262청크 색인의 파일
크기가 **113MB**이고, 그 대부분을 청크당 1,024개짜리 숫자 배열(텍스트로 직렬화된 float)이
차지한다. JSON 숫자 배열은 (a) 파일 크기를 부풀리고, (b) `loadIndex`의 `JSON.parse`를 무겁게
하며, (c) `saveIndex`의 `JSON.stringify`를 무겁게 한다. 021은 저장 **빈도**를 스로틀했지만
저장·로드 **1회당 비용**(색인 크기 비례)은 손대지 않았고, "벡터 바이너리 사이드카로 분리"를
Open question 2로 남겼다 — 그것이 이 스펙이다.

## Problem

- **P1 (JSON 벡터 지배)**: 벡터를 JSON 숫자 배열로 인라인 저장해 파일이 비대하고
  로드 파싱·저장 직렬화 비용이 색인 크기에 비례한다. 기기 전환·복구 직후 전체 색인의
  체감이 가장 크다(019·020·021의 공통 시나리오).
- **P2 (마이그레이션 비용)**: 포맷을 바꾸면 기존 관례("버전 불일치 = 전량 재빌드")는
  1,065파일 재임베딩(실측 수 분~수십 분, 저사양 기기 수 시간)을 부른다 — 벡터가 이미
  v4 JSON에 존재하는데도 재계산하는 낭비.

## Objective

1. **O1 — 벡터 바이너리 사이드카**: 청크 벡터를 JSON에서 분리해 바이너리 사이드카(Float32
   연속 + 헤더)로 저장한다. 로드 파싱·파일 크기·저장 직렬화 비용을 낮추되, JSON·사이드카
   2파일의 **원자성과 정합**(락·reload-merge·다중 프로세스)을 그대로 유지하고, 부분 손상은
   재임베딩으로 자가 치유한다.
2. **O2 — 무재임베딩 마이그레이션**: 기존 v4 색인을 임베딩 호출 없이 v5로 무손실 변환한다.
   변환 불가(손상)만 전량 재빌드로 폴백한다.

## Success metrics

측정은 **동일 노트 코퍼스**에 대해 스텁이 아닌 실 임베딩으로 1회 전체 색인 후 비교한다
(도그푸드 — 색인 무효화 비용이 있어 다음 자연스러운 전체 재색인 기회에 측정).

- **파일 크기**: v5의 `.brain-index.json` 바이트 < v4 바이트 (기대 대폭↓ — 벡터가 JSON에서
  사라짐). 측정: 같은 코퍼스 색인 후 `wc -c .brain-index.json`. (사이드카 바이너리 크기 =
  `헤더 + 청크수 × dims × 4`바이트를 함께 보고.)
- **로드 파싱 시간(콜드)**: `cachedIndex` 무효화 후 `loadIndex` 1회 wall-clock. v5가 v4보다
  짧을 것(기대 수 배↓ — 대형 숫자 배열 파싱 제거). 측정법을 plan에 고정.
- **저장 직렬화 시간**: 동일 색인 `saveIndex` 1회 wall-clock(사이드카 write 포함).
- **마이그레이션 재임베딩 0건**: v4 색인을 v5로 로드·저장할 때 임베딩 호출 0(스텁 계측
  `calls()==0`으로 결정적 검증).
- **자가 치유**: 사이드카 부재·손상 시 영향 파일만(또는 전량) 재임베딩되고 이후 검색 정상.
- **회귀 0**: 기존 스위트(`npm test` · `LOCALMIND_INTEGRATION=1 npm test`) green.

성능 배수 주장은 모두 "기대"로 표기한다 — 스텁(4차원)으로는 방향성만 확인되고, 배수는
실측(도그푸드)으로만 확정한다.

## Non-goals

- **검색 알고리즘 변경**: 인메모리 cosine·스코어링·folder 스코프 필터는 불변. 인메모리
  `BrainIndex`는 여전히 벡터를 보유한다(사이드카는 **디스크 인코딩만** 바꾼다).
- **외부 벡터 DB 도입**(pgvector·ANN 인덱스 등) — 로컬 파일 + 인메모리 코사인 유지.
- **라벨↔경로 바인딩** — 024로 분리(additive 필드라 버전 파괴 불필요).
- **022 범위** — 무변경 저장 생략(dirty 플래그)·doctor 정렬. 별도 스펙(022).
- **청크 `text`의 별도 파일 분리** — 텍스트는 검색 결과 표시·사람 열람에 쓰이므로 JSON에
  유지(다음 지배 요인이면 후속 검토, Open questions).
- **append-only 저널·스트리밍 JSON** — 이번엔 generation 사이드카 전량 재작성으로 충분
  (append-only는 Open questions).
- **`BRAIN_CONCURRENCY`/`BRAIN_BATCH` 기본 변경**, 임베딩 파이프라인·모델 변경.

## Constraints

- **하위호환 파괴 급 변경 — 롤백을 정직하게 다룬다.** v5는 `INDEX_VERSION`을 올린다.
  사용자가 구버전(v4) 코드로 롤백하면, v4 코드는 v5 색인을 "버전 불일치"로 보고 기존
  관례대로 **전량 재빌드**(재임베딩)해 v4 포맷으로 자가 치유한다 — **데이터 유실 없음**
  (노트가 정본), 비용은 1회 재임베딩. 이때 v5 사이드카 파일은 고아로 남는다(무해 —
  수동 정리 안내). 즉 롤백은 안전하되 재임베딩 비용을 지불한다. (즉시 롤백용 v4 백업
  보존은 Open questions — 디스크 2배 일시 점유 vs 재임베딩 회피 트레이드오프.)
- **원자성**: JSON과 사이드카 2파일의 rename 2회는 원자적이지 않다. 이를 "JSON이
  사이드카를 단방향 참조 + JSON rename만이 커밋점"으로 해소한다(spec FR-2). 락·
  reload-merge·다중 프로세스 정합은 그대로 두되 사이드카 로드를 포함한다.
- **BRAIN_INDEX 관례 확장**: 사이드카·temp·lock은 모두 `INDEX_PATH` 인접에 둔다
  (`${INDEX_PATH}.vec-<gen>` 등). 색인 파일이 노트 폴더 안에 있을 수 있으므로, 사이드카도
  git 추적·백업 시드에서 **기존 `.brain-index.json`과 동일하게 제외**한다.
- 비개발자도 사용자다 — 마이그레이션·자가치유·롤백 안내는 평이한 한국어로, 해결 명령을
  함께 제시한다. brain 모듈은 stdout에 아무것도 쓰지 않는다(MCP 전용) — 안내는 stderr
  또는 재색인 API 반환값(`scripts/reindex.ts` 출력)으로만.

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 특히 대량 노트 보유자, 기기 전환·복구
직후 사용자.

## Risks

- **다중 버전 혼재 스래싱**: 업그레이드 과도기에 구버전 MCP(v4)와 신버전 MCP(v5)가 같은
  색인 파일에 동시에 붙으면 서로 상대 포맷으로 재빌드해 재임베딩이 반복될 수 있다 —
  데이터 유실은 없으나 낭비. 완화: 업그레이드 후 MCP 클라이언트(Claude Desktop/Code/
  Cursor) 재시작 안내 + 업그레이드 후 `make reindex` 권장. 코드 완화 여부는 Open questions.
- **롤백 재임베딩 비용**: 위 Constraints — 파생물이라 수용. 선택적 v4 백업으로 즉시
  롤백은 Open questions.
- **사이드카/JSON 불일치(부분 손상·강제종료)**: 자가 치유(재임베딩) 비용이 든다. 헤더
  (매직·dims·count) 검증으로 조기 감지하고 영향 범위를 파일 단위로 국소화한다.
- **엔디안 가정**: 사이드카는 Float32 little-endian. 대상 플랫폼(macOS/Linux/Windows,
  x64/arm64)은 전부 LE — 헤더에 표식을 두고 불일치 시 자가 치유한다.
- **generation 사이드카 누수**: GC 실패로 옛 `.vec-*`가 남아도 무해(다음 저장에서 정리,
  디스크 소량 낭비). JSON이 참조하는 것만 유효.
- **reader-GC 경합**: `loadIndex`는 성능상 락을 잡지 않아, JSON을 읽고 사이드카를 읽는
  사이에 다른 프로세스의 저장·GC가 끼어들 수 있다(v4 단일 파일에는 없던 경합) — GC의
  직전 세대 유예(keep=2)와 로드 재시도(FR-3)로 흡수한다. 이를 자가 치유로 오판하면 전량
  재임베딩이 유발되므로 양성 경합 구분이 스펙 요건이다(spec FR-2·3, AC-3b).
- **stale 락 강제 해제 + 동시 저장**: 락이 stale로 강제 해제돼 두 프로세스가 동시에
  저장하면, 서로의 GC가 상대가 방금 커밋한 사이드카를 지워 dangling이 생길 수 있다
  (드묾 — 10초+ 행 전제). keep=2 유예가 창을 줄이고, 남는 잔여 위험은 자가 치유
  (재임베딩)로 수렴 — 데이터 유실 없음.
