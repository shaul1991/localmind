# sdd-implement — SDD 구현 워크플로

준비된 `specs/{NNN}-*/`의 goal/spec/plan을 TDD로 구현하고, 필수 self-review와 문서 체크를 거쳐
clean 완료까지 조율한다. 완료 규칙(TDD·surgical change·self-review·evidence·commit/push/CI)의 최종
정본은 저장소 **`AGENTS.md`**다. 이 워크플로는 그 규약을 읽고 조율할 뿐 별도 정본을 복제하지 않는다.

역할은 능력(capability)으로 표현한다. 격리된 리뷰 능력이 있으면 우선 쓰고, 없으면 현재 세션이 같은
체크리스트를 수행하되 그 사실을 보고에 밝힌다. 특정 공급자·모델·도구 이름을 완료 조건으로 삼지 않는다.

## 1. 활성화 판정 — side effect 전 권한 확인

이 워크플로는 코드 변경·commit·push·CI까지 갈 수 있으므로 **실행 권한(execution grant)** 없이는
어떤 side effect도 내지 않는다. 첫 동작으로 다음을 판정한다.

- **런타임이 명시 호출을 보증(provenance)하고** 사용자가 넘긴 원인자(raw arguments)가 **전체가 정확히
  3자리 숫자**(`^[0-9]{3}$`)이면 실행 권한이다.
- 런타임이 그 보증을 제공하지 못하면(provenance 부재), **side effect가 있는 어떤 동작도 하지 않고**
  해당 3자리 번호와 함께 일회용 확인 문구(challenge)만 보낸 뒤 멈춘다. **바로 다음 사용자 턴이** 그
  확인 문구와 번호를 정확히 일치시켜 응답한 경우에만 실행 권한으로 전환한다.
- 인용문·부정문·설명/리뷰 전용 요청, 추가·복수 인자, 지난 턴의 낡은/재사용된 확인 문구는 권한이
  아니다. 프롬프트에 명령 문자열이나 생성된 요청 텍스트가 있다는 사실 자체도 권한이 아니다.

이 확인 절차는 워크플로 행동 지침이다. 런타임에 실행 전 hook이 없으면 이를 기술적으로 강제된
"도구 호출 0회 보장"이라고 과장하지 않고, 지침 수준(instruction-level)의 가드임을 보고에 밝힌다.
런타임이 provenance도, 이 새 확인 턴도 제공할 수 없으면 이 워크플로는 중단한다.

## 2. AGENTS.md를 정본으로 읽기

실행 권한을 얻으면 저장소 `AGENTS.md`를 최우선 정본으로 읽는다. `AGENTS.md`가 없거나 요구한 번호의
goal/spec/plan 세 문서 중 하나라도 없으면 구현 전에 멈추고 평이한 한국어로 사유를 보고한다.

## 3. TDD — 실패 테스트부터

plan의 단계, spec의 FR/AC, goal의 의도를 따라 **AC별 실패 테스트를 먼저** 쓰고 최소 구현으로 통과시킨다.

## 4. 외과적 변경

요청 범위 밖의 리팩터·포매팅을 하지 않고, 사용자의 무관한 변경을 보존한다.

## 5. 필수 self-review

구현·검증 후 적대적 self-review를 반드시 수행한다. 격리된 리뷰어 능력이 있으면 우선 사용하고, 없으면
현재 세션 fallback임을 명시한다. 결함이 있으면 수정 후 다시 검토한다(clean까지).

## 6. clean일 때만 문서 체크

치명·중대 결함 0 + 테스트 green + AC 전부 충족이 확인된 뒤에만 goal의 Success metrics, spec의 FR/AC,
plan의 단계·테스트 전략 항목에 `[x]`와 검증 근거를 표기한다. 미충족 항목은 체크하지 않고 사유를 부기한다.

## 7. 완료(commit/push/CI)는 AGENTS.md 규칙대로

실행 권한을 얻은 두 경로(런타임 보증 명시 호출, 또는 바로 앞 턴의 새 확인)는 **동일한 권한**이므로
완료 시 `AGENTS.md`가 정의한 commit/push/CI 규칙을 똑같이 따른다. 실행 권한 없이 들어온 일반 자연어
구현 요청은 이 자동 side-effect 권한을 만들지 않는다. 이 워크플로가 `AGENTS.md`보다 약하거나 강한
완료 규칙을 따로 두지 않는다.

## 8. 지속 목표(goal) 기능은 선택

런타임에 지속적 목표 유지 기능이 있으면 활용할 수 있으나, 이 워크플로의 필수 전제는 아니다.

## 정직한 보고

없는 문서, 무관한 dirty 파일, 환경 제약으로 못 돌린 테스트는 숨기지 않고 사실대로 보고한다.
