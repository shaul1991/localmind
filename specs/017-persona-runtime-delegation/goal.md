# Goal: Persona Runtime Delegation — 지식 워크플로우의 페르소나 자동 위임

## Background — 배경

specs/016으로 페르소나 에이전트 정의(역할·지침·모델)를 정본으로 관리하고 외부 도구
(Claude Code·Codex)로 배포할 수 있게 됐다. 그러나 localmind **자신의 런타임**은 여전히
페르소나를 모른다 — 답변 합성은 고정된 기본 모델·내장 프롬프트로 돌고, 캡처는 분류 없이
쌓이며, 검색 품질 로그는 사람이 직접 리포트를 돌려야 읽힌다.

페르소나 구성 설계(2026-07-03 인터뷰, docs/personas.md)에서 지식 워크플로우의 개입
4지점과 **최대 자동화** 원칙이 확정됐다. 이 작업은 3단계 로드맵(레지스트리 → 런타임
위임 → 오케스트레이션)의 두 번째 단계다.

## Problem — 문제

- localmind가 "직접 다 처리"한다: 답변 합성의 지침·모델이 코드에 박혀 있어, 사용자가
  페르소나(사서)를 정의해도 자신의 두뇌가 그것을 쓰지 않는다.
- 답변의 품질을 아무도 검증하지 않는다: 출처와 안 맞는 문장(hallucination)이 그대로
  사용자에게 간다 — 같은 모델 계열만 쓰면 맹점도 공유된다.
- 캡처가 정리되지 않은 채 쌓인다: 분류·태그가 없어 나중에 찾기 어렵다.
- 검색 품질 데이터(실패 질의 로그)가 해석되지 않는다: 리포트는 수동이고, 결과는 두뇌
  밖(터미널)에 머물러 시스템이 자신의 상태를 기억하지 못한다.

## Objective — 목표

localmind 런타임이 레지스트리의 페르소나에게 일을 **자동으로 위임**하게 한다:
답변 합성은 사서가, 답변 검증은 크리틱(교차 모델)이, 캡처 분류·태깅은 큐레이터가,
검색 품질 해석은 분석가가 맡는다. 자동화는 기본 켜짐(최대 자동화)이되, 각 지점을
끌 수 있고 지연·실패가 본래 기능을 해치지 않는 **통제 장치**를 갖춘다.

## Expected outcome — 기대 결과

- 페르소나 정의를 고치면 localmind의 답변 스타일·모델이 즉시 바뀐다(정의가 곧 동작).
- 답변마다 교차 모델 검증이 붙어, 출처 없는 주장이 경고와 함께 표시된다.
- 캡처가 태그를 갖고 쌓여 검색·정리가 쉬워진다.
- 검색 품질 해석이 주기적으로 노트로 남아, 두뇌가 자기 상태를 스스로 기억하고
  recall로 회수할 수 있다.

## Success metrics — 성공 지표

- 사서 페르소나의 모델·지침 변경이 재시작 없이 다음 호출부터 ask_brain 동작에 반영된다.
- 출처에 없는 내용을 포함한 답변에 검증 경고가 표시된다(테스트로 재현).
- 페르소나·레지스트리가 없거나 검증·태깅이 실패해도 본래 기능(답변·캡처)은 항상 성공한다.
- 각 자동화 지점을 환경변수 하나로 끌 수 있다.
- 주기 리포트가 노트로 저장되어 recall 검색에 걸린다.

## Non-goals — 비목표

- **SDD 워크플로우 오케스트레이션**(단계 팬아웃, Codex exec 위임 스킬) — 018.
- **페르소나 구성 변경**(새 페르소나 추가·모델 재배치) — docs/personas.md 소관.
- **레지스트리 스키마 확장**(노트 폴더 scope 필드 등) — 이번 4지점에 불필요하면 더하지 않는다.
- **ask(교차 상담) 도구의 페르소나화** — 임의 모델 지정이 이미 가능하므로 이번 범위 아님.
- **비개발자용 설정 토글(자연어로 "검증 꺼줘")** — 이번엔 환경변수 + 문서 안내까지.
  MCP 설정 도구는 효용이 확인되면 후속 스펙.
- **검증 실패 시 답변 차단·재합성** — 경고 부착으로 확정(2026-07-03 인터뷰). 재합성
  루프는 효용이 확인되면 별도 스펙.

## Constraints — 제약

- 자동화보다 본래 기능이 우선한다: 위임 대상이 없거나 실패·지연되면 해당 자동화만
  조용히 생략하고 본래 동작을 완수한다(페르소나 없이 설치한 사용자도 정상 사용).
- 지연 통제: 자동 검증은 시간 상한을 갖고, 초과 시 답변을 지연시키지 않는다.
- 개입은 관측 가능해야 한다: 어떤 페르소나가 무엇을 했는지 응답·산출물에 드러난다(§8 도그푸드).
- 추가 요금 없음: 모든 위임 호출은 기존 게이트웨이(구독 CLI) 경유 — 메터드 API 금지.
- 비개발자가 이해할 수 있는 표시·메시지.

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함). 페르소나를 정의한 사용자는 위임
  효과를, 정의하지 않은 사용자는 기존과 동일한 동작을 얻는다.

## Risks — 리스크

- **지연 누적**: 검증·태깅이 매 호출에 붙으면 체감 속도가 나빠진다 → 시간 상한 +
  지점별 끄기 + 실패 시 생략으로 통제한다.
- **구독 쿼터 잠식**: 검증은 매 답변에 LLM 호출을 더한다 — 정액제라 돈은 안 들지만
  사용량 한도를 본업(코딩)과 나눠 쓴다(추론 강도 high는 사소한 질문에도 수만 토큰을
  씀이 실측됨) → 자동 검증의 추론 강도 강등 + 일일 횟수 상한으로 통제한다.
- **검증 오판**: 크리틱이 맞는 답변에 경고를 붙일 수 있다 → 차단이 아닌 경고 부착이므로
  판단은 사용자에게 남는다. 경고 문구에 "추정"임을 명시한다.
- **태그 오염**: 큐레이터가 일관성 없는 태그를 달면 정리가 오히려 어려워진다 → 기존
  태그 어휘를 우선 재사용하게 하고, 파일이 정본이라 수정이 쉽다.
- **리포트에 질의 원문 포함**: 리포트 노트에 검색 질의가 실린다(로컬 전용 로그의 파생) —
  백업 저장소에 커밋되므로 문서에 명시한다(기존 백업 민감정보 경고와 동일 계열).
