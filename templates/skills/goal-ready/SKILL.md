# goal-ready — SDD 문서 준비 워크플로

개방형 요구를 조사·정의해 `specs/{timestamp}-{slug}/`의 goal/spec/plan/tasks 네 문서를 준비하고 사용자
확인을 받는다. 이 워크플로는 **문서까지만** 만든다 — 구현은 확인 뒤 별도 워크플로(SDD 구현)가 맡는다.

용어는 역할(role)과 능력(capability)으로 쓴다. 런타임이 제공하는 도구의 구체 이름에 의존하지 않는다.
어떤 역할을 격리된 별도 컨텍스트로 위임할 능력이 있으면 그것을 우선 쓰고, 없으면 현재 세션이 같은
역할 체크리스트를 수행하되 그 사실을 보고에 밝힌다. 사용자와의 대화는 항상 현재 세션이 맡는다.

## 1. 활성화 판정 — 문서 준비 의도 확인

현재 사용자가 문서 준비를 **명시적으로 호출**했거나 **분명히 요청**했는지 먼저 확인한다.
이름을 인용·부정·설명·리뷰 맥락에서 언급했거나 의도가 모호하면, **파일·도구를 전혀 건드리지 않고**
무엇을 준비하면 되는지 되묻고 확인 전에는 진행하지 않는다.

## 2. 맥락 읽기

진행이 확실해지면 저장소의 SDD 규약(`AGENTS.md`), 기존 `specs/`, 관련된 과거 기록을 먼저 읽는다.

## 3. 사실 검증(Live-Verify)을 초안보다 먼저

시점에 따라 변할 수 있는 외부 사실(외부 API·SDK 거동, 가격·모델·버전·한도, 표준 등)이 필요한지
분류한다. 필요하면 **문서 초안을 쓰기 전에** 공식 문서를 실시간으로 조회해 확인한다. 실시간으로
확인할 수 없으면 단정하지 말고 Open question + 검증 과제로 남긴다.

## 4. 개방형 결정 공간 설계

결정이 필요한 개방형 요구이면 인터뷰어(interviewer) 역할로 질문 공간을 설계하고 현재 세션이
사용자에게 직접 묻는다. 요구가 단순해 이 단계를 생략하면 그 이유를 보고한다.

## 5. 트레이드오프 결정 기록

사용자에게 트레이드오프 결정을 받으면 저장소의 결정 로그(decision-log) 규약을 읽고, 사용 가능한
지속적 기록 능력으로 질문·선택지·선택·근거·관련 spec 포인터를 남긴다. 그런 능력이 없으면 문서와
보고에 결정 근거를 남기고, 지속 기록을 하지 못한 사유를 밝힌다. 특정 도구 이름을 전제하지 않는다.

## 6. 네 문서 작성

생성 시점 timestamp(`YYYYMMDDHHmm`) 프리픽스로 `specs/{timestamp}-{slug}/`에 goal.md·spec.md·plan.md·tasks.md를 쓴다. (2026-07-17 개정: 3자리 일련번호는 동시작업 충돌로 폐기 — 기존 NNN- 폴더는 레거시 유지.)

**기존 spec을 덮어쓰지 않는다** — `mkdir -p`가 아니라 **`mkdir`(`-p` 금지)** 로 만든다. 같은 경로가 이미 있으면 EEXIST로 실패하므로("있는지 확인 후 생성"의 경쟁 창 없이) 병렬 세션의 goal/spec/plan을 덮을 수 없다. EEXIST면 **현재 시각을 다시 읽어** 초까지 확장(`YYYYMMDDHHmmss`)해 재시도한다 — 시각이 진행하므로 곧 빈 경로에 도달한다(같은 경로 재확인은 무한 반복 — 반드시 시각을 다시 읽는다). 단 `mkdir`은 **경로**(프리픽스+슬러그) 충돌만 막는다 — 같은 분·다른 슬러그면 프리픽스가 같아도 둘 다 성공하므로 프리픽스는 유일하지 않을 수 있다(goal-impl의 프리픽스 매칭이 모호성을 처리한다).

- goal.md — 왜(Background·Problem·Objective·Success metrics·Non-goals·Constraints·Stakeholders·Risks)
- spec.md — 무엇을(FR은 goal 항목 지지, Acceptance Criteria는 Given-When-Then으로 테스트와 1:1)
- plan.md — 어떻게(도메인 경계, 영향 모듈, 단계, 테스트 전략). 테스트 전략에는 **verification
  matrix**를 두고 `| AC | 검증 방법·레벨 | 최소 evidence | 통과·종료 조건 | 상태 |`의 5열을
  사용한다. spec의 **모든 AC는 정확히 한 행**에 대응해야 하며 각 셀은 실행 가능하게 채운다.
  필수 검증 capability가 없거나 `skipped/degraded`라면 green으로 쓰지 않고 Open question 또는
  미충족으로 표면화한다. 이 matrix는 사용자 확인 대상이며 구현 워크플로가 첫 dogfood 직전에
  readiness를 재확인하고 freeze할 입력이다.
- tasks.md — 무엇을 실행(plan을 구체 task로 분해). 각 phase는 `goal-impl`의
  `references/tasks-format.md` 선언 문법(phase 헤더 직하 `depends-on:`·`files:`)을 포함해,
  구현 스킬의 fan-out 판정이 병렬 여부를 읽을 수 있게 한다.

버전은 여기서 정하지 않는다 — 저장소 릴리스 규약(AGENTS.md 버전·릴리스 절) 참조. 작업 시작
시점에는 변경 내용 서술만 누적한다.

## 7. UI 범위이면 디자인 게이트

웹/앱 UI 작업이 범위에 들어가면 디자인 사전 정의(design.md) 게이트를 포함한다.

## 8. 프로젝트 SSoT 정합성(조건부)

컨텍스트 맵·유비쿼터스 언어·아키텍처 헌법·ADR 같은 계약 문서가 **실제로 존재하는** 프로젝트에서만
관련 정합성을 검사한다. 없는 문서를 필수라고 추측하지 않는다.

## 9. 역할 위임

아키텍트·리서처·디자이너 역할은 이름이 붙은 특정 페르소나가 아니라 **역할 + 과제 + 기대 산출물**로
위임할 수 있어야 한다. 격리 위임 능력이 없으면 현재 세션이 같은 역할을 수행한다.

## 10. 곁가지·슬라이스 병렬

병렬은 문서 작성 워크플로에도 적용된다. 두 층위를 구분한다.

**(a) 한 슬라이스 안** — 하드 체인(goal→spec→plan→tasks)은 항상 직렬로 유지한다. 뒤 문서는 앞
문서를 전제하므로 순서를 건너뛰지 않는다. 이 체인을 막지 않는 **곁가지**는 병렬로 위임할 수
있다: 사실수집(researcher) 조사를 goal/spec 초안 작성과 동시에 진행, design.md 정의를 plan
작성과 동시에 진행, 서로 독립적인 리서치 질문 N개를 동시에 위임. **크리틱(critic)은 항상 모든
곁가지 산출물이 모인 뒤의 마지막 배리어**다 — 곁가지가 끝나기 전에 크리틱을 앞당기지 않는다.

**(b) 독립 슬라이스·spec 간** — 각 spec은 자기 폴더(`specs/{timestamp}-{slug}/`)에만 쓰고, 저작을 맡는 역할은
모두 **Read 전용**이다(마크다운을 반환할 뿐 파일에 직접 쓰지 않는다 — 파일 쓰기는 현재 세션이
직렬로 수행한다). 폴더가 겹치지 않고 동시 쓰기 자체가 없으므로 **여러 문서 준비 작업을 기본적으로
동시에 진행할 수 있다**. 한계는 파일 충돌이 아니라 **내용·결정 의존**뿐이다(예: 뒤 슬라이스가 앞
슬라이스의 결정을 전제한다면 그 의존만큼은 직렬로 대기한다).

**두 체제의 구분(중요)** — SDD 구현 워크플로는 여러 워커가 **공유 코드트리**를 같이 건드릴 수
있어 파일이 겹치지 않는지 판정하고 필요하면 직렬화하거나 격리된 작업공간을 쓰는 절차가 필요하다.
문서 준비 워크플로는 폴더가 겹치지 않고 저작이 Read 전용이라 그 파일 충돌 절차가 **대체로
적용되지 않는다** — 여기서 병렬의 한계는 파일이 아니라 내용·결정 의존이다.

## 11. 크리틱 검토

크리틱(critic) 역할이 FR-goal 추적성, AC-테스트 매핑, 엣지 케이스, 미검증 사실, SSoT 정합성,
Non-goals·Constraints·Open questions를 적대적으로 점검한다("결함을 찾으러 간다"). 격리된 리뷰어
능력이 없으면 현재 세션이 같은 체크리스트를 수행하되 이를 독립(independent) 검토라고 부르지 않는다.

## 12. 결함 수정·재검

발견한 결함을 모아 수정하고 다시 검토한다(clean까지). 이 clean-loop는 **문서 준비 critic**의
품질 게이트이며, 구현 완료 뒤 self-review의 automatic round budget과 섞지 않는다.

## 13. 보고·확인 요청

네 문서 경로, 결정, Open questions, 각 역할과 fallback 수행 내역, verification matrix의 AC 전수
매핑 결과, 그리고 **현재 런타임에서 이어서 SDD 구현을 시작하는 정확한 호출 방법**을 보고한 뒤
사용자 확인을 명시적으로 요청한다. 수정 요청을 받으면 문서 반영 → 크리틱 재검 → 재확인 루프를
반복한다.

tasks.md의 tracked checkbox는 최종 versioned commit 전에 닫을 수 있는 구현·검증·publish handoff
준비까지만 만든다. push 뒤 PR 번호·CI 상태·run ID 같은 동적 외부 상태를 `[x]`로 되쓰기 위한
post-push checkbox나 status-only follow-up commit 계획은 만들지 않는다. 상세 형식은 구현 스킬의
`references/tasks-format.md`를 따른다.

## 14. 확인 전 구현 금지

사용자 확인 전에는 구현하지 않는다. 문서 준비 자체는 자동 커밋 대상이 아니므로 사용자 요청 없이
commit/push하지 않는다.
