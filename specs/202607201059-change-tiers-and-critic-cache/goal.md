# Goal: 작업 크기에 맞게 SDD 의식(ceremony)을 줄인다 — 티어링 + critic 재독 비용 절감

> **TL;DR** — 지금 SDD 흐름은 한 줄 수정이든 신규 도메인이든 동일한 풀 코스(goal/spec/plan/tasks
> 4문서 + 적대적 critic)를 강제해, 작은 변경에서 문서 작성과 critic 콜드리드(cold read)가 과하다.
> 이를 **변경 등급 3티어**(트리비얼/작음/실질적)로 나눠 문서·critic 강도를 크기에 맞춰 right-size하고,
> critic의 재독 비용을 **verification matrix를 읽기범위 축소(SSoT) + 라운드 간 무효화 재검(cache)**
> 으로 낮춘다. **품질 게이트는 낮추지 않는다** — 캐싱은 검증 *깊이*가 아니라 재독 *넓이*만 줄인다.
>
> **누가/언제** — LocalMind의 SDD 문서 준비(goal-ready)·구현(goal-impl)·검수(sdd-self-review)
> 워크플로를 쓰는 설치 사용자·유지보수자가, 작은 변경에서 의식 비용이 변경 자체보다 커질 때 적용한다.

## Background — 배경

LocalMind의 기본 작업 흐름은 SDD다: 모든 변경이 `goal → spec → plan → tasks` 문서로 시작하고
(`goal-ready`), 구현은 TDD + 도그푸드 + 적대적 self-review로 닫는다(`goal-impl`). 이 강한 규율은
의도 정렬·추적성·회귀 방지에 효과적이었고, 최근 `202607181125-bounded-goal-impl-verification`이
self-review 자동 재검을 2라운드로 제한하고 verification matrix 동결·base freshness gate·외부 완료
상태 SSoT를 도입해 "검증 꼬리가 무한히 길어지는" 문제를 이미 막았다.

그러나 그 규율은 **작업 크기에 따라 줄어들지(scale down) 않는다**. 오타 하나를 고치든 신규
바운디드 컨텍스트를 도입하든 진입에서 4문서를 요구하고, 구현 후 격리 critic이 돈다. 규약에
`트리비얼/메타 예외`가 있으나 트리거가 흐릿해 실제로는 매번 풀 코스로 흐른다.

사용자는 2026-07-20 세션에서 두 비용을 구체적으로 지목했다:
① 작은 수정에도 **goal/spec/plan/tasks 4문서를 다 만드는 것**,
② self-review 시 격리 critic이 **맥락 없이 처음부터 전체를 다시 읽는(cold read) 재독 비용**.
이어 개선 방향으로 **(a) 등급 티어 신설 + (b) critic의 재독을 캐싱/SSoT 문서로 경량화**를
채택했다.

## Problem — 문제

1. **진입 의식이 크기와 무관하게 고정이다.** 국소적 소변경에도 4문서를 강제해, 문서 작성 시간이
   변경 자체보다 커진다. 기존 트리비얼 예외는 트리거가 모호해 신뢰 있게 적용되지 않는다.
2. **critic 재독(cold read)에 조사 지도가 없다.** 현행 self-review 계약은 이미 대상을 `FR/AC + diff +
   테스트 근거`로 스코프한다(전체 프로젝트를 읽지는 않는다). 그러나 그 diff 안에서 "각 AC가 어느
   코드·evidence에 대응하는지"를 가리키는 지도가 없어, critic이 대응 관계를 매번 스스로 재구성한다.
3. **round 간 재검이 이전 라운드 결과를 재활용하지 못한다.** round 2 격리 리뷰어는 round 1이 이미
   독립 검증한, 이번 수정과 무관한 부분까지 처음부터 다시 콜드리드한다.
4. **의식 축소를 품질 저하로 오해할 위험.** 티어·캐싱이 "작으니 대충"으로 오독되면 필요한 검증이
   조용히 생략될 수 있다(202607181125의 "상한을 품질 면제로 오해"와 같은 종류의 위험).

## Objective — 목표

- **O-1 — 변경 등급 티어:** 진입에서 변경을 **트리비얼(Tier 0) / 작음(Tier 1) / 실질적(Tier 2)**
  로 분류하는 **객관적 트리거**를 정의하고, 각 티어가 요구하는 문서·critic 강도를 명문화한다.
  판정 불확실 시 상위 티어로 올리는(escalate-on-doubt) 보수 규칙을 포함한다.
- **O-2 — critic 조사 지도 제공(matrix-as-map):** 현행 계약이 이미 diff로 스코프한 위에, verification
  matrix 행을 **AC↔코드·evidence 대응 지도**로 제공해 critic이 그 대응을 매번 스스로 재구성하지
  않게 한다(실제 신규 델타는 "전체 프로젝트 읽기 제거"가 아니라 "이 지도"다). 독립성은 보존한다 —
  matrix는 "어디를 보라"만 정하고, 해당 행은 critic이 실제 코드로 검증한다.
- **O-3 — 라운드 간 재구성 비용 절감(독립성 저하 없이) [보수형 확정]:** self-review는 라운드마다
  **모든 matrix 행을 전량 재검증**하되, frozen matrix의 map(AC↔코드·evidence 대응)을 재사용해 그
  대응을 매번 재구성하는 비용만 없앤다. **verdict 승계·행 스킵은 하지 않는다** — per-round 독립성을
  완전히 보존한다(OQ-5 사용자 결정: 보수형). round-to-round 무효화-스킵(적극형)은 이 slice에서
  도입하지 않는다.
- **O-4 — map 재사용 범위 한정:** matrix map의 재사용은 **한 goal-impl 실행 내(within-run)** 으로만
  허용하고, **세션·실행 간(cross-session) 재사용은 금지**한다(base·의존성 이동으로 map이 낡을 수 있다).
- **O-5 — 품질 게이트 보존:** Tier 2의 기존 규율(전 AC green·필수 도그푸드·격리 적대 critic·2라운드
  상한·PR 게이트·Live-Verify)은 그대로 유지한다. 캐싱은 검증 *깊이*를 줄이지 않는다. 티어 다운그레이드
  가 필요한 신호(신규 도메인·계약·보안·마이그레이션)가 나타나면 상위 티어로 승격한다.
- **O-6 — 사용자 대면 문서 정합:** 티어링은 "언제 사용자가 문서를 써야 하는가"를 바꾸는 사용자
  대면 흐름 변경이므로, 비개발자도 티어 lane을 발견·이해할 수 있게 사람용 문서(`docs/`)를 함께
  갱신한다(202607181125가 흐름 변경 시 사람용 문서 갱신을 요구한 선례와 AGENTS.md의 "비개발자도
  사용자" 원칙 준수).

## Expected outcome — 기대 결과

- 진입에서 티어가 먼저 판정·기록되고, Tier 0은 문서·critic 없이, Tier 1은 경량 단일 문서 + in-session
  단일 라운드로, Tier 2는 현행 풀 코스로 흐른다.
- 작은 변경에서 goal/spec/plan/tasks 4문서 강제와 격리 critic 2라운드 콜드리드가 사라진다.
- critic이 repo 전체가 아니라 diff + matrix 행만 필수로 읽으므로, 같은 변경에 대한 재독 시간이
  프로젝트가 커져도 늘지 않는다.
- round 2 격리 리뷰어가 frozen matrix map을 물려받아 AC↔코드 대응을 재구성하지 않으므로, 전량
  재검증을 유지하면서도 콜드리드 재구성 비용이 준다(독립성은 그대로).
- 티어·캐싱을 도입해도 신규 도메인·계약·보안 변경은 Tier 2 격리 critic으로 남아 회귀·자기확증
  방지가 약해지지 않는다.
- 비개발자가 `docs/`만 읽어도 세 티어와 각 lane의 문서·검증 요구를 이해할 수 있다.

## Success metrics — 성공 지표

- **SM-1:** 세 티어의 객관적 트리거(무엇이 Tier 0/1/2인가)와 각 티어별 문서·critic 요구가 규약
  정본(AGENTS.md)과 워크플로 계약(goal-ready·goal-impl·sdd-self-review)에서 같은 의미로 고정되고,
  판정 불확실 시 상위 티어로 escalate하는 규칙이 명문화된다. 검증: 계약 텍스트 대조 테스트 + 티어
  분류 예시(경계 케이스 포함) 시나리오 테스트.
- **SM-2:** critic 프롬프트/계약이 필수 읽기 대상을 diff + matrix 행으로 스코프하되, 각 행을 실제
  코드로 검증하라(독립성 보존)는 지시를 함께 담는다. 검증: sdd-self-review·critic 계약 텍스트 검사 +
  "matrix만 믿고 통과 금지" 문구 존재 확인.
- **SM-3 [보수형]:** self-review가 라운드마다 **전량 재검증**하고 map만 재사용한다(verdict 승계·행
  스킵 없음, per-round 독립성 보존)는 규칙과 within-run map 재사용·cross-session 금지가 계약으로
  존재한다. 검증: 계약 텍스트 대조(AC-11·12·13). (round-to-round 무효화-스킵은 미도입.)
- **SM-4:** 티어·캐싱 도입 후에도 202607181125의 확정(2라운드 자동 상한, matrix 동결, base freshness,
  외부 완료 SSoT)과 Tier 2 품질 규율이 **문구·의미상 약화되지 않는다**. 검증: 기존 bounded-verification
  계약 테스트 전부 green 유지 + 회귀 대조.
- **SM-5:** 배포 파이프라인(templates/skills 정본 → 데이터 폴더 → Claude/Codex/Gemini)이 변경된
  워크플로 계약을 드리프트 없이 실어 나른다. 검증: skills-deploy 멱등 배포 테스트 green + 계약 드리프트 0.
- **SM-6:** 전체 테스트·빌드가 green이고 기존 goal-ready/goal-impl/self-review 동작이 회귀하지 않는다.
  검증: 전체 스위트 green + 도그푸드(각 티어 1건씩 실제 흐름 관찰).

## Non-goals — 비목표

- self-review 자동 재검 상한(2라운드)·matrix 동결·base freshness gate·외부 완료 상태 SSoT를
  **재설계하지 않는다** — 202607181125가 정본이며 이 slice는 그 위에 티어·캐싱을 얹는다.
- Tier 2의 검증 강도(TDD·필수 도그푸드·격리 적대 critic·PR 게이트·Live-Verify)를 낮추지 않는다.
- **세션·실행 간(cross-session) critic 캐싱을 만들지 않는다** — base·의존성 이동으로 무효화 불가.
- 티어를 **자동으로 라우팅하는 분류기·엔진**을 새로 구현하지 않는다 — 트리거는 규약 텍스트로
  판정하고 워크플로가 읽는다(기존 fan-out 판정이 tasks 선언을 읽는 방식과 동일한 결).
- 시간 상한·evidence 줄 수·transcript 크기 같은 수치 예산을 추가하지 않는다.
- Tier 2 **문서 준비 단계(goal-ready)의 critic clean-loop 자체에 라운드 상한을 부여하지 않는다**
  (202607181125 Non-goal 존중) — Tier 1은 애초에 경량 단일 라운드라 이 문제를 크기로 회피한다.
- 기존 완료된 spec·감사 artifact를 다시 손대지 않는다.

## Constraints — 제약

- **C-1 — 사용자 확정 두 축:** (a) 변경 등급 티어링, (b) verification matrix를 SSoT(읽기 축소) +
  cache(라운드 간 무효화)로 이중 활용 — 둘 다 누락 없이 반영한다.
- **C-2 — 가드레일 2개 필수 [보수형]:** ① 독립성(map은 조사 지도일 뿐 검증 대체 불가, 각 행은
  실제 코드로 재검증, 라운드마다 전량) ② map 재사용 범위(within-run 한정, cross-session 금지).
  (적극형의 무효화 건전성 가드레일은 무효화-스킵을 도입하지 않으므로 이 slice 범위 밖.)
- **C-3 — SSoT 경계:** 티어 정의·워크플로 행동·critic 계약·verification matrix 형식은 각각 하나의
  정본을 갖고 소비 surface(AGENTS.md, 새 프로젝트 기본, 배포된 skill) 사이 의미 드리프트를 허용하지
  않는다.
- **C-4 — 외과적 규약 변경:** 관련 정책·계약·사용자 설명만 최소 변경한다. 새 실행 backend·분류
  엔진·배포 분기를 도입하지 않는다.
- **C-5 — 202601181125 정합:** 이번 변경은 그 slice의 확정과 충돌하지 않고 참조·확장한다. 상충하는
  문구를 만들면 결함이다.
- **C-6 — provider/model 중립:** 계약은 역할·능력·추상 실행 등급으로만 서술하고 구체 provider/model
  ID에 의존하지 않는다.
- **C-7 — no-work-without-doc 정합:** Tier 0(무문서)은 기존 트리비얼 예외의 트리거를 명확화하는
  것이지 새 구멍이 아니다. Tier 1은 경량이나 문서가 존재한다.

## Stakeholders — 이해관계자

- **단일 사용자(설치한 개인 누구나 — 비개발자 포함):** 작은 변경을 빠르게 끝내면서도 큰 변경의
  품질 게이트는 그대로 얻는다.
- **LocalMind 유지보수자:** critic 콜드리드·문서 의식으로 인한 불필요한 작업을 줄인다.
- **SDD 문서 준비·구현·검수 워크플로 소비자:** 진입부터 같은 티어 트리거와 critic 캐싱 semantics를
  쓴다.
- **새 프로젝트 사용자:** 새로 만든 SDD repo에서도 동일한 티어·캐싱 규칙을 얻는다.

## Risks — 리스크

- **R-1 — 티어 오분류(하향):** 실질 변경을 작음으로 잘못 분류해 필요한 격리 critic을 건너뛴다 →
  객관적 하드 트리거(신규 도메인·계약·보안·마이그레이션·데이터 모델 = 무조건 Tier 2)와
  escalate-on-doubt를 규약·테스트로 고정한다.
- **R-2 [보수형으로 완화] — 미변경 파일 파손 누락:** 적극형이었다면 파일 X 수정이 미변경 파일 Y를
  깨뜨렸는데 Y 행을 승계로 넘겨 놓칠 수 있었다 → **보수형은 라운드마다 전량 재검증하므로 이 위험이
  구조적으로 제거**된다. (map 재사용은 검증 대상이 아니라 조사 지도일 뿐이라 파손 은폐에 관여하지
  않는다.)
- **R-3 — matrix 도장찍기:** critic이 matrix 주장만 믿고 코드를 안 봐 결함을 통과시킨다 → "행마다
  실제 코드 검증, matrix만으로 통과 금지"를 critic 계약에 명문화한다.
- **R-4 — 티어가 품질 면제로 오독:** "작으니 검증 생략"으로 흐른다 → Tier 1도 TDD·in-session 적대
  자기검증을 요구하고, 무문서는 Tier 0(무행동변경/기계적)에만 허용한다.
- **R-5 — 정본 간 드리프트:** 티어·캐싱 규칙이 AGENTS.md와 배포된 skill에서 달라진다 → 계약 텍스트
  대조·드리프트 0 테스트로 고정한다.
