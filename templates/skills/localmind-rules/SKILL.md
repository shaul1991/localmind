# /localmind-rules — 거버넌스 규칙 저작·관리

localmind 규칙 정본(base + 프로젝트 overlay)을 **즉흥적 봉합이 아니라 구조화된 절차로** 추가/제거/편집한다. 본질은
interview-protocol의 검증 게이트를 **규칙 변경**에 재귀 적용하는 것 — 기존 규칙과 부딪히면 조용히
봉합하지 말고 사용자에게 올린다.

## 불변식 (절대 — 어기면 실패)

- **I-1 게이트는 write·deploy *이전*에 발화한다.** 충돌/고아를 감지하고도 조용히 적용 금지.
- **I-2 충돌 자동 해소 금지.** 조정안을 네가 정하지 말고 조정 선택을 **사용자에게** 올린다.
- **I-3 링크 파싱은 위키 `[[name]]` *와* 마크다운 `](name.md)` 둘 다 + `## 관련` 항목.** (코퍼스 다수가 마크다운형.)
- **I-4 base vs overlay 를 *매번* 묻는다.** 추측·기본값 금지.
- **I-5 배포는 managed 산출물만 건드린다.** 사용자 파일 불가침(rules-deploy 불변식 보존).
- **I-6 규칙 로드·중복 검사는 기존 레지스트리 규약을 따른다.** 파싱을 재발명하지 않는다.
- **I-7 중복 name 은 조용히 덮지 않는다.** add *전* 파일명/name 존재 확인(같은 경로 write=원본 소실 방지) + 저작 후 재확인.
- **I-8 배포 분기 정확**: base→글로벌(`--no-repo`), overlay→대상 repo 안에서.
- **I-9 인터뷰 5단계는 게이트 유무와 무관하게 *매* 변경에 강제.** 재진술·검수 전 write/deploy 금지.

## 규칙 정본 위치·포맷

- 폴더: `LOCALMIND_RULES_DIR`가 있으면 그것, 없으면 `<노트폴더>/rules`(이 기기 기본 `~/.localmind/rules`).
  `base/*.md`(전역) · `overlays/<project>/*.md`(프로젝트, 충돌 시 overlay 승).
- 규칙 1개 = `.md` 1개. **name = frontmatter `name:` 또는 파일명(basename), kebab-case 필수**
  (`^[a-z0-9]+(-[a-z0-9]+)*$`). frontmatter는 선택(대부분 산문); 있으면 `name`·`order`만 의미.
- 본문 관례: `# 제목` · `## Why` · `## How to apply` · `## 관련`(다른 규칙으로 링크).

## 흐름 (5단계 인터뷰 매핑 — I-9)

### 1) 로드 (변경 제안 *전*)
`base/*.md` + `overlays/*/*.md`를 모두 Glob·Read 한다. name·본문·상호 링크를 파악한다(I-6).

### 2) 의도 파악 → 재진술 (질문→답→재진술→검수)
사용자가 무엇을 하려는지 확정한다: **add / remove / edit**, 대상 규칙, 그 규칙의 목적·맥락.
네 이해를 **자기 말로 재진술**하고 검수를 받는다. 검수 전 다음 단계로 가지 않는다(I-9).

### 2.5) 초안 (파일 미기록)
새/편집 규칙의 **본문 초안**을 잡는다 — 아직 파일에 쓰지 않는다. 이 구체 초안으로 §3 게이트를 건다
(의도 수준이 아니라 실체로 충돌·고아를 판정).

### 3) 충돌·고아 게이트 (I-1·I-2)
- **add/edit → 의미 충돌 검사**: 로드한 전 규칙과 새/편집 규칙이 **모순**되는지 *의미로* 판단한다
  (키워드 아닌 내용). 예: "hotfix는 먼저 고치고 사후 문서화"는 `no-work-without-doc`(변경 전 문서
  필수)와 충돌. 충돌이면 **멈추고** 게이트를 올린다:
  ```
  ⚠️ 충돌: <새 규칙>이 <기존 규칙 name>과(와) 모순됩니다.
    <어떻게 모순되는지 1~2줄>
  조정 선택:
    (a) 기존 규칙에 예외 절 추가   (b) overlay로 프로젝트 한정 오버라이드
    (c) 새 규칙 재서술            (d) 취소
  ```
  사용자가 고르기 전엔 **어떤 파일도 쓰지 않는다**(I-1·I-2).
- **remove/edit(개명) → 고아 검사**: 대상 규칙 name을 **다른 규칙들이 링크**하는지 전수 검사한다 —
  `[[name]]`, `](name.md)`, `## 관련` 항목 **셋 다**(I-3). 의존이 있으면 게이트로 목록을 제시하고
  확인/재연결 전엔 진행하지 않는다.
- **충돌·고아 없으면** 헛된 게이트 없이 다음으로(오탐 금지).

### 4) 배치 → 저작 (검수 후)
- **배치를 매번 묻는다**(I-4): `base`(모든 프로젝트) vs `overlays/<project>`(이 프로젝트 한정).
- **add 전 이름/경로 충돌 검사(필수, I-7)**: 대상 파일명(또는 frontmatter `name`)이 §1에서 로드한 기존
  규칙과 겹치면 **write 금지** — 같은 경로 write는 원본을 조용히 덮어써 소실시키고, 사후 `loadRules`
  검사로는 못 잡는다(basename 하나면 중복 problem 미발생). 겹치면 멈추고 표면화(다른 name 제안 또는 edit로 전환).
- 규칙을 위 포맷대로 `.md`로 쓴다(kebab name, Why/How/관련). add=신규 파일(위 검사 통과 후), edit=대상 파일
  수정, remove=파일 삭제(+ 게이트에서 합의한 재연결 반영).

### 5) 구조 검사 → 배포 제안
- **구조 검사**(I-7): 저작 후 같은 layer에 **중복 name이 없는지** 확인한다. 있으면 덮지 말고 표면화.
  (레지스트리는 중복 name을 problems로 격리하고 그 layer의 prune을 건너뛴다 — 조용한 사고를 막는 안전밸브.)
- **배포 제안**(I-8): 변경을 정본에 반영하려면 배포가 필요하다고 안내하고 실행을 제안한다:
  - base 변경 → `npm run rules:deploy -- --no-repo` (Claude·Codex 글로벌).
  - overlay 변경 → **대상 프로젝트 repo 안에서** `make rules-deploy`(또는 `npm run rules:deploy`).
    현재 cwd가 그 repo가 아니면 그 사실을 안내한다. 또한 **overlay 프로젝트명이 대상 repo 폴더명과
    (kebab 정규화) 일치해야 배포**되며, 불일치면 조용히 건너뛰므로 그 조건을 안내한다.
  - 배포는 managed 섹션/파일만 건드린다(I-5) — 사용자 파일은 불변.

## 참고
- 규칙 저작 자체가 이 저장소의 거버넌스를 바꾸는 변경이므로, 위 5단계와 게이트를 생략하지 않는다.
- 관련 규칙: `no-work-without-doc` · `interview-protocol` · `deep-interview-elicitation`.
