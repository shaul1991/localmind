# 022 — 색인 쓰기·진단 위생: 무변경 저장 생략 + doctor 관측 정렬 (why)

## Background

localmind의 색인(`src/brain.ts`)은 두 축에서 "실제와 어긋난" 비용·표시를 남긴다.

**(1) 무변경에도 색인 전량을 쓴다.** `doEnsureIndexed`는 스캔·프루닝을 마친 뒤 말미에
항상 `saveIndex(idx)`를 부른다(brain.ts 말미). 그런데 `ensureIndexed`는 검색·캡처·링크
조회마다 호출되고(brain.ts의 다수 호출부), `saveIndex`는 색인 전체를 JSON으로 직렬화해
원자적으로 재기록한다. 즉 **신규 임베딩·프루닝·삭제가 하나도 없는 검색 한 번이 색인 전량
쓰기 1회를 유발**한다.

021 도그푸드 실측(2026-07-04): 색인 파일 113MB(1,065파일·8,262청크). 이 규모에서 매 검색이
무변경에도 113MB를 직렬화·재기록한다 — 021이 재색인의 "진행 저장"을 시간 스로틀로 줄였지만,
검색·캡처가 유발하는 **말미 저장 1회는 021 범위 밖으로 남겨졌다**(021 spec "적용 범위" 절,
021 Open questions 4가 이 스펙의 출발점).

**(2) doctor의 관측이 실제 판정과 어긋난다.** `scripts/doctor.sh`의 라우팅 판정은
`.env`·`litellm.config.yaml`를 raw grep한다 — 주석·예시 라인의 host URL에도 반응해
`.env.example` 기본 구성(주석에 host URL, 유효값은 Docker)의 CPU 사용자에게 "호스트(빠름)"를
오표시한다(021 Open questions 5에서 실증). 021은 재색인 진입점(reindex.sh)의 같은 판정을
`read_env_val OLLAMA_API_BASE` 유효값 검사로 이미 고쳤으나, doctor는 정렬되지 않았다. 또한
`litellm.config.yaml`는 모든 `api_base`가 `os.environ/OLLAMA_API_BASE` 참조라 리터럴
host URL이 존재할 수 없는 **죽은 가지**다(실측 확인).

**(3) doctor가 고아·부재 라벨을 못 보여준다.** 020은 재색인 요약(`ReindexSummary`)에
고아 라벨(설정에 없는데 색인에 남은 라벨)·부재 라벨(등록됐지만 폴더를 열 수 없는 라벨)
정보를 담았지만, 이는 **명시적 재색인(make reindex)을 돌려야만** 보인다. 진단 전용 읽기
명령인 doctor에서는 확인할 수 없다(020 Open questions 4).

## Problem

- 검색 경로의 무변경 말미 저장은 대량 색인에서 순수 낭비다 — 읽기여야 할 검색이 매번 색인
  전량을 다시 쓴다(디스크 I/O·SSD 마모·다중 프로세스 락 경합). 유발 빈도가 재색인보다
  훨씬 잦다(검색마다).
- doctor는 사용자가 "내 임베딩이 지금 어디서 도는가"를 확인하는 첫 창구인데, 오표시가
  나면 최적화 안내 자체가 틀린다 — 진단의 신뢰가 깨진다.
- 020의 프루닝 가드가 "보존한" 고아·부재 라벨을 사용자가 doctor에서 확인할 길이 없어,
  make reindex를 돌리기 전에는 색인 상태를 관측할 수 없다.

## Objective

**관측이 정확하고, 쓰기가 낭비되지 않게 한다** — 두 갈래로.

1. **쓰기 생략**: 색인 갱신 경로에서 변경(신규 임베딩·프루닝·삭제)이 하나도 없으면 말미
   저장을 생략한다(dirty 플래그). 변경이 있으면 기존과 정확히 동일하게 저장한다.
2. **관측 정렬**: doctor의 라우팅 판정을 021 FR-3(reindex.sh)과 동일한
   `read_env_val OLLAMA_API_BASE` 유효값 판정으로 바꾸고, 020이 보존한 고아·부재 라벨을
   doctor에서 읽기 전용으로 보여준다. 라벨 분류 판정은 TS 단일 소스를 유지한다(셸 재구현
   금지 — 019 원칙).

## Success metrics

- **무변경 색인 갱신(검색·캡처 경유 및 무변경 재색인)의 색인 파일 쓰기가 0회**가 된다.
  변경(신규·프루닝·삭제)이 있으면 저장 횟수·내용은 기존과 동일하다(회귀 0). 테스트 저장
  카운터(`_saveRunCountForTest`)로 결정적으로 검증한다.
- doctor의 "현재 라우팅" 표시가 유효 `OLLAMA_API_BASE`와 일치한다 — **주석·예시 라인의
  host URL로 인한 오탐 0**(`.env.example` 기본 구성에서 "Docker"로 정확히 표시).
- 고아·부재 라벨이 있으면 doctor가 안내하고(고아엔 정리 명령, 부재엔 "폴더 없음"만),
  색인이 없거나 조회 불가하면 조용히 생략하며 exit 0을 유지한다(오탐 금지).

## Non-goals

- **색인 파일 포맷 변경**(벡터 바이너리 사이드카·append-only 저널·스트리밍 JSON) — 023으로
  분리한다. 이번엔 "무변경 시 쓸지 말지"만 다루고 "무엇을·어떻게 직렬화하는가"는 안 건드린다.
- `removeFromIndex`·capture의 단건 즉시 저장 — dirty 게이트 대상이 아니다(단건 경로는 항상
  변경이 있어 저장, 불변).
- 성능 파라미터 튜닝(`BRAIN_BATCH`·`BRAIN_CONCURRENCY`·`BRAIN_SAVE_INTERVAL` 기본값).
- `loadIndex`의 전체 파싱 비용·검색 경로 자체의 속도(021 Non-goals 계승) — 이번은 "쓰기
  생략"과 "표시 정렬"만.
- 021 진행 저장 스로틀·finally 보강 로직 변경(그대로 둔다 — dirty 게이트는 성공 경로 말미
  저장에만 얹는다).

## Constraints

- 중단 안전·정합성이 성능보다 우선 — dirty 판정이 틀려 "변경을 안 썼다"가 되면 안 된다.
  판정은 보수적으로: **애매하면 저장한다**(잘못된 생략 < 불필요한 저장 1회).
- 021의 진행 저장(중단-이어감)·finally 오류 보강·원자적 쓰기·락·reload-merge는 불변(FR-5).
- doctor는 읽기 전용이며 아무것도 바꾸지 않는다 — 라벨 조회가 재색인·임베딩을 유발해서는
  안 되고, exit 0을 유지한다.
- 라벨 분류 판정은 TS 단일 소스(019 "판정 단일 소스 = TS") — 셸에서 재구현하지 않는다.
- 비개발자도 사용자다 — doctor 안내·경고 문구는 평이한 한국어로, 해결 명령을 함께 안내한다.

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 특히 대량 노트 보유자(검색당 색인 전량
쓰기의 낭비가 가장 큼)와, doctor로 임베딩 구성을 점검하는 CPU/호스트 혼용 사용자.

## Risks

- dirty 판정 누락 → 실제 변경을 저장하지 않아 유실. 상쇄: 판정을 "파일 엔트리(idx.files)의
  추가·삭제"라는 관측 가능한 사실에만 묶고, 임베딩이 커밋되면(pending>0) 무조건 dirty로
  본다. 애매한 스탬프(embeddingModel/dims)-only 변화는 Open questions에서 다룬다.
- 021 성공 경로 저장을 dirty 게이트가 잘못 없애 회귀 → 021 스위트(AC-1·2·4) green 유지를
  회귀 게이트로 고정(FR-5·AC-5).
- doctor 라벨 조회가 113MB 색인을 파싱해 느려질 가능성 → 진단은 검색처럼 잦지 않은 수동
  명령이고, 색인 없음·파싱 실패·node 부재 시 조용히 생략한다. 파싱 비용의 상한·경량
  추출은 Open questions로 정직 표기.
- doctor가 넘긴 NOTES_DIR와 brain의 FOLDERS 파싱이 갈라지면 고아/부재 분류가 틀림 →
  doctor는 셸 단일 소스(`resolve_notes_dir`)로 해석한 NOTES_DIR 문자열을 TS 리더에 그대로
  넘기고, 분류는 TS(brain FOLDERS 파서)에서만 한다.
