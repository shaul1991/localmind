# Goal: Device Sync Pipeline (기기 동기화 파이프라인 — device-sync 한 방 명령)

> **044 확장 포인터**: specs/044가 device-sync 수신 후 배포 마커 검증을 workflow **target별**(공용 `.agents/skills`의 3종 marker 등)로 additive 확장한다. unavailable target은 실패로 오인하지 않는다. 과거 검증 체크는 다시 쓰지 않는다.

## Background — 배경

019(기기 로컬 자산 동기화)가 `make backup` → `make recover`/`make restore`로
노트·메모리·페르소나·스킬의 **데이터 계약**(무엇을 어디에 싣고 어떻게 복원하는가)을
확정했다. 그러나 "이미 두 기기를 함께 쓰는 사용자가 주 기기의 변경을 원격 기기에
반영하는 일상 절차"는 여전히 손으로 엮여 있었다.

2026-07-05 회고(019~030 스프린트): "다른 기기(M5) 최신화·검증" 절차를 **4회 수동
반복**했다. 매회 동일한 6단계였다 — ① (push 후) CI green 확인 → ② 주 기기에서
`make backup`(정본 변경분 push) → ③ 원격 기기에서 localmind repo `git pull` → ④ 노트
저장소 `git pull`(백업 수신) → ⑤ 테스트 실행(`seed.test.ts`) + `agents:deploy` →
⑥ 정본·배포 검증(grep). 개입 분석에서 이 절차는 **"대체 가능 구간 1순위"**로
판정됐다: 6단계가 4회 실측 검증돼 각 단계의 입력·성공 조건이 이미 굳어 있어
스크립트화 위험이 낮다.

이 절차는 recover(새 기기에서 통째 되살리기)와 다르다 — recover는 "빈 기기 부트스트랩"
(스택 기동·모델 다운로드·메모리 복원 포함)이고, 이 절차는 "이미 돌고 있는 두 기기 간
증분 최신화"다. recover를 반복해서 쓸 수는 없다(수 분·파괴적 재클론 위험). 그래서 별도
경량 명령이 필요하다.

## Problem — 문제

- 6단계가 매번 손으로 엮인다. 순서를 틀리면(예: CI 확인 전에 원격 전파) 깨진 상태를
  원격에 밀어 넣거나, backup push 전에 원격 pull을 돌려 빈 것을 받는다.
- 원격 실행은 `ssh <host> '...'`인데 비대화 셸이라 **node(nvm) PATH가 로그인 셸에
  자동 노출되지 않는 기기**가 있다(실측: M5에서 매번 PATH 수동 주입). 절차가 암묵지로
  남아 다른 기기·다른 사용자가 재현할 수 없다.
- 노트/백업 저장소 pull이 **fast-forward 불가**(양 기기 백업 교차)한 사례가 실제
  1회 발생했다. 손으로 할 땐 그때그때 판단했지만, 절차에 이 분기가 명문화돼 있지
  않으면 자동화가 데이터를 파괴적으로 병합할 위험이 있다.
- CI 확인은 `gh` CLI에 의존한다 — 미설치·미인증 기기에서 절차가 멈추거나, 반대로
  확인을 건너뛴 걸 사용자가 모른 채 전파한다.
- 원격 기기의 repo 경로·호스트명은 기기마다 다르다(오픈소스: 특정 개인 기기 하드코딩
  금지). 설정 소스가 없으면 명령을 일반화할 수 없다.

## Objective — 목표

주 기기에서 `make device-sync HOST=<라벨|호스트>` **한 명령**으로, 이미 함께 쓰는 원격
기기를 안전하게 최신화한다 — CI green 확인 → 주 기기 백업(push) → 원격 코드 pull →
원격 노트 수신 → 페르소나·스킬 배포 → 배포 검증까지, **각 단계 실패를 어디까지 됐는지와
함께 보고**하고(backup.sh의 부분 완료 문체 계승), **파괴적 동작(자동 merge·reset·force)은
하지 않는다**. ssh 호스트·경로·원격 node 환경은 전부 설정으로 일반화하고, 미설정 시
안내 후 종료한다.

## Expected outcome — 기대 결과

- 주 기기에서: `make device-sync HOST=<라벨>` 후 원격 기기가 주 기기와 동일한
  코드·노트 파일·페르소나/스킬 배포 상태가 된다(미러 구성 포함 — 자산은 019의
  restore-assets 경로로 전파). 수동 ssh 단계 0회. 주 기기의 독립 노트 git 저장소도
  push를 시도해 원격이 stale 노트를 받지 않게 한다(실패는 경고·명시).
- CI가 green이 아니면(또는 확인 불가면) 원격 전파를 **보류**하고 이유를 명시한다 —
  깨진 상태가 원격으로 새지 않는다.
- pull이 fast-forward 불가면 그 저장소만 **건드리지 않고** 보고한다(다른 저장소는
  계속) — 데이터가 자동 병합으로 훼손되지 않는다.
- 원격 node 환경은 로그인 셸로 기본 해석하되, 노출 안 되는 기기는 설정
  (`SYNC_ENV_PREP`)으로 명시 주입한다 — 특정 기기 하드코딩 없음.
- 대상 기기 미설정 시 파괴 동작 없이 설정 예시를 안내하고 비0으로 종료한다.

## Success metrics — 성공 지표

- device-sync 시나리오 테스트(ssh/gh/npm 모의): 성공 경로에서 6단계가 정해진 순서로
  실행되고 0 종료, 원격의 페르소나·스킬 배포 산출물에 `managed-by: localmind`
  마커가 존재한다.
- 임의 단계 실패(백업 push 거부·CI red·ff 불가·seed test 실패·node 부재) 각각에서:
  이후 파괴적 단계가 실행되지 않고, "어디까지 됐는지" 요약 + 비0으로 끝난다.
- 대상 미설정에서 `make device-sync`가 원격을 건드리지 않고 안내 + 비0으로 끝난다.
- 4회 수동 반복하던 절차의 명령 수가 1로 준다(회고 실측 절차의 대체).

## Non-goals — 비목표

- **양방향·실시간 동기화** — 원격→주 기기 역방향, 파일 워처, 충돌 자동 병합은 다루지
  않는다(019 Non-goals 계승). 이 명령은 주 기기 발신, 단방향(당겨오기)이다.
- **메모리(mem0) import·노트 재색인** — 스택 의존 단계는 이 명령 범위 밖이다. 019의
  `make backup`(메모리 export·push)까지만 걸치고, 원격의 메모리 import·reindex는
  `make restore`/`make recover`·스택 기동 소관이다(경계는 spec Non-goal 재확인).
- **새 기기 부트스트랩** — 스택 기동·모델 다운로드·통째 복원은 `make recover`가
  담당한다. device-sync는 이미 돌고 있는 기기의 증분 최신화 전용이다.
- **원격 스택 기동·헬스 대기** — recover의 4단계(스택 기동)를 재현하지 않는다.
- **/goal 규약 7에서의 자동 실행** — 원격 상태를 바꾸는 부작용·원격 가용성 불확실성
  때문에 자동 실행하지 않는다(제안까지만 — 확정 사항 참조).

## Constraints — 제약

- 파괴 방지 우선(019 계승): 모든 pull은 `--ff-only`, 검증은 읽기 전용, 원격을
  reset·merge·force하지 않는다. 판단이 안 서면 멈추고 보고한다.
- 015 정책 계승 + 의존 체인 구분: 서로 독립인 단계(원격 노트 저장소 여러 개의 pull)는
  경고 후 계속 + 요약 + 비0. 의존 체인(CI→backup→코드pull→수신)은 앞 단계 실패 시
  이후를 중단(fail-fast)하고 부분 완료를 보고한다 — 깨진/빈 상태의 전파 금지.
- 오픈소스 일반화: ssh 호스트·경로·node 환경을 하드코딩하지 않는다. 설정에 시크릿을
  담지 않는다(ssh 인증은 키/agent — notes-connect.sh의 `GIT_SSH_COMMAND` BatchMode
  관례 계승).
- 비개발자 사용자: 에러·안내는 평이한 한국어 + 해결 명령 동반(backup.sh 문체).
- 완전 하위호환: 기존 backup/restore/recover 동작·무설정 사용자를 깨지 않는다.
  device-sync는 순수 추가(신규 타깃·신규 설정 키).

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함), 특히 **두 대 이상의 기기를 이미
  함께 쓰며** 주 기기의 변경을 원격에 자주 반영하는 사용자.

## Risks — 리스크

- ssh 원격 실행의 node 환경 편차 → 로그인 셸 기본 + `SYNC_ENV_PREP` 명시 탈출구 +
  node 부재 시 파괴 전 중단·안내로 완화.
- pull ff 불가(백업 교차) → `--ff-only` 고정 + 해당 저장소 보류·보고(자동 병합 금지)로
  완화(backup.sh의 "pull 후 재실행" 안내와 정합).
- CI 게이트 오탐(gh가 이전 커밋의 run을 green으로 보고) → 전파 대상 커밋(HEAD sha)과
  run의 sha 일치를 확인, 불확실하면 보류(green으로 간주하지 않음). `SYNC_SKIP_CI=1`
  탈출구는 명시적 opt-in.
- 원격 명령 주입(설정 문자열이 ssh로 전달) → 설정은 신뢰 불가 입력으로 취급, host·경로
  검증 후에만 ssh에 넘긴다(notes-connect의 입력 검증 관례 계승).
- device-sync를 recover 대용으로 오인(빈 기기에 실행) → 원격 repo 부재 시 명확히
  "먼저 make recover" 안내로 완화.
