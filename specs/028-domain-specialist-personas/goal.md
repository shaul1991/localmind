# Goal: 바이브 코딩용 도메인 스페셜리스트 페르소나 세트 (why)

spec: [spec.md](spec.md) · plan: [plan.md](plan.md)

## Background

016이 페르소나 레지스트리(정본 `agents/*.md` → 다중 타깃 렌더 배포)를, 017이 런타임 자동
위임을, 018이 self-review 오케스트레이션을, 026·027이 디자인 무대(designer·ux-reviewer)를
세웠다. 현재 구성은 [docs/personas.md](../../docs/personas.md)(SSoT)의 **10종** — 기능 축
코어 5(아키텍트·크리틱·인터뷰어·리서처·워커) + 무대 확장 5(사서·분석가·큐레이터·디자이너·
UX 리뷰어)다. 모든 페르소나는 **역할 축**(어떤 종류의 사고를 하는가)으로 정의돼 있고,
**기술 도메인 축**(무엇에 대한 작업인가 — 백엔드·앱·인프라·DB…)의 lane이 없다.

사용자 결정(2026-07-04): **바이브 코딩** 무대 — 기술 도메인별 전문 페르소나가 자동
라우팅으로 개입해, 각자 도메인 가이드·프롬프트를 기반으로 구현한다. 원 요청은 11종
(백엔드·프론트엔드·iOS·안드로이드·하이브리드 앱·인프라·데이터 플랫폼·인증/인가·보안·
DBA·풀스택)이었고, lane 충돌을 제거해 **9종으로 정리**했다:

- **구현 lane 8종**: backend-dev, frontend-dev, ios-dev, android-dev, infra,
  data-platform, auth-dev(인증/인가 구현 도메인), dba.
- **검증 lane 1종**: security-reviewer — 보안은 critic과 분리된 **도메인 리뷰 lane**
  (ux-reviewer 패턴 복제: 최종 게이트는 critic으로 핸드오프).
- **풀스택 제외**: 백엔드+프론트 트리거 어휘의 합집합 = 라우팅 tie 제조기 → 두 도메인
  페르소나의 협업으로 커버.
- **하이브리드 앱(React Native·Flutter)**: 프레임워크 어휘로 ios/android와 분리 가능하다는
  근거가 있으나 유지/흡수 판단을 미룸 → Open question.

## Problem

- 기술 도메인 작업이 오면 지금은 제너럴리스트(worker)가 도메인 구분 없이 처리한다 —
  백엔드 컨벤션·앱 플랫폼 관례·DB 튜닝·보안 관점이 한 컨텍스트에 뭉개진다. 각 도메인의
  축적된 스택·컨벤션·금지사항을 재사용할 구조가 없다.
- 보안 리뷰가 critic의 일반 품질 게이트에 흡수돼 있다 — 위협 모델링·시큐어 코딩·공급망
  같은 도메인 깊이가 표면화되지 않는다.
- 도메인 지식(스택·컨벤션·하지 말 것)이 세션마다 휘발한다 — 축적·재사용할 정본이 없다.

## Objective

바이브 코딩 무대를 레지스트리에 추가한다 — **역할 축(10종)은 불변으로 두고, 구현 역할
하나에만 기술 도메인 특화 8종을 분화**하고, 검증 lane에 보안 도메인 리뷰어 1종을 더한다
(ux-reviewer 패턴 복제). 각 도메인 페르소나가 **노트 폴더의 축적형 도메인 가이드**
(`<첫 노트 폴더>/guides/{domain}.md` — 스택·컨벤션·금지사항)를 참조하는 **완화 게이트**를 세운다:
가이드가 있으면 따르고, 없으면 일반 모범 사례로 진행하되 그 사실을 명시한다(design.md식
강제 금지가 아님 — 백엔드 API 하나 고치는데 문서 선행 강제는 마찰 과다). 배포는 026
파이프라인(seedAgents fill-missing)을 그대로 재사용해 기존 사용자도 자동 수급한다.
이 확장은 **명시 위임(Agent 도구)으로만 시작**한다 — 자동 라우팅·가이드 자동 로딩은 후속.

## Success metrics

- `make agents-deploy`가 도메인 9종을 기존 10종과 동일하게 렌더·배포한다(멱등, 사용자
  정본 보호 — 016/026 계승). loadRegistry 검증 통과(problems 0, 총 19종).
- 9종 각 description의 트리거 어휘가 기존 10종·스킬·서로 간에 겹치지 않는다(라우팅 tie 0 —
  AC로 고정). critic이 보안 도메인 트리거를 security-reviewer에 양도한다.
- 도메인 가이드 템플릿(`templates/guides/guide.template.md`)이 존재하고, 각 도메인 dev
  본문이 완화 게이트 문구("가이드 있으면 따르고, 없으면 일반 관례 + 명시")를 담는다.
- docs/personas.md(SSoT)가 19종 구성·바이브 코딩 무대·도메인 축 재조정 논리를 반영한다.

## Non-goals

- **실제 도메인 코드 구현·스택 선택 대행** — 페르소나·가이드 스키마·워크플로우 문서만 만든다.
- **런타임 자동 라우팅(017 확장)** — 명시 위임으로 시작. 자동 개입은 사용 관찰 후(Open q).
- **도메인 가이드의 자동 로딩·백업 배관** — 가이드는 규약 경로의 콘텐츠일 뿐, brain.ts
  인덱스 제외·asset-mirror(019) 확장은 하지 않는다. 인덱스는 기본 검색 대상으로 두고,
  백업 갭은 정직한 한계로 명시(Open q).
- **풀스택 페르소나** — 어휘 tie 제조기라 영구 제외(협업으로 커버).
- **배포 메커니즘 변경** — 렌더러·타깃(Claude)은 016/026 그대로. 도메인 dev는 Claude
  타깃만(Codex 교차는 critic만 유지 — Open q).

## Constraints

- 오픈소스: 페르소나·가이드 문구는 특정 개인·특정 스택이 아니라 일반 사용자용으로. 개인
  절대경로 금지(플레이스홀더). 비개발자 포함 사용자가 이해할 평이한 한국어.
- 모델 배치(AGENTS.md 실패 파장 × 난이도): 도메인 구현 = **opus**(사용자 결정 2026-07-05 —
  바이브 코딩은 정의상 촘촘한 스펙 없이 가이드+대화로 진행하므로 AGENTS.md의 "명세가
  촘촘하지 않거나 판단이 필요한 코딩 = Opus" 행에 해당. worker=sonnet과 다른 이유: worker는
  spec/plan이 확정된 SDD 무대의 루틴 구현, 도메인 dev는 명세 부재가 기본값인 바이브 코딩
  무대). 보안 검증 = 적대적 검증(다운시프트 금지) → **opus 고정 🔖**. (비용 관찰 후 잘
  조여진 도메인은 sonnet 다운시프트 재론 — Open question.)
- lane 상호배타: 소유/비소유/핸드오프를 기존 10종·서로와 겹치지 않게 명시(boundary
  preamble). description 트리거 어휘 서로소를 AC로 고정.
- 정본은 사용자 데이터(노트 폴더 `agents/`) — repo는 templates 동봉 + 시드만(026 계승).

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 특히 여러 기술 도메인을 넘나들며 바이브
코딩하는 개발자 — 도메인별 축적 가이드가 매번의 컨벤션 재설명을 없애 준다.

## Risks

- **페르소나 증식(10→19, 단일 확장 최대치)** → "소수 정예" 긴장. 완화: (a) 한 무대에서만
  활성, (b) 서로소 어휘 AC 고정, (c) 라우터가 태스크당 1개 도메인만 표면화. 추가 도메인
  (게임·ML 등)의 진입 문턱을 Open question으로 못박아 무한 증식을 막는다.
- **worker ↔ 도메인 dev tie** → worker="spec/plan 확정+TDD"(도메인 noun 없음), 도메인
  dev="도메인 noun+바이브 코딩". 대부분 서로소이나 "그냥 구현해줘"(spec·도메인 noun 둘 다
  없음)는 잔여 tie — 라우팅이 Claude 내부라 테스트 불가(정직한 한계 명시).
- **critic ↔ security-reviewer 충돌** → critic이 현재 "보안 리뷰"를 트리거로 claim. 보안
  도메인 트리거를 security-reviewer에 양도(critic body 핸드오프). critic은 최종 게이트에서
  명백한 보안 결함만 잡는다(ux-reviewer↔critic 패턴과 동형).
- **도메인 가이드 백업 갭** → 규약 위치(~/.localmind/guides)는 019 asset-mirror 대상도
  git 저장소도 아니라 기본 위치면 미백업. 028 범위 밖 → git 노트 저장소에 두도록 안내 +
  2차에서 019 확장 검토(Open q).
- **완화 게이트의 실효** → "가이드 없으면 명시"는 코드 강제가 아니라 페르소나 규율에
  의존한다(design.md식 강제 아님 — 의도된 마찰 최소화). 스펙이 이 한계를 인정한다.
- **analyst와 data-platform·dba의 의미적 근접** → full 문구로는 서로소이나 bare "데이터"·
  "인덱스"를 공유해, 해석 작업("이 데이터 왜 이래?")과 구현 작업("파이프라인 만들어줘")
  사이의 오라우팅 여지가 있다(analyst=해석·제안, data-platform/dba=구현). 어휘 서로소로
  해소 불가한 잔여 tie로 인정 — 각 정본의 소유/비소유에 "해석↔구현" 경계를 명시해 완화.
- **탁상 설계** → 가이드 템플릿·페르소나 문구는 첫 실전 바이브 코딩에서 개정 전제(Open q).
