## Background — 배경

2026-07-17, localmind는 첫 CalVer 릴리스 `2026.07.0`(specs 044·050·051·052 통합)을 수행했다.
이 릴리스는 규약 없이 진행된 도그푸드였다 — 버전 체계(SemVer `0.2.0` → CalVer `2026.07.0` 전환),
버전 확정 시점, 태그·release 생성 절차를 모두 **릴리스 도중 즉석에서** 결정했다. 결정 자체는
합의됐지만(CalVer 채택, 내용/버전 관심사 분리, 머지→tag→release 순서), 어디에도 성문화되어
있지 않다. 성문 규약이 없으면 다음 릴리스는 이번의 즉석 결정을 기억에 의존해 재구성해야
하고 — 이는 저장소의 Live-Verify Facts 원칙(기억 불신)이 금지하는 진행 방식이다.

AGENTS.md 규약7은 현재 "self-review clean이면 feature 브랜치 커밋·push + PR 생성까지가
완료"(specs/051)로 끝난다. 그 이후 — PR 머지, 버전 확정, tag, release — 는 규약의 공백이다.

## Problem — 문제

1. **버전 시점 부정확 위험**: CalVer는 "언제 릴리스했나"가 버전 그 자체인데, 버전을 작업
   시작·문서 작성 시점에 미리 박으면 작업이 달을 넘길 때 버전이 실제 릴리스 시점과 어긋난다.
   실제로 현행 CHANGELOG 상단 주석은 "문서 작성(goal-ready) 시점 기준"이라고 적혀 있어
   이번 합의(머지 직전 확정)와 이미 모순 상태다.
2. **릴리스 절차 암묵**: PR 머지 → tag → release의 순서, 태그 표기(`v` 접두 여부), CHANGELOG
   갱신 시점이 어느 문서에도 없다. 다음 릴리스마다 재발명·드리프트가 생긴다.
3. **도그푸드에서 발견된 함정**: (a) gh CLI로 PR을 조작하려면 소유자 계정으로 전환이
   필요했다(collaborator 권한 문제). (b) "머지했다"는 보고가 실제 머지를 보증하지 않는다 —
   미머지 상태에서 태그를 만들면 변경이 없는 main 커밋에 빈 태그가 붙는다. 이 함정들이
   규약으로 남지 않으면 재발한다.

## Objective — 목표

localmind의 버전 체계(CalVer)와 릴리스 절차(PR 머지 → tag → release)를 **성문 규약**으로
정착시켜, 누가(사람·AI 런타임 무관) 언제 릴리스해도 같은 절차·같은 버전 규칙이 적용되게 한다.
핵심은 **관심사 분리** — 변경 내용의 서술은 작업 중에 누적하고, 버전 숫자의 확정은 릴리스
시점(PR 머지 직전)으로 미룬다.

## Expected outcome — 기대 결과

- 버전 체계·릴리스 절차가 AGENTS.md에 성문화되어 규약7의 자연스러운 연장(PR 생성 →
  머지 → tag → release)으로 읽힌다.
- 작업(goal-ready)은 버전을 정하지 않고 시작하며, 변경 내용 서술만 PR에 누적된다 —
  작업이 달을 넘겨도 버전이 어긋나지 않는다.
- 다음 릴리스가 이 규약만 읽고 재질문 없이 수행된다(도그푸드 검증).
- gh 계정·머지 검증 함정이 규약의 안전장치로 편입되어 빈 태그·헛 릴리스가 차단된다.

      담겨 있다(정적 테스트로 규약 문구 존재 검증 — green).

- [ ] 저장소 안에서 버전 확정 시점에 대한 서술이 단일하다 — "문서 작성 시점 기준" 같은

- [ ] (도그푸드 — 후속 릴리스에서 확인) 다음 릴리스가 규약 문서만으로 수행되고, 버전이
      실제 머지 시점의 연·월과 일치한다.

## Non-goals — 비목표
<!-- 이번 범위에서 제외하는 것. (필수) -->
- **SemVer 복귀 없음** — CalVer 채택은 확정 결정이다. 호환성 시그널이 필요해지면 별도
  spec으로 다룬다.
- **CI 릴리스 자동화 없음** — tag push 시 자동 `gh release create` 같은 파이프라인은 범위
  밖. 이번엔 사람·AI가 따르는 수동 절차의 성문화까지다.
- **과거 릴리스 소급 재태깅 없음** — `2026.07.0` 이전 이력(0.x 등)에 태그를 소급 부여하지
  않는다.
- **패키지 레지스트리 배포(npm publish 등) 없음** — 릴리스는 git tag + GitHub release까지다.

## Constraints — 제약
- AGENTS.md 규약7("PR 생성까지가 완료", main 직접 push 금지)과 충돌 없이 그 **연장**으로
  편입되어야 한다 — 규약7을 약화·재서술하지 않는다.
- goal-impl·goal-ready 스킬 정본과 정합해야 한다(specs/051 단일 정본 원칙 — 이중 서술 금지,
  스킬에는 참조만).
- 오픈소스 규약: 특정 개인·개인 절대경로를 규약 문안에 넣지 않는다. 비개발자가 읽고 따라
  할 수 있는 평이한 서술이어야 한다.
- 산출물은 코드가 아니라 governance 산문 규약이 중심 — 검증은 정적 점검 + 도그푸드 2층
  (specs/050·052의 지침 수준 검증 방식 계승).

## Stakeholders — 이해관계자
- 단일 사용자(설치한 개인 누구나 — 비개발자 포함): 릴리스 이력으로 "언제 무엇이 바뀌었나"를
  읽는 소비자.
- 기여자: PR을 만들고 릴리스 절차를 따르는 사람.
- AI 런타임(Claude Code·Codex·Gemini CLI): AGENTS.md 규약을 읽고 릴리스 보조를 수행하는 주체.

## Risks — 리스크
- **규약이 산문이라 드리프트 재발 가능** — 정적 테스트(규약 문구 존재)로 최소 가드하지만,
  절차 준수 자체는 도그푸드로만 확인된다. 완화: 다음 릴리스를 도그푸드 게이트로 명시.
- **관심사 분리의 마찰**: "버전 없는 CHANGELOG 항목"을 작업 중 어디에 어떻게 쌓을지가
  모호하면 오히려 기록이 누락될 수 있다. 완화: spec에서 누적 위치·형식을 확정한다.
- **MICRO 산정 모호성**: 같은 달 복수 릴리스에서 순번의 산정 기준(태그 목록 vs CHANGELOG)이
  불명확하면 충돌 가능. 완화: spec Open questions로 표면화 후 확정.
