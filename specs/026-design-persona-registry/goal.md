# Goal: 디자인·UI/UX 페르소나 확장 — 정의 없이 구현 금지의 디자인판 (why)

spec: [spec.md](spec.md) · plan: [plan.md](plan.md)

## Background

016이 페르소나 레지스트리(정본 `agents/*.md` → Claude/Codex 다중 타깃 렌더 배포)를,
017이 런타임 자동 위임을, 018이 오케스트레이션 스킬을 세웠다. 현재 구성은
[docs/personas.md](../../docs/personas.md)(SSoT)의 8종 — 기능 축 코어 5(아키텍트·크리틱·
인터뷰어·리서처·워커) + 무대 확장 3(사서·분석가·큐레이터)이다. **디자인 계열 lane이 없다.**

사용자 작업 규칙(2026-07-04 확정): 디자인·UI/UX 작업은 ① **전문 페르소나를 가진
에이전트가** 수행해야 하며, ② 작업 전에 **디자인 시스템 패턴·정의와 컴포넌트·디자인
토큰·사용할 프롬프트까지 전부 정의한 뒤** 실행해야 한다 — SDD의 "문서 없이 구현
금지"(AGENTS.md `/goal` 규약 6)의 디자인판이다.

또한 스킬·페르소나 경계 분석(2026-07-04)에서 확립된 불변식이 있다: **스킬은 페르소나를
호출·조율만 하고 lane을 재구현하지 않는다**, 그리고 **description은 자동 라우팅 키이므로
트리거 어휘가 겹치면 안 된다**(critic ↔ sdd-self-review의 "self-review" 어휘 충돌 사례).

## Problem

- 디자인·UI/UX 작업이 오면 지금은 제너럴리스트(또는 architect·worker)가 lane 구분 없이
  처리하게 된다 — 디자인 시스템 판단(토큰·패턴)과 구현·검증이 한 컨텍스트에 섞인다.
- "실행 전 사전 정의" 규칙을 강제할 실행 가능한 게이트(문서 템플릿·페르소나 preamble)가
  없다 — 규칙이 기억에만 있으면 세션마다 어긴다.
- UX 품질 기준(직관성·상태 가시성·디버깅 용이성 — 백엔드 개발자 성향에서 일반화)이
  어떤 페르소나의 소유인지 정의돼 있지 않다.

## Objective

레지스트리에 디자인 계열 lane을 추가한다 — **정의(designer) / 실행(기존 worker) /
검증(ux-reviewer)의 삼권분립**을 기존 8종의 상호배타 원칙 위에 확장하고, 디자인 작업의
사전 정의 문서(`design.md`: 패턴·토큰·컴포넌트·에이전트 프롬프트) 템플릿과 "정의 없이
구현 금지" 게이트를 실행 가능하게 만든다. 배포 메커니즘은 016 파이프라인을 그대로
재사용한다(스킬의 templates 시드 관례를 에이전트로 확장). 이 과정에서 **신규 설치
사용자의 레지스트리 부트스트랩 갭도 함께 해소**한다 — 기존 8종 정본은 지금까지 개발
기기의 사용자 데이터에만 존재해 신규 사용자에게 전파 수단이 없었다(리뷰 F3 발견).
10종 전부를 templates에 동봉·시드한다.

## Success metrics

- `make agents-deploy`가 designer·ux-reviewer를 기존 8종과 동일하게 렌더·배포한다
  (멱등, 직접 만든 파일 보호 — 016 계승).
- 새 페르소나 description의 트리거 어휘가 기존 8종·스킬과 겹치지 않는다(라우팅 tie 0).
- 디자인 작업 사전 정의 템플릿(`design.md`)이 존재하고, designer·worker의 페르소나
  본문이 "design.md 없이 UI 구현 착수 금지 — 사용자에게 알림"을 명시한다.
- docs/personas.md(SSoT)가 10종 구성·무대 지도·핸드오프를 반영한다.

## Non-goals

- **실제 UI 구현·프론트엔드 프레임워크 선택** — 페르소나와 워크플로우 문서만 만든다.
- **디자인 툴(피그마 등) 연동·MCP 도구 추가**.
- **런타임 자동 위임(017) 확장** — designer·ux-reviewer는 명시 위임(Agent 도구)으로만
  시작한다. 자동 개입은 사용 관찰 후(Open question).
- **배포 메커니즘 변경** — 렌더러·타깃(Claude/Codex)은 016 그대로. Codex 타깃 렌더는
  기존 대상(critic)만 유지(새 페르소나는 Claude 타깃만 — Open question).

## Constraints

- 오픈소스: 페르소나 문구는 특정 개인이 아니라 일반 사용자용으로. 사용자 성향(직관성·
  상태 가시성·디버깅 용이성)은 "좋은 UX의 기본 원칙"으로 **일반화해** 내장한다 —
  누구에게나 타당한 원칙이므로 개인화가 아니다.
- 모델 배치는 AGENTS.md 원칙(실패 파장 × 난이도): 디자인 시스템 정의 = 축소 불가능한
  추론(여기가 틀리면 아래가 다 틀림) → 최상위 티어. UX 검증 = 적대적 검증(다운시프트
  금지 원칙) → 최상위 티어.
- lane 상호배타: 소유/비소유/핸드오프를 기존 8종과 겹치지 않게 명시(boundary preamble).
- 정본은 사용자 데이터(노트 폴더 `agents/`) — repo는 templates 동봉 + 시드만 담당
  (스킬 관례). 기존 사용자 정본을 덮어쓰지 않는다.

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 특히 백엔드 성향 개발자처럼 디자인
판단을 체계 없이 하기 어려운 사용자 — 사전 정의 문서가 판단을 구조화해 준다.

## Risks

- 페르소나가 늘수록 라우팅 혼선 위험 → description 어휘 분리를 AC로 고정, 소수
  정예(10종 상한 인식) 원칙 유지.
- ux-reviewer와 critic의 검증 lane 충돌 → ux-reviewer는 **디자인 도메인 리뷰**(사용성·
  일관성·토큰 준수), critic은 **최종 품질 게이트** — 핸드오프를 양쪽 정본에 명시.
- 시드 추가(deploy 파이프라인 확장)가 기존 배포 멱등성을 깨뜨릴 가능성 → 스킬 시드와
  동일 규칙(정본 있으면 건너뜀 — 사용자 수정 보호)을 테스트로 고정.
- 기존 8종을 templates로 동봉하면 정본이 두 곳(사용자 데이터 + templates)에 존재 —
  드리프트 위험 → 시드는 부트스트랩 전용(없을 때만)이고 운영 정본은 사용자 데이터라는
  위계를 문서에 명시. templates 갱신은 릴리스 변경으로 취급.
- design.md 템플릿이 실제 디자인 작업 없이 설계되면 탁상공론 위험 → 첫 실전 디자인
  작업에서 템플릿을 개정할 것을 전제(Open question — 템플릿은 시작점).
