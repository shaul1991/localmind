# Goal: ask_brain Source Tracking (출처 추적 + Hallucination 감지)

## Background — 배경

`ask_brain` 은 노트 청크를 컨텍스트로 넣어 LLM에게 답변을 생성하게 한다.
그런데 현재는 답변 텍스트만 반환되고 "어떤 노트를 근거로 했는가"가 표시되지 않는다.
이는 개인 second-brain 의 핵심 신뢰 문제다: LLM이 실제 노트를 기반으로 답했는지,
아니면 학습 데이터에서 지어낸 것인지 구별할 수 없다.

## Problem — 문제

- `ask_brain` 응답이 실제 노트 기반인지 hallucination인지 알 수 없다.
- 출처 없는 답변을 신뢰하면 잘못된 기억을 사실로 받아들이는 위험이 있다.
- 어느 노트가 자주 활용되고 어느 노트가 전혀 안 쓰이는지 알 수 없어 품질 개선 방향도 모른다.

## Objective — 목표

`ask_brain` 응답에 참조한 노트 청크의 출처(파일명, 폴더 라벨)를 포함하고,
출처가 없는 경우(관련 청크를 찾지 못한 경우)를 명시적으로 표시한다.

## Expected outcome — 기대 결과

- `ask_brain` 응답 말미에 `[출처: 폴더/파일명, …]` 형태의 출처 목록이 표시된다.
- 관련 청크가 없어 노트 기반 답변이 불가능하면 "⚠️ 관련 노트 없음 — 모델 자체 지식 기반 답변" 경고를 표시한다.
- 사용자는 답변을 받는 즉시 신뢰도를 판단할 수 있다.

## Success metrics — 성공 지표

- `ask_brain` 응답 중 출처 목록 포함 비율 = 100% (관련 청크가 있는 경우)
- "관련 노트 없음" 경고 표시 비율 = 100% (청크 미히트 경우)
- 출처에 실제 존재하는 파일만 표시 (유령 출처 0건)

## Non-goals — 비목표

- 출처별 신뢰도 점수(relevance score) 표시는 이번 범위 밖
- 노트 내 특정 줄/섹션 링크는 이번 범위 밖 (파일 단위만)
- `recall` (OpenMemory) 레이어의 출처 추적은 이번 범위 밖
- Loop 4(실패 패턴 분석)에 활용될 로그 수집은 004에서 다룸

## Constraints — 제약

- 기존 `askBrain()` 함수의 LLM 호출 구조를 유지 (프롬프트 변경은 최소화)
- 출처는 이미 `searchNotes()` 가 반환하는 `path` 필드에서 추출 — 추가 파일 IO 없음
- MCP 응답 텍스트 포맷 변경이므로 기존 텍스트 파싱에 의존하는 코드가 있으면 호환 필요
- TypeScript, 기존 `brain.ts` 모듈 패턴 유지

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함) — 답변 신뢰성 향상, 노트 활용도 파악

## Risks — 리스크

- 출처 표시로 응답 텍스트가 길어지면 MCP 호스트 UI에서 잘릴 수 있음
- LLM이 프롬프트에 주입된 청크를 무시하고 자체 지식만 쓰는 경우: 이 경우 출처가 있어도 실제 hallucination일 수 있음 (완전한 해결은 아님)
