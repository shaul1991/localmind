# tasks 병렬 메타데이터 규약 (정본)

tasks 문서(plan을 구체 task로 분해한 산출물)의 각 phase가 병렬 판정을 받으려면 지켜야 할
선언 문법·판정 규칙이다. 저작은 문서 작성 스킬(또는 손작성)이 맡고, 이 문서는 **읽는 쪽**
(구현 스킬의 fan-out 판정)의 계약이다.

## 선언 문법

phase 헤더 바로 아래에 blockquote 선언 줄을 1개 둔다.

```markdown
## Phase 3 — 백업 격리 배선 (worker)
> depends-on: 없음 · files: `scripts/backup.sh`, `scripts/backup-init.sh`, `scripts/backup.test.sh`
```

- `depends-on:` — `없음` 또는 선행 phase 참조(쉼표 구분, 예 `Phase 1, Phase 2`).
- `files:` — 저장소 상대 경로(백틱으로 감싸고 쉼표 구분). 디렉토리는 `/` 접미로 표시하고,
  글롭도 허용한다(판정 방법은 아래 disjoint 판정 절).
- `files: 없음` — 파일을 건드리지 않는 phase(확인·조회만). 모든 노드와 disjoint로 취급한다
  (겹침이 없으므로 병렬 안전 — 단 병렬 대상이 되려면 유의미한 크기 조건은 별도로 만족해야 한다).
- 기존 `[P]` 표기는 **파생 힌트로 유지 가능** — 선언(depends-on·files)에서 도출되는 결론을
  사람이 스캔하기 쉽게 요약한 것일 뿐, 병렬 판정의 근거는 항상 선언이다.
- 선언이 없는 기존 tasks 문서도 **레거시로 유효**하다 — 다만 아래 disjoint 판정의 보수 기본이
  적용되어 직렬로 취급된다.
- task 라인 인라인 선언(예 `- [ ] **T3.2** ... [files: ...]`)은 한 phase 안에서 task 단위
  병렬이 필요할 때 쓸 수 있는 **선택 확장**이다 — phase 단위 선언을 대체하지 않는다.

## 파일 disjoint 판정

메인이 선언 텍스트만으로 기계적으로 판정한다(별도 도구 없이 산문+메타로 판단).

1. 경로를 저장소 상대로 정규화한 뒤, 두 노드의 files 쌍에 대해 **(a) 동일 경로** 또는
   **(b) 한쪽이 다른 쪽의 디렉토리 접두**(`src/rules/` vs `src/rules/deploy.ts`)이면 겹침으로
   판정한다.
2. 글롭(`*` 포함) 선언은 **고정 접두 디렉토리로 보수적으로 확장**해 (b)로 판정한다
   (`src/rules/*.ts` → `src/rules/`로 취급).
3. **선언이 없는 phase는 모든 노드와 겹침으로 간주한다 → 직렬**(보수 기본). 선언 누락을
   "독립"으로 해석하지 않는다.
4. 이 판정은 1차 방어일 뿐이다. 2차 방어는 배리어에서 메인이 수행하는 통합 검증(테스트·diff
   정합 확인)이며, 선언 품질이 틀렸을 때(과소 선언 등) 여기서 잡는다.

## worktree 옵션

files가 겹치면 직렬이 기본이며, worktree 격리(worker별 저장소 복제 후 병합)는 사용자가
명시적으로 선택했을 때만 쓰는 옵션이다.

## 동시 spawn 권고

한 배리어 레이어의 동시 worker는 메인이 결과를 통합·검증할 수 있는 수(권장 2~3)를 넘기지
않는다. 하드 상한은 두지 않는다 — 실질 제어는 아래 비용 가드가 맡는다.

## 비용 가드 — 병렬 spawn의 필요조건

병렬 spawn은 다음 세 조건을 **모두** 만족할 때만 한다.

1. 의존이 모두 완료됐다(independent — depends-on이 없거나 선행 phase가 배리어를 통과했다).
2. 서로 파일 disjoint하다(위 판정 규칙).
3. 각 노드가 유의미한 크기다 — worker 결과가 메인 컨텍스트로 돌아오는 고정 비용을 상회하는
   작업이어야 한다.

셋 중 하나라도 만족하지 못하면 병렬화하지 않는다. 유의미한 크기에 못 미치는 작은 task(잔task)
여러 개는 개별 병렬 spawn 대상이 아니다 — 직렬로 처리하거나 하나의 worker로 묶어 수행한다.
병렬 여지가 없는 단순한 작업은 불필요한 병렬 spawn 없이 기존 직렬 흐름 그대로 완주한다(이
규약은 병렬을 강제하지 않는다).

## 레거시

선언(depends-on·files)이 없는 기존 tasks 문서도 무효가 아니다 — 다만 선언이 없는 phase는
위 disjoint 판정의 보수 기본(누락 = 겹침 간주)에 따라 직렬로 취급된다.
