# Goal: SDD 구현 스킬 정합 — goal-impl 통일

## Background — 배경

SDD 구현 워크플로의 정본이 두 갈래로 갈라져 있다.

- **코드 정본 = `sdd-implement`**: specs/044가 내장 `/goal` 충돌 회피를 위해 신설한
  packaged 스킬(69줄). provider-neutral이고 활성화 게이트(execution grant)를 갖추며,
  완료 규칙(commit/push/CI·self-review)을 저장소 AGENTS.md에 위임한다. 대신 오케스트레이션
  지침(끊김 방어·TDD 세부·DoD 순서)은 얇다.
- **배포·governance 정본 = `goal-impl`**: 사용자 데이터 폴더에 배포된 스킬(122줄).
  끊김 방어(tasks 체크박스·phase 커밋)·TDD/RED 확인·중단 규율·DoD 순서·보고 체계가
  풍부하지만, 특정 런타임에 특화돼 있다 — 페르소나 고유명, 구체 모델명, 특정 벤더 문서
  URL, 내장 명령과의 구분 절이 하드코딩되어 중립성 스캔을 통과할 수 없다.

같은 충돌(내장 `/goal` shadow 회피)을 044는 `sdd-implement`로, governance는 `goal-impl`로
풀어 **이름이 갈렸고**, 내용도 "게이트+위임(중립)" vs "풍부한 오케스트레이션(특화)"으로
갈렸다. specs/050(페르소나/모델 바인딩)이 머지되어, 특화 내용을 "실행 등급·역할" 추상으로
바꾸고 바인딩이 해석하게 할 계약이 이미 존재한다 — 050 spec이 명시한 후속 slice가 이 051이다.

## Problem — 문제

- 하나의 워크플로에 정본이 둘이다: 이름(`sdd-implement` vs `goal-impl`)과 내용(얇은 중립
  vs 풍부한 특화)이 갈려, 어느 쪽을 고쳐야 하는지가 모호하고 드리프트가 누적된다.
- 풍부한 쪽(`goal-impl` 배포본)은 중립성 스캔의 금지 토큰(provider명·구체 모델명·런타임
  URL) 때문에 packaged 스킬로 배포할 수 없다 — 044의 provider-neutral 목표가 절반만 달성됐다.
- 얇은 쪽(`sdd-implement`)만 배포하면 끊김 방어·TDD/RED·중단 규율 같은 실행 품질 지침이
  빠진다 — 두 강점이 한 몸에 없다.

## Objective — 목표

SDD 구현 스킬을 **`goal-impl` 단일 이름·단일 정본**으로 통일한다. 정본 본문은
goal-impl(122줄 구조)의 오케스트레이션 강점을 base로, 044의 두 강점(활성화 게이트,
완료 규칙의 AGENTS.md 위임)을 병합하고, 특화 서술을 실행 등급·역할 추상으로 완전
중립화해(해석은 050 바인딩) **중립성 스캔을 통과하는 packaged 스킬로 전 지원 런타임에
배포**한다. `sdd-implement` 이름은 은퇴한다.

## Expected outcome — 기대 결과

- SDD 구현 워크플로의 논리 ID·배포 이름·governance 호칭이 `goal-impl` 하나로 수렴한다.
- 한 본문에 활성화 게이트 + AGENTS.md 위임 + 풍부한 오케스트레이션 지침이 공존한다.
- 완료 규칙의 SSoT는 AGENTS.md 하나 — 스킬 본문과의 이중 서술·드리프트가 사라진다.
- 중립화된 goal-impl이 지원 3개 런타임 타깃에 동일하게 배포된다(044 목표 완전 달성).
- specs/052(SDD 병렬 규약)가 편집할 대상 스킬이 확정된다.

## Success metrics — 성공 지표

- [ ] 저장소 활성 표면(코드·테스트·catalog·AGENTS.md·docs·templates)에 `sdd-implement`
      참조 0건 — 역사 기록(과거 specs 문서·개정 이력)은 제외.
- [ ] packaged `goal-impl` 스킬이 중립성 스캔에서 결함 0으로 통과한다.
- [ ] `goal-impl`이 지원 3개 런타임 타깃 전부에 배포되고, 배포 표면에서 `sdd-implement`가
      제거(은퇴)된다.
- [ ] goal-impl 본문에 commit/push/CI·self-review 완료 규칙의 자체 서술이 없고 AGENTS.md
      참조만 존재한다(단일 SSoT).
- [ ] 기존 전체 테스트 green(회귀 0) + 개명·중립화·게이트에 대한 신규 테스트 green.

## Non-goals — 비목표

- **052 fan-out 병렬 규약** — goal-impl에 병렬 fan-out 절을 추가하는 것은 specs/052의
  몫이다. 051은 그 편집 대상(goal-impl)을 확립할 뿐 fan-out을 포함하지 않는다.
- **050 바인딩 재작업 없음** — 바인딩 스키마·온보딩·소비 규약은 머지된 그대로 참조만
  한다. 스키마 변경·확장 없음.
- **SDD 흐름 의미 변경 없음** — goal/spec/plan → 구현 → self-review 흐름 자체, goal-ready·
  sdd-self-review 등 다른 스킬의 개편은 범위 밖이다. **예외**: AGENTS.md 완료 규칙 내용은
  기본적으로 불변이나, **PR 게이트(main 직접 push 금지 → PR 생성) 명문화는 포함**한다 —
  F-12 상충(base 본문의 PR 규칙 vs 규약7의 push-완료)을 SSoT 일원화로 해소하려면 규율이
  AGENTS.md 쪽에 살아 있어야 하기 때문(사용자 결정 2026-07-17, plan D-6).

## Constraints — 제약

- **중립성 스캔 통과**: packaged 본문에 provider명·구체 모델 ID·런타임 전용 도구명·런타임
  placeholder가 남으면 배포 불가 — 스캔이 기계적으로 강제한다.
- **AGENTS.md = 완료 규칙 SSoT**: 스킬 본문이 AGENTS.md보다 약하거나 강한 완료 규칙을
  따로 두지 않는다(044 결정 계승). **PR 게이트(main 직접 push 금지 → PR 생성)도 이 SSoT에
  포함**되어야 한다 — 즉 규율은 AGENTS.md 규약7이 담고 스킬 본문은 참조만 한다.
- **050 바인딩 참조**: 실행 등급·역할의 구체 해석(모델·페르소나)은 바인딩이 담당하고,
  바인딩 미설정 시 050이 정의한 소비 규약(안내/fallback)을 따른다.
- **개명 완전성**: 참조가 코드·테스트·catalog·규약 문서에 넓게 퍼져 있다(작업 시점 실측
  기준 30여 파일) — 부분 개명은 이중 이름 상태를 연장하므로 허용하지 않는다.
- 오픈소스 규약: 특정 개인·개인 절대경로를 산출물에 넣지 않는다.

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함): 배포된 goal-impl 스킬의 소비자.
- localmind 유지보수·기여자: 단일 정본으로 드리프트 관리 부담이 줄어든다.
- SDD 워크플로를 실행하는 AI 런타임들: 동일 본문을 각자의 바인딩으로 해석한다.

## Risks — 리스크

- **배포본 대체로 사용자 혼란**: 데이터 폴더의 기존 goal-impl(특화 122줄)이 신 중립본으로
  대체된다 — 익숙한 문구·절이 바뀌고, 사용자가 로컬에서 고쳐 쓴(fork) 내용이 있다면 유실
  위험. 관리형(managed) 판별·백업 기제로 완화하되 spec에서 엣지로 다룬다.
- **개명 누락**: 참조가 많아 한 곳이라도 남으면 "이름 둘" 문제가 재발한다. 잔존 참조를
  기계 검증(테스트)으로 잡는다.
- **중립화로 풍부함 상실**: 구체 이름을 추상(등급·역할)으로 바꾸면 지침의 즉시성이
  떨어질 수 있다 — 바인딩 미설정 환경에서 실행 품질이 낮아질 위험. 050 fallback 규약으로
  완화한다.
- **052와의 순서 결합**: 052가 sdd-implement 이름을 전제로 쓰였다면 051 개명이 052 문서를
  낡게 만든다 — 착수 전 정합 확인 필요(Open question).
