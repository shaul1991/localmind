# AGENTS.md — localmind 작업 규약

이 저장소에서 작업하는 모든 AI 에이전트(Claude Code, Codex 등)가 따르는 규약이다.

## SDD 흐름 — 기본값

모든 기능·변경은 `specs/{NNN}-{feature-slug}/` 폴더에 3개 문서로 시작한다:

- `goal.md` — 왜(why): Background·Problem·Objective·Success metrics·Non-goals·Constraints·Stakeholders·Risks
- `spec.md` — 무엇을(what): FR(각 FR은 goal 항목을 지지), Acceptance Criteria(Given-When-Then, 테스트와 1:1 매핑), Open questions
- `plan.md` — 어떻게(how): 도메인 경계, 영향 모듈, 단계, 테스트 전략

번호는 3자리(`001`, `002`, ...), 슬러그는 kebab-case. 다음 사용 가능 번호는 **작업 시점에
`specs/` 폴더를 조회한 최댓값 + 1**이다 — 문서에 적힌 예시가 아니라 실제 폴더 목록이 기준.

## `/goal {NNN}` 처리 방법

사용자가 `/goal {NNN}`(숫자만)으로 작업을 지시하면:

1. `specs/{NNN}-*/` 폴더를 찾는다(번호 프리픽스로 매칭, 슬러그는 몰라도 됨).
2. 해당 폴더의 `goal.md` · `spec.md` · `plan.md`를 모두 읽는다.
3. `plan.md`의 단계를 기준으로 구현한다 — FR/AC는 `spec.md`, 배경/의도는 `goal.md`를 따른다.
4. 구현 후 AC를 테스트로 검증한다(TDD — 실패 테스트를 먼저 쓰고 통과시킨다).
5. **self-review를 반드시 수행한다 — 생략 불가.** 구현·테스트가 끝났다고 스스로 판단해 곧장
   "완료"로 보고하지 않는다.
   - 가능하면 구현 컨텍스트와 분리된 서브에이전트(예: Claude Code의 `Agent` 도구)로 독립
     리뷰한다. 분리된 에이전트를 띄울 수 없는 환경이면, 결함을 찾으러 간다는 자세로(자기확증
     편향 배제) 직접 재검토한다.
   - 점검 범위 4가지: (1) `spec.md`의 모든 FR·AC가 구현+테스트로 1:1 충족됐는지 추적,
     (2) 유저 시나리오·엣지 케이스가 실제 테스트로 커버됐는지, (3) 로직·경계·에러처리 버그,
     (4) 불필요한 복잡도·보안 취약점.
   - 명백한 결함이나 미충족 AC를 찾으면 즉시 수정하고 다시 review한다(clean해질 때까지 반복).
     판단이 애매하거나 트레이드오프가 있는 사안만 사용자에게 보고한다.
   - self-review에서 결함 0 + 테스트 green + AC 전부 충족(미충족분은 사용자에게 명시 보고)이
     확인된 뒤에야 "완료"로 보고한다.
6. 세 문서 중 하나라도 없으면 진행 전에 사용자에게 알린다 — 문서 없이 구현하지 않는다.

## 구현 규율

- TDD: 유저 시나리오 → 실패 테스트 → 최소 구현 → 리팩터. AC를 테스트로 1:1 매핑한다.
- 외과적 변경: 요청과 무관한 리팩터·포매팅 변경을 하지 않는다.
- git commit/push는 사용자가 명시적으로 요청했을 때만 수행한다.

## 디자인·UI/UX 작업

디자인·UI/UX 작업은 "문서 없이 구현 금지"의 디자인판을 따른다(specs/026):

1. **사전 정의 먼저**: 디자이너(designer) 페르소나가 `specs/{NNN}-{slug}/design.md`를
   완성한다 — 디자인 시스템 패턴, 디자인 토큰(값 표), 컴포넌트 정의(변형·상태·화면 상태
   전이), **실행 에이전트용 프롬프트 전문**까지. 시작점은
   `cp templates/sdd/design.template.md specs/{NNN}-{slug}/design.md`.
2. **사용자 확인 후 실행**: design.md가 확인되기 전에는 워커가 UI 구현에 착수하지
   않는다. design.md가 없으면 진행 전에 사용자에게 알린다(규약 6과 동일).
3. **검증 분리**: 구현 후 UX 리뷰어(ux-reviewer)가 design.md 대비 일관성·상태 가시성·
   접근성 기본을 점검한다(도메인 리뷰). 최종 품질 판정은 크리틱(다운시프트 금지).
4. 기본 원칙: 직관성 · 상태 가시성(로딩/성공/실패의 명시적 표면화) · 디버깅/트래킹
   용이성 — 미적 완성도와 충돌하면 가시성·추적성이 우선한다.

## 모델 역할 배치

단계 라벨이 아니라 **실패 시 파장 × 난이도**로 모델 티어를 고른다. 기준(구독 요금제, Opus/Sonnet/Haiku):

| 역할(단계) | 기본 모델 | 이유 |
|---|---|---|
| 아키텍처·goal/spec/plan 정의 | **Opus** 🔖 | 축소 불가능한 추론 — 여기가 틀리면 아래가 다 틀림 |
| 적대적 리뷰·보안·최종 품질 게이트 | **Opus** 🔖 | 검증은 능력이 가장 크게 값을 하는 자리 — 절대 다운시프트 금지 |
| 복잡·신규 로직 구현 | **Opus** | 명세가 촘촘하지 않거나 판단이 필요한 코딩 |
| 잘 명세된 루틴 구현 | **Sonnet** | 스펙이 촘촘하면 코드 생성은 싼 모델에서도 수렴(012 실험 근거) |
| 결정론적 테스트 작성·실행, 기계적 확인 | **Sonnet** | 명확한 통과 기준을 돌리는 일 — "결함 캐기"는 여기 아님 |
| 대량·저위험 기계 작업(포매팅·파일 이동·로그 grep·단순 조회·보일러플레이트) | **Haiku** | 빠르고 저렴 — 정확성이 걸리지 않는 범위만 |

원칙 3가지:

1. 파장이 큰 추론(아키텍처·최종 리뷰)은 최상위 티어, 잘 명세된 기계적 실행은 다운시프트한다.
2. **최종 검증은 절대 내리지 않는다** — 약한 검증자는 거짓 안심을 준다. 사용자의 "검증"을
   (a) 기계적 테스트 실행(Sonnet)과 (b) 적대적 결함 찾기(Opus)로 쪼갠다.
3. Haiku는 코딩·검증·스펙 어디에도 두지 않는다 — 저위험 대량 작업 전용.

독립성: 구현과 리뷰가 같은 모델(Opus)이 되면 맹점을 공유하므로, **분리된 컨텍스트 + 적대적
프롬프트**("결함을 찾으러 간다")로 독립성을 확보한다(→ SDD self-review 규약과 동일).

🔖 **API 요금제 도입 시**: 위 🔖 표시 자리(아키텍처·스펙 정의, 최종 리뷰/보안 게이트)를
상위 모델 **Fable**로 승격한다. Fable은 구독 요금제 밖이라 2026-07-08부터 쓸 수 없어 기본은
Opus다(2026-07-01~07-07만 임시 가용이었음). API 도입 시 이 마커로 승격 지점을 바로 찾는다.

## 오픈소스 대상 — 비개발자 포함, 특정 개인 아님

localmind는 누구나 설치해 쓰는 오픈소스 개인 second-brain 도구다. **비개발자도 사용자다.**

- `goal.md`의 Stakeholders 등에 특정 인물(예: 저장소 소유자)을 사용자로 특정하지 않는다 —
  "단일 사용자(설치한 개인 누구나 — 비개발자 포함)"로 쓴다.
- 예시·AC에 실제 개인 절대경로를 넣지 않는다 — 플레이스홀더(`/home/<user>/...`)를 쓴다.
- 에러 메시지·MCP 도구 응답은 비개발자가 이해할 수 있는 평이한 한국어로 작성한다.
