# Goal: 검색 관측성 — 소프트 실패를 볼 수 있게

## Background — 배경

004(실패 쿼리 분석)가 쿼리 로그(`query-log.jsonl`)와 `make query-report`를 세웠고,
017이 집계 계산을 `src/query-analysis.ts`(순수 모듈)로 추출하며 `ask_brain`에 검증
결과(verify/model/persona)까지 로그를 넓혔다. 019는 이 로그를 opt-in(`BACKUP_QUERY_LOG=1`)
으로 기기 간 병합할 수 있게 했다.

2026-07-04 분석가 리포트가 확인한 사실:

- **실사용 쿼리 데이터가 사실상 없다.** 로그 51건이 전부 당일 테스트 트래픽이었다(테스트
  격리는 커밋 1ba140b로 이미 해결 — 이 스펙 범위 아님, 기결 사항으로만 기록). W27 리포트가
  집계했던 실쿼리 11건은 현재 로그에서 소실됐고 원인은 미상이다(조사 대상 — spec Open question).
- **`search_notes` 로그에 `topScore`·`sources`가 없다.** 현재 레코드는
  `ts·tool·query·hitCount·success·folder`만 남긴다(`src/brain.ts` searchNotes 로깅부). 그래서
  "결과는 나왔는데(`hitCount>0`) 1등이 정답이 아닌" **소프트 실패**를 탐지할 수단이 전혀 없다.
  두 값은 로깅 시점에 이미 계산·보유돼 있다(`searchNotesInternal`이 반환한 `NoteHit[]`의
  `score`·`path`) — 기록에 드는 추가 연산은 0이다.
- **`ask_brain` 로깅은 이미 존재하고 정상이다.** `mcp-server.ts` → `askBrain()` →
  `logAsk`가 `sources`·`success`·`verify`를 단일 레코드로 남긴다(004 + 017). 실로그에
  ask_brain 레코드가 0건인 것은 **로깅 부재가 아니라 사용 패턴**이다 — 실트래픽은
  `search_notes`와 일반 `ask`(노트를 안 보는 크로스모델 도구, 로깅 없음)로
  갔고 `ask_brain`은 거의 호출되지 않았다. 단, `ask_brain`에도 `topScore`는 빠져 있다.

향후 검색 개선(MMR·청크 병합·BM25 하이브리드·재랭킹 — 별도 후속 스펙 예정)의 효과는
데이터로만 판정할 수 있다. 리서처 조사 결론도 동일하다: **관측이 없으면 어떤 개선도 평가 불가.**
지금 로그는 "찾았나/못 찾았나"(이진)만 보이고, "얼마나 잘 찾았나"(스코어)는 안 보인다.

## Problem — 문제

- `search_notes`가 결과를 반환해도 그 결과가 얼마나 관련 있는지(topScore)를 기록하지 않아,
  "낮은 확신으로 성공 처리된" 소프트 실패를 사후에 볼 수 없다.
- `search_notes`가 어떤 노트를 반환했는지(sources)를 기록하지 않아, 반복 검색이 늘 같은
  엉뚱한 노트를 상위로 올리는 패턴을 식별할 수 없다.
- 스코어 분포 자체가 로그에 없어, 후속 검색 개선의 before/after를 수치로 비교할 기준선이 없다.

## Objective — 목표

`search_notes`·`ask_brain` 쿼리 로그에 이미 보유 중인 `topScore`와 (search_notes의) `sources`를
**하위호환(additive)** 으로 추가하고, `query-report`가 성공 검색의 **스코어 분포**를 출력하게
해서, 소프트 실패를 눈으로 볼 수 있게 한다. 소프트 실패 임계값은 이번 스펙에서 하드코딩하지
않고, 먼저 분포를 관측해 근거를 쌓는다.

## Expected outcome — 기대 결과

- 신규 `search_notes` 레코드에 `topScore`(최상위 스코어)·`sources`(반환 노트 키 목록)가 담긴다.
- 신규 `ask_brain` 레코드에 `topScore`가 담긴다(기존 `sources`·단일 레코드 불변).
- `make query-report`가 성공 검색의 스코어 분포(건수·중앙값·하위 분위·최솟값 등)를 출력해,
  "저스코어 성공"이 얼마나 있는지 임계 없이도 눈에 보인다.
- `topScore`·`sources`가 없는 기존 로그 라인이 섞여 있어도 리포트가 에러 없이 집계한다.
- `BACKUP_QUERY_LOG` opt-in 고지가 "검색어에 더해 도달한 노트 경로가 백업 git 이력에 남음"을
  명확히 알린다(비개발자 이해 가능한 한국어).

## Success metrics — 성공 지표

- 신규 기록되는 `search_notes` 레코드의 100%에 `topScore`·`sources` 필드가 존재한다.
- 레거시(필드 없음) 라인과 신규 라인이 섞인 로그로 `make query-report`가 exit 0 한다.
- 성공 검색 표본이 최소치 이상일 때 리포트에 스코어 분포 섹션이 출력된다.
- `BACKUP_QUERY_LOG` 활성화 고지 문구에 "노트 경로"가 명시된다(문구 검증 통과).

## Non-goals — 비목표

- **검색 알고리즘 자체 변경** — MMR·청크 병합·BM25 하이브리드·재랭킹은 이 스펙이 판정하려는
  대상일 뿐, 여기서 구현하지 않는다(후속 별도 스펙).
- **소프트 실패 임계값 확정** — 분포를 관측한 뒤 결정한다. 이번엔 관측만 세운다.
- **로그 포맷 파괴적 변경** — 기존 필드 제거·의미 변경 없음. 오직 필드 추가(additive).
- **원격 전송** — 로그는 여전히 로컬 전용(백업은 019의 opt-in만).
- **일반 `ask` 도구 로깅** — 노트를 근거로 하지 않는 크로스모델 상담이라 검색 품질 분석에
  기여하지 않는다(관측 대상 아님).
- **W27 로그 소실 원인의 코드 수정** — 원인 규명 전까지는 조사 항목(Open question)으로만 둔다.

## Constraints — 제약

- 완전 하위호환: 기존 파서(`readRecords` — `ts`·`query`만 요구)·기존 라인·기존 리포트 출력을
  깨지 않는다. 추가 필드는 전부 optional.
- fire-and-forget 유지: 로깅 확장이 검색 응답을 블로킹하지 않는다(004 FR-2 계승).
- 추가 연산 사실상 0 원칙: `topScore`·`sources`는 이미 계산된 값의 재사용(중복 제거 Set 등
  무시 가능한 소량만) — 검색 경로에 임베딩·스코어링 등 새 계산을 넣지 않는다.
- 프라이버시: `search_notes` `sources`(노트 경로)가 로그에 들어가면 `BACKUP_QUERY_LOG=1`
  사용자의 백업 git 이력에 실린다. 정직한 사실관계: **노트 경로는 019 시점부터 이미 백업에
  실리고 있었다**(004가 정의한 ask_brain sources) — 019 고지가 "검색 기록"만 말한 과소
  고지였고, 025는 노출을 새로 만드는 게 아니라 **과소 고지를 늦게 바로잡으며** 표면을
  확대(search_notes까지)하는 것이다. 비개발자가 이해할 평이한 한국어로 갱신한다.
- TypeScript(`brain.ts`·`query-analysis.ts`·`query-report.ts`) — 기존 관례 준수.

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함) — 검색이 "왜 어정쩡한지"를 데이터로 보고,
  후속 검색 개선의 효과를 스스로 판정하려는 사용자.

## Risks — 리스크

- `sources` 로깅으로 노트 경로가 백업 git에 실려 프라이버시 노출 표면이 넓어짐 → opt-in 유지 +
  고지 문구 갱신으로 완화(백업에서 필드만 골라 제외하는 방식은 019 dedupe 병합을 깨므로 채택
  안 함 — plan 트레이드오프 참조).
- 스코어 분포를 임계 없이 출력하면 사용자가 해석하기 어려울 수 있음 → 짧은 안내 문구 + 중앙값
  중심의 소수 통계로 제한, 임계 결정은 후속으로 미룸.
- W27 실쿼리 소실이 재발하면 이 스펙의 관측 개선이 무의미해질 수 있음 → 원인 조사를 Open
  question으로 명시하고, `--clean`의 rewrite 경합(query-report.ts 기지 리스크)을 후보로 지목.
