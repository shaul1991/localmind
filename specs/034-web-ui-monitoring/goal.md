# Goal — 모니터링 웹 UI (localmind 상태를 브라우저에서 한눈에)

## Background

- localmind의 상태 확인·관리가 전부 터미널 `make` 명령(health/doctor/secrets/query-report …)에
  흩어져 있다. 오픈소스 대상에 **비개발자 사용자**가 포함되는데, 터미널은 그 자체가 장벽이다.
- 2026-07-05 결정: 기기 상태는 **정본(origin 대비 ahead/behind)과 파생물(로컬 정합)**로 갈라
  판단한다(specs/033 결정 노트). 이 판단을 사람이 명령 조합으로 하지 않고 화면 하나로 보이게
  할 차례다.
- 2026-07-05 인터뷰 확정(4분기): ① 1차는 **read-only 모니터링**(실행은 2차) ② 실행을 열 때도
  **안전 화이트리스트만** ③ 스택은 **경량 빌드리스**(기존 Express 동거) ④ 설정은 **읽기 전용**.

## Problem

"지금 localmind가 건강한가, 정본은 최신인가, 인덱스는 정합한가, 무엇이 배포돼 있나"를 알려면
여러 명령을 알고 조합해야 한다 — 상태가 조용히 낡아도 눈에 보이지 않는다.

## Objective

브라우저(`http://127.0.0.1:8788/ui`)에서 localmind의 상태 — 스택 헬스·인덱스·정본 최신성·
설정(마스킹)·에이전트 배포·검색 품질 리포트 — 를 **읽기 전용으로** 한눈에 모니터링한다.

## Expected outcome

- `make ui` 한 번이면 브라우저에서 대시보드·설정·에이전트·리포트 4페이지가 보인다.
- 터미널 없이도 "무엇이 최신이고 무엇이 낡았는지"와 "낡았으면 어떤 make 명령을 치면 되는지"를
  알 수 있다(UI가 명령을 실행하지 않고 안내한다 — 1차).

## Success metrics

- [x] 상태 파악에 필요한 터미널 명령 0회 — 대시보드 1페이지로 health+인덱스+정본 상태가 보인다.
  *(검증: 실기기 도그푸드 — 스택 헬스 3종·인덱스 1077파일·정본 4repo가 한 화면에)*
- [x] 시크릿 원문이 브라우저에 전송되는 일 0회(마스킹은 서버 측에서 강제).
  *(검증: spec AC-5·AC-11 자동 테스트 + 심링크 우회 차단 AC-10 + 실데이터 grep 확인)*
- [x] 새 프론트엔드 빌드 파이프라인·런타임 의존 추가 0개(정적 파일 + 기존 Express).
  *(검증: package.json dependencies 변화 0, vanilla JS, 외부 URL 참조 0 자동 테스트)*

## Non-goals

- **뮤테이션 없음(1차)** — make 타깃 실행(운영 페이지)은 2차. 2차에서도 멱등·비파괴 화이트
  리스트(reindex·backup·update·health·report)만 허용하고 clean/purge/trash-empty는 영구 제외.
- **설정 편집 없음** — .env 웹 수정 불가. 변경 방법(make 명령) 안내만.
- **원격 접근 없음** — 루프백 전용 유지(BACKLOG A5 결정 계승). LAN/외부 노출·계정 체계 없음.
- 채팅/검색 UI(ask_brain 웹판) — 별도 스펙의 몫.

## Constraints

- **호스트 사이드 UI 서버**(127.0.0.1:8788, `make ui`) — 기존 서버 코드(Express·인증·Host
  가드)를 재사용하되 도커 컨테이너(:8787)에 동거하지 않는다. *(2026-07-06 개정: 컨테이너에는
  노트·인덱스·.env·에이전트 레지스트리가 마운트되지 않고, 동적 NOTES_DIR 마운트·git 자격증명
  주입은 불가/보안 후퇴라 상태 정본이 있는 호스트에서 서빙해야 한다 — plan 개정 이력 참조.)*
- 빌드리스: 정적 HTML/CSS/JS(vanilla). CDN·외부 폰트 등 외부 네트워크 요청 금지(specs/010
  공급망 기조 — 오프라인에서도 동작).
- 인증: 루프백(hostGuard) + `LOCALMIND_API_KEY`. 시크릿 마스킹은 서버 응답 단계에서 강제.
- UI 문구는 비개발자가 이해할 평이한 한국어. 상태 가시성(로딩/성공/실패 표면화)이 미적
  완성도보다 우선(AGENTS.md 디자인 원칙).

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 여러 기기 사용자는 기기마다 자기 UI를 본다.

## Risks

- **시크릿 노출**: 설정 페이지가 .env를 다룬다 — 마스킹을 클라이언트가 아닌 서버에서 하지
  않으면 원문이 네트워크에 실린다(FR로 강제, 보안 리뷰 lane 점검 항목).
- **상태 수집 비용**: 정본 ahead/behind 확인은 git fetch가 필요해 느릴 수 있다 — 명시적 새로
  고침(사용자 트리거) + 로딩 상태 표면화로 대응.
- **UI-실서버 드리프트**: 상태 API가 기존 스크립트 로직과 갈라지면 두 진실이 생긴다 — 기존
  정본(스크립트·인덱스 파일)을 읽는 방향으로만 구현(재유도 금지).
