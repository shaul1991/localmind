# Goal: Device-Local Asset Sync (기기 로컬 자산 동기화 완결 — 멀티 디바이스 복구)

## Background — 배경

localmind의 백업·복원 축은 두 번에 걸쳐 다져졌다: 006이 노트+memory.md에 개인 설정
파일(extras, opt-in)을 더했고, 015가 부분 실패 허용·정직한 보고 등 신뢰성을 보강했다.
한편 004·016~018은 새 종류의 기기 로컬 자산을 도입했다 — 페르소나 레지스트리와 스킬
정본(둘 다 **첫 노트 폴더**의 `agents/`·`skills/`, 기본 `~/.localmind` 하위), 그리고
쿼리 로그(`query-log.jsonl`, 004의 입력 데이터).

2026-07-04 실제 기기 전환(구 기기 → 새 기기)에서 이 자산들의 이동을 도그푸드했다.
git으로 도는 것(코드·노트)은 자동으로 넘어왔지만, git 밖 자산은 전부 ssh/scp
수동 이관이 필요했다: 페르소나 정의 8개, mem0 메모리 3건(export 미실행분),
쿼리 로그 14건. 또한 셸에서 돌린 `make reindex`가 **에러 없이** 전체 1,090개 중
5개 파일만 색인하는 조용한 오동작을 겪었다 — `NOTES_DIR` 정본이 MCP 등록 env에만
있어 셸 실행이 기본값(`~/.localmind`)으로 후퇴했기 때문이다.

## Problem — 문제

- 페르소나 레지스트리·스킬 정본이 **첫 노트 폴더** 하위라, 멀티 노트 저장소
  구성(012)에서 첫 노트 폴더가 `BACKUP_DIR` 밖이면 둘 다 백업 범위에서 통째로
  빠진다. 기본 구성에서도 "우연히 포함"일 뿐 보장이 없다.
- `make restore`/`make recover`는 레지스트리 파일을 복원해도 **배포를 재실행하지
  않는다**. 이는 016 FR-9의 명시적 설계 결정(복원≠배포, 명시적 실행)이었으나,
  기기 전환 도그푸드에서 "복원했는데 에이전트에 페르소나가 없다"는 함정으로
  확인됐다 — 이 스펙은 **복원 맥락에 한해 이 결정을 뒤집는다**(2026-07-04 사용자
  승인, spec 확정 사항).
- *(025 개정: 백업 고지가 "검색어 + 도달한 노트 경로"를 명시하도록 확장됨 —
  [specs/025](../025-search-observability/spec.md) FR-5. 019 시점 고지는 "검색 기록"만
  말한 과소 고지였다.)*
- `query-log.jsonl`은 004가 **프라이버시 제약**(개인 검색 패턴 — "로컬에만 보관
  필수")으로 백업에서 명시 제외했다. 그 결과 로그가 기기별로 흩어져 004 리포트가
  만성 데이터 부족("13건 < 20건")이 된다. 이 스펙은 **004의 로컬 전용 제약을
  사용자 opt-in에 한해 의도적으로 완화한다**(2026-07-04 사용자 승인, spec 확정
  사항 — 파생물이어서 제외된 것이 아니다: 재생성 불가능한 사용 데이터다).
- `NOTES_DIR`의 정본이 MCP 등록 env에만 있어(`.env`에는 주석뿐), 셸 실행 경로
  (`make reindex`, backup/restore의 자산 단계, restore의 재색인)가 조용히
  기본값으로 후퇴한다. 색인은 부분 색인이 되고, 자산 백업은 엉뚱한(빈) 폴더를
  소스로 삼을 수 있다. 실패보다 나쁜 유형이다.
- mem0 메모리는 `make backup`이 export하지만, **백업을 돌리지 않은 기기**의 최신
  메모리는 기기 전환 시 유실 위험이 있다(이번 도그푸드에서 실제로 3건이 미커밋
  상태였다). 백업 모델 밖의 데이터는 옮길 수 없으므로, 전환 절차가 이를 사용자에게
  알려주는 것까지가 이 스펙의 범위다.

## Objective — 목표

`make backup` → (새 기기) `make recover`로, 노트·메모리에 더해 **기기 로컬
자산(페르소나 레지스트리·스킬 정본·(opt-in) 쿼리 로그)까지 이전 기기와 동일한 사용
상태**가 되게 한다 — 기본 구성은 recover 한 번으로, 멀티 노트 저장소 구성(012)은
recover 후 노트 연결(notes-connect)과 `make restore` 1회 추가로(자산의 올바른
위치는 노트 연결 뒤에야 확정되므로 — recover가 이 순서를 안내한다).
`NOTES_DIR`의 정본을 단일화해 셸/MCP 실행 경로가 같은 폴더 집합을 보게 하고,
발산이 남아 있으면 `make doctor`가 감지·안내한다.

## Expected outcome — 기대 결과

- 새 기기에서: 기본 구성은 `make recover` 후, 멀티 노트 저장소 구성은 recover가
  안내한 2단 흐름(노트 연결 + `make restore` 1회) 후 — 페르소나·스킬이 **배포까지
  완료**된 상태, 쿼리 로그가 병합된 상태(opt-in 시). 수동 ssh/scp 단계 0회.
- 백업이 첫 노트 폴더 위치와 무관하게 레지스트리·스킬 정본을 포함한다. 자산의
  삭제·덮어쓰기는 **소스 판정을 확정할 수 있을 때만** 진행된다 — 판정 입력이
  없어 기본값으로 후퇴한 상태에서는 백업·복원 모두 자산의 파괴적 반영을 보류하고
  경고한다(가드가 방어하는 범위는 spec FR-1/FR-2에 정의).
- `make doctor`가 MCP 등록 `NOTES_DIR`와 셸 유효값의 불일치를 경고하고 해결 명령을
  안내한다.
- recover가 시작 시 "이전 기기에서 `make backup`을 마지막으로 실행한 시점"을
  확인하도록 안내해, 미백업 메모리·자산의 유실을 사용자가 인지하고 진행한다.
- 기존 사용자의 `make backup`/`restore` 동작은 그대로 유지된다(추가 자산은 자동
  포함 또는 opt-in — spec에서 확정).

## Success metrics — 성공 지표

- 기기 전환 시나리오 테스트: backup(기기 A) → recover(기기 B) 후 페르소나·스킬
  배포 목록(삭제 반영 포함)·메모리 수·쿼리 로그 건수가 기기 A와 일치한다.
- 자산 소스 폴더 부재·오해석 상태에서 `make backup`을 돌려도 백업 repo의 기존
  자산이 삭제되지 않는다(가드 동작).
- `NOTES_DIR` 불일치 상태에서 `make doctor`가 경고를 출력한다(현재는 침묵).
- 셸에서 돌린 재색인의 색인 파일 수가 MCP 경로로 돌린 것과 같다.

## Non-goals — 비목표

- **실시간·양방향 동기화** — 파일 워처나 충돌 자동 병합은 다루지 않는다. 같은
  파일을 두 기기에서 다르게 수정한 충돌은 `.bak-<ts>` 보존(006 extras 패턴)으로
  데이터를 지키는 것까지만 한다.
- **mem0 DB 볼륨 복제** — 정본 이동은 기존 export/import 경로만 쓴다(멱등 병합).
  백업을 돌리지 않은 기기의 메모리를 자동으로 건져오는 메커니즘은 만들지 않는다
  (안내로 완화 — Expected outcome 참조).
- **모델 티어 별칭**(016 open question) — 별도 소형 작업으로 분리.
- **SDD 전 단계 팬아웃**(018 Non-goals에서 이관된 후속 후보) — 이 스펙과 무관하게
  별도 번호로 다룬다.

## Constraints — 제약

- 완전 하위호환: 기존 backup repo 형식·기존 사용자의 무설정 동작을 깨지 않는다.
- 015 정책 계승: 단계는 서로의 인질이 아니다 — 자산별 실패는 경고 후 계속 + 실행
  말미 요약 + 비0 종료.
- 파괴 방지 우선: 삭제를 전파하는 모든 경로는 "소스가 신뢰 가능할 때만" 동작해야
  하며, 판단이 안 서면 삭제하지 않고 경고한다.
- 비개발자 사용자: 에러·안내 문구는 평이한 한국어, 해결 명령을 함께 제시한다.
- 쿼리 로그는 검색어 원문이 담긴다 — 백업 repo가 private 전제라도 opt-in으로만
  포함하고, **git 이력에 한 번 실리면 opt-out 후에도 지워지지 않음**을 활성화
  시점에 고지한다.

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함), 특히 두 대 이상의 기기를 오가며
  쓰는 사용자.

## Risks — 리스크

- 삭제-반영 미러가 소스 오해석(NOTES_DIR 후퇴) 시 백업본을 지우고 git으로 전파될
  수 있다 → 정본 단일화 + 빈/부재 소스 가드로 이중 완화(spec FR).
- 레지스트리·스킬 복원이 새 기기의 로컬 수정본을 덮어쓸 수 있다 → `.bak-<ts>`
  보존으로 완화.
- 쿼리 로그의 git 이력 비가역성 → opt-in + 활성화 시 경고 고지로 완화.
- 쿼리 로그 병합의 중복 계상·로테이션 부활 → 라인 dedupe + 병합 시 보존 기간
  적용으로 완화.
- 004(쿼리 로그 로컬 전용)·016(복원≠배포) 결정의 반전이 문서 간 불일치를 남길 수
  있다 → 해당 스펙에 개정(supersede) 표기를 남긴다(plan 단계).
