# Goal: Note Link Graph (위키링크 기반 1-hop 노트 그래프)

## Background — 배경

samchon의 `@ttsc/graph`(TypeScript 컴파일러 기반 코드 그래프 MCP) 레퍼런스를 검토하며,
"정본 분석기(ground truth analyzer) 기반 구조 색인 + 코퍼스 규모와 무관한 고정 토큰 비용"
원리가 TypeScript에 한정되지 않고 일반화 가능하다는 점을 확인했다(`reference_compiler-graph-mcp`).

localmind는 개발자뿐 아니라 비개발자도 설치해 쓰는 오픈소스 개인 second-brain 도구이며,
Obsidian은 노트 작성자(개발자·비개발자 불문)들 사이에서 가장 널리 쓰이는 PKM 도구 중 하나로,
위키링크(`[[...]]`) 문법이 사실상 업계 표준이다. 따라서 위키링크 기반 관계 추출은 특정
사용자의 습관이 아니라 Obsidian 사용자 전반에 적용 가능한 일반 기능이다.

개발 중 실제 검증용 vault(개발자 본인의 1071파일 규모 second-brain) 샘플에서 **약 24%의
파일이 본문에 위키링크를 사용**하고 있음을 확인했다 — 이는 "위키링크를 쓰는 사용자에게
유의미한 비율의 노트가 그래프로 연결된다"는 가설을 검증하는 사례일 뿐, 모든 사용자의
위키링크 사용률이 동일함을 의미하지 않는다. 위키링크를 전혀 쓰지 않는 사용자에게는
빈 그래프(연결 없음)로 graceful하게 동작해야 한다(Risks 참고).

## Problem — 문제

- `ask_brain`/`search_notes`는 임베딩 유사도 기반 퍼지 검색만 제공한다.
- 저자가 본문에 명시적으로 선언한 노트 간 관계(`[[...]]`)는 현재 어떤 도구로도 조회할 수 없다.
- "이 노트와 직접 연결된 노트는?"이라는 단순한 구조적 질문에 답하려면 전체 vault를 텍스트
  검색하거나 임베딩 유사도에 의존해야 하며, 둘 다 정확하지도 토큰 효율적이지도 않다.

## Objective — 목표

노트 본문의 위키링크(`[[...]]`)를 파싱해 노트 간 1-hop 관계(나가는 링크/들어오는 링크)를
인덱싱하고, 이를 조회하는 전용 MCP 도구를 제공한다. 응답은 노트 경로 목록만 포함하고
본문은 포함하지 않아, vault 규모와 무관하게 응답 크기가 고정된다.

## Expected outcome — 기대 결과

- 새 MCP 도구가 주어진 노트의 outgoing(나가는 링크)·incoming(들어오는 링크, backlink) 목록을
  반환한다.
- 응답에는 노트 경로(`label/파일명`)만 포함되고 본문 텍스트는 포함되지 않는다.
- 위키링크가 가리키는 대상이 vault 내에 없으면(미해결 링크) 에러 없이 "(미해결)"로 표시된다.

## Success metrics — 성공 지표

- 위키링크가 있는 실제 노트(예: `[[프로젝트/하위폴더/노트명]]` 형식)에 대해
  outgoing/incoming 링크가 정확히 해석된다.
- 응답 크기가 vault의 전체 파일 수와 무관하게 질의 대상 노트의 엣지 수에만 비례한다
  (전체 vault를 컨텍스트에 넣지 않음).
- 미해결 링크·링크 없는 노트도 에러 없이 graceful하게 처리된다.

## Non-goals — 비목표

- frontmatter `tags`/`related` 필드를 엣지로 취급하는 것은 이번 범위 밖 — 위키링크만 다룬다.
- 2-hop 이상 다단계 그래프 탐색, 고아 노트/허브 노트 같은 그래프 통계는 이번 범위 밖.
- `ask_brain`/`search_notes`에 그래프 결과를 자동 통합하는 것은 이번 범위 밖 — 별도 전용 도구로만 제공.
- Obsidian `aliases` frontmatter를 통한 별칭 노트 매칭은 이번 범위 밖 — 파일 basename 매칭만.
- 그래프 시각화는 이번 범위 밖.

## Constraints — 제약

- 기존 `brain.ts`의 `ensureIndexed()` 인덱싱 파이프라인에 통합한다 — 파일은 이미 임베딩을 위해
  전체 텍스트를 읽으므로 링크 추출에 추가 파일 IO가 없어야 한다.
- 인덱스 스키마 변경이 필요하면 `INDEX_VERSION`을 올려 기존 인덱스와 호환성 문제를 명시적으로 처리한다.
- 새 MCP 도구는 기존 `search_notes`/`ask_brain`과 분리된 전용 도구로 등록한다(`mcp-server.ts` 패턴 유지).
- TypeScript, 기존 `brain.ts` 모듈 패턴(stdout 미사용 등) 유지.

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함) — Obsidian 등 위키링크 문법으로 노트를
  구조화하는 사용자가, 그 관계를 에이전트가 활용할 수 있게 함

## Risks — 리스크

- **링크 타겟 모호성**: 서로 다른 폴더에 같은 basename의 파일이 존재하면 어떤 파일을 가리키는지
  모호할 수 있음(Obsidian의 동작과 유사한 우선순위 규칙 필요).
- **미해결 링크**: 위키링크 대상이 vault에 없거나(오타·삭제됨) 외부 참조일 수 있음 — 에러 처리 필요.
- **backlink(incoming) 계산 비용**: 노트 수가 많아지면 역방향 조회가 느려질 수 있음 — 초기엔
  on-demand 순회로 시작하고 성능 이슈 시 사전 계산으로 전환.
- **위키링크 사용률이 낮은 vault**: 위키링크를 거의/전혀 쓰지 않는 사용자의 경우 그래프가
  vault 전체를 커버하지 못해 대부분 노트가 고립된 것처럼 보일 수 있음 — 이는 기능 결함이
  아니라 사용자의 노트 작성 습관에 따른 자연스러운 결과임을 도구 응답에서 명확히 해야 한다
  (예: "연결된 노트 없음"이지 "기능 오류"가 아님을 사용자가 오해하지 않도록).
