# 020 — 색인 프루닝 가드 (what)

goal: [goal.md](goal.md) · plan: [plan.md](plan.md)

## 용어 (라벨 3분류 + 후퇴)

색인 항목의 라벨 정본은 `FileEntry.folder`다(키 문자열 파싱이 아니라 — 라벨에 `/`가
들어갈 수 있어 키의 첫 `/` 앞 조각은 라벨이 아닐 수 있다).

- **등록 라벨** — 이번 실행의 NOTES_DIR 해석 결과(FOLDERS)에 있는 라벨.
- **대상 라벨** — 등록 라벨 중 폴더 디렉토리를 실제로 읽는 데 성공한(readdir 성공)
  라벨. 삭제 반영은 이 범위에서만 일어난다.
- **부재 라벨** — 등록됐지만 readdir이 실패한(디렉토리 없음·권한 거부 등) 라벨.
  클론 전·외장 디스크 미마운트가 대표 사례 — 보존 대상이며 탈출구로도 못 지운다.
- **고아 라벨** — 색인에는 있으나 등록되지 않은 라벨. 보존하되 정리 안내의 대상.
- **후퇴 재색인** — NOTES_DIR가 환경변수·.env 어디에도 없어 기본값 또는
  `REINDEX_FALLBACK_DIR`로 도는 재색인(019 FR-6). 셸 진입점이 이를 알고 있으므로
  명시적 신호(`REINDEX_FALLBACK=1`)로 전달한다.

## FR

- **FR-1 라벨 스코프 프루닝** *(goal: Objective)* — 삭제 반영(프루닝)은 대상 라벨에
  속한 키로 한정한다. 대상 밖(부재·고아) 라벨의 키는 보존한다 — "스캔 안 됨"은
  "삭제됨"이 아니다. 라벨 판정은 `FileEntry.folder` 기준.
- **FR-2 부재 폴더 가드** *(goal: Problem 변종)* — 대상/부재 판정 기준은 **readdir
  성공 여부**다(존재 검사만으로는 권한 거부를 못 거른다). 부재 라벨은 보존하고 폴더
  경로와 함께 평이한 한국어로 경고한다. readdir에 성공한 빈 폴더는 정상적인 "모두
  삭제"로 간주해 기존대로 반영한다. 판정과 스캔은 같은 readdir 결과를 쓴다 —
  갈라지면 보존과 반영이 모순된다.
- **FR-3 후퇴 재색인은 무프루닝** *(goal: 사고 1~3)* — 후퇴 재색인에서는 삭제 반영을
  **전면 보류**한다(대상 라벨 포함). 후퇴 시 폴더·라벨 구성 자체가 신뢰 불가이므로
  (예: 폴백 라벨이 사용자의 등록 라벨과 우연히 같으면 라벨 스코프 가드가 뚫린다),
  색인 추가·갱신만 하고 삭제는 하지 않는다. 019 AC-26 경고와 함께 "삭제 반영은
  보류했어요" 안내를 낸다. 탈출구(FR-5)도 후퇴 중에는 무시한다. **고아 요약·정리
  안내(FR-4)도 산출·출력하지 않는다** — 등록 집합이 신뢰 불가라 고아 계산 자체가
  무의미하고, 후퇴 상태(make restore 직후가 대표)에서 전 코퍼스를 "고아이니
  정리하라"고 오인 안내하면 사고 원인과 같은 수동 개입을 유발한다.
- **FR-4 고아 라벨 가시화** *(goal: Risks)* — **명시적 재색인 경로**(make reindex /
  make restore)의 요약에만 고아 라벨·항목 수·보존 사실·정리 방법(FR-5)을 안내한다.
  부재 라벨은 정리 안내 없이 "폴더 없음" 경고만 낸다(정리 대상이 아니다). MCP 검색·
  캡처가 유발하는 색인 갱신 경로에서는 계산·출력하지 않는다(매 검색 스팸 금지,
  brain 모듈의 stdout 금지 규약 유지 — 전달은 재색인 API의 반환값으로).
- **FR-5 명시적 정리 탈출구** *(goal: Objective)* — `REINDEX_PRUNE_LABELS=<라벨,...>`
  지정 시 **고아 라벨만** 색인에서 제거한다. 규칙: 항목별 공백 트림, 빈 값은 조용한
  no-op, 등록 라벨(대상·부재 포함) 지정은 무시하고 그 이유를 안내, 색인에 없는
  라벨 지정은 "그런 라벨은 색인에 없어요"로 안내(조용한 성공 금지 — 비개발자가
  "정리됨"으로 오인하지 않게). `make reindex`가 변수를 명시적으로 전달한다.
- **FR-6 기존 동작 불변** *(goal: Success metrics 회귀 0)* — NOTES_DIR가 해석되는
  정상 구성에서, 대상 폴더 안의 파일 삭제 반영·watcher의 단일 파일 삭제 처리·
  single-flight·증분(해시) 스킵은 그대로다. (NOTES_DIR를 env·.env 어디에도 두지
  않고 기본 폴더만 쓰는 구성은 FR-3의 후퇴로 판정되어 `make reindex`의 삭제 반영이
  보류된다 — watcher 단일 삭제는 계속 동작하며, `make mcp-install`이 NOTES_DIR를
  .env에 기록하면 정상 반영된다. 이 트레이드오프를 docs에 안내한다.)

## 알려진 한계 (스코프 밖 — 정직한 문서화)

- 색인은 라벨↔폴더 경로 바인딩을 기록하지 않는다(`FileEntry.folder`는 라벨뿐).
  따라서 사용자가 **같은 라벨을 다른 폴더에 재사용**하면(NOTES_DIR에서 라벨의 경로를
  바꾸면) 이전 폴더의 항목은 정상 재색인에서 삭제 반영된다 — 라벨 기준으로는 "그
  폴더의 파일이 사라진 것"과 구분 불가. 근본책(바인딩 기록)은 Open questions 1.
- FR-2의 readdir 가드는 **폴더 최상위**에만 적용된다. 대상 폴더 안의 하위
  디렉토리가 권한 거부 등으로 열리지 않으면 스캔이 그 하위만 조용히 건너뛰고,
  해당 하위 경로의 키는 프루닝된다(원래 버그의 하위 디렉토리 축소판). 하위 단위
  가드는 스캔 구조 변경이 필요해 이번 범위 밖.

## Acceptance Criteria

각 AC는 테스트와 1:1 매핑한다. 경로 예시는 플레이스홀더(`/home/<user>/...`)를 쓴다.

- **AC-1 (FR-3)** Given 색인에 `a`·`b`·`c` 3개 라벨의 항목이 있고, When 후퇴
  재색인(`REINDEX_FALLBACK=1` 또는 NOTES_DIR 완전 부재)이 돌면, Then 색인에서 키가
  1건도 제거되지 않고(폴백 라벨 스캔에서 안 보인 키 포함), "삭제 반영 보류" 안내가
  출력되며, 요약에 고아 라벨 정리 명령(`REINDEX_PRUNE_LABELS`)이 포함되지 않는다.
- **AC-2 (FR-3, 복구 무비용)** Given AC-1 직후, When NOTES_DIR를 3개 폴더로 되돌려
  재색인하면, Then `a`·`b`·`c`의 파일은 해시 불변으로 전량 증분 스킵되고(재임베딩
  0건) 삭제 반영은 다시 동작한다.
- **AC-3 (FR-2)** Given NOTES_DIR에 3개 폴더가 등록됐고 그중 1개의 디렉토리가 없으면
  (또는 권한 거부로 readdir 실패), When 재색인하면, Then 그 라벨의 키는 보존되고,
  경고에 폴더 경로가 표시되며, 나머지 2개 폴더는 정상 색인·삭제 반영된다.
- **AC-4 (FR-1·6)** Given 대상 폴더에서 파일 1개를 삭제하고, When 재색인하면, Then
  그 파일의 키만 제거되고 같은 라벨의 다른 키와 다른 라벨의 키는 보존된다.
- **AC-5 (FR-2)** Given 대상 폴더가 존재하고 readdir 가능하지만 마크다운 파일이
  0개이면, When 재색인하면, Then 그 라벨의 기존 키는 전부 제거된다(기존 동작 회귀
  고정).
- **AC-6 (FR-4)** Given 고아 라벨이 있는 상태로, When 명시적 재색인(make reindex)을
  실행하면, Then 요약에 라벨명·항목 수·"보존 중" 문구·정리 명령이 출력된다. 부재
  라벨에 대해서는 정리 명령이 출력되지 않는다.
- **AC-7 (FR-5)** Given 고아 라벨 `b`가 있고, When `REINDEX_PRUNE_LABELS=b`로
  재색인하면, Then `b` 라벨의 키만 제거되고 다른 라벨은 불변이다. 공백 포함
  표기(`" b , c "`)도 트림되어 동작한다.
- **AC-8 (FR-5)** Given 대상 라벨 `a`와 부재 라벨 `m`을, When
  `REINDEX_PRUNE_LABELS=a,m`으로 지정해 재색인하면, Then 둘 다 제거되지 않고 각각
  무시 사유가 안내된다.
- **AC-9 (FR-5)** Given 색인에 없는 라벨 `x`를, When `REINDEX_PRUNE_LABELS=x`로
  지정하면, Then 아무것도 제거되지 않고 "그런 라벨은 색인에 없어요" 안내가 출력된다.
  빈 값(`REINDEX_PRUNE_LABELS=`)은 안내 없이 no-op이다.
- **AC-10 (FR-6)** Given watcher가 감지한 단일 파일 삭제 이벤트는, When 처리되면,
  Then 기존과 동일하게 해당 키만 제거된다(기존 테스트 green 유지로 검증).
- **AC-11 (배선)** Given `make reindex REINDEX_PRUNE_LABELS=<라벨>`을 실행하면, Then
  reindex.sh가 변수를 재색인 프로세스에 **명시적으로** 전달한다(make의 암묵 export에
  의존하지 않음). 그리고 reindex.sh는 `resolve_notes_dir`가 빈 값을 반환하면 —
  이후 `REINDEX_FALLBACK_DIR`로 NOTES_DIR를 재할당했는지와 **무관하게** —
  `REINDEX_FALLBACK=1`을 함께 전달한다(make restore의 BACKUP_DIR 폴백 경로 포함.
  NOTES_DIR가 세팅돼 있다는 이유로 신호를 생략하면 폴백 라벨 충돌 시 FR-3이 뚫린다).

## Open questions

1. 라벨↔폴더 경로 바인딩을 색인에 기록해 "라벨 재사용"과 "파일 삭제"를 구분할지
   (알려진 한계 해소) — 색인 포맷 확장이 필요해 별도 스펙 후보.
2. 명시적 정리(FR-5) 직후 다른 프로세스의 동시 저장이 mergeIndexFromDisk 경로로
   지운 항목을 되살릴 수 있다(기존 프루닝과 동일 성질, 미실증). 명시 정리에 merge
   우회가 필요한지.
3. 존재하는 빈 폴더의 전량 프루닝(AC-5)에 확인 게이트를 둘지 — 빈 폴더가 마운트
   지점인 시나리오. 이번 범위에서는 기존 동작 유지.
4. 고아·부재 라벨 안내를 doctor([노트 폴더 정합] 섹션)에도 추가할지.
