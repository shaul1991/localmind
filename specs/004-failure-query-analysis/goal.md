# Goal: Failure Query Analysis (실패 쿼리 분석 — Loop 4)

## Background — 배경

001~003 이 Loop 2·3을 구현하면 캡처 검증·출처 추적·자동 인덱싱 데이터가 쌓이기 시작한다.
Loop 4(언덕 오르기 루프)는 이 누적 데이터를 분석해 시스템 자체를 개선하는 단계다.
구체적으로: `search_notes` / `ask_brain` 에서 "관련 결과 없음" 으로 끝난 쿼리 패턴을 파악하고,
청크 크기·임베딩 모델·인덱싱 전략을 개선하거나 노트 작성 습관을 바꿀 수 있게 제안을 생성한다.

## Problem — 문제

- 지금은 `search_notes` / `ask_brain` 가 "결과 없음"을 반환해도 어디에도 기록되지 않는다.
- 같은 질문이 반복 실패해도 사용자는 "왜 못 찾는지" 알 방법이 없다.
- 인덱싱 품질 개선(청크 크기, 임베딩 모델 교체 등)을 위한 데이터 근거가 없다.

## Objective — 목표

`search_notes` / `ask_brain` 의 쿼리와 결과를 로컬 로그 파일에 기록하고,
주기적(또는 온디맨드) 분석으로 "자주 실패하는 쿼리 유형"과 "개선 제안"을 리포트한다.

## Expected outcome — 기대 결과

- 쿼리 로그(`~/.localmind/query-log.jsonl`)에 각 검색의 쿼리·결과 수·성공 여부가 기록된다.
- `make query-report` 실행 시 최근 N일 실패 쿼리 분석 리포트가 출력된다.
- 리포트에는 "자주 실패하는 키워드", "청크 크기 개선 제안", "노트 갭(없는 주제)" 이 포함된다.

## Success metrics — 성공 지표

- 쿼리 로그 기록 누락 0% (search_notes / ask_brain 호출 시 100% 기록)
- `make query-report` 가 로그 없어도 오류 없이 "데이터 없음" 반환
- 리포트에 최소 3가지 개선 제안 카테고리 포함

## Non-goals — 비목표

- 실시간 자동 파라미터 튜닝(예: 청크 크기 자동 변경)은 이번 범위 밖 — 분석 + 제안까지만
- 원격 서버로 로그 전송은 이번 범위 밖 (로컬 파일만)
- `remember` / `recall` (OpenMemory) 쿼리 분석은 이번 범위 밖
- LLM을 활용한 자동 노트 생성(갭 채우기)은 이번 범위 밖

## Constraints — 제약

- 로그는 로컬 JSONL 파일로만 (`~/.localmind/query-log.jsonl` 기본, `QUERY_LOG` 환경변수로 재설정)
- `.gitignore` 에 로그 파일 추가 (개인 쿼리 패턴 커밋 방지)
- 로그 기록 실패가 검색 응답을 블로킹하지 않음 (fire-and-forget)
- 분석 리포트는 스크립트로 온디맨드 실행 (`make query-report`) — 백그라운드 스케줄러 없음
- 001(capture 검증)·002(출처 추적) 데이터를 로그에 함께 포함 (통합 로그)
- TypeScript(brain.ts 로깅) + 분석 스크립트(Node.js 또는 shell)

## Stakeholders — 이해관계자

- 단일 사용자(jihoonkim) — second-brain 검색 품질 개선 방향 파악

## Risks — 리스크

- 쿼리 로그가 개인 검색 패턴을 담으므로 로컬에만 보관하는 제약 필수
- 로그 무한 증가: 로테이션 정책 필요 (30일 초과 항목 삭제 등)
- 분석 리포트 품질: 데이터가 적으면 제안이 의미 없음 → 최소 샘플 수 경고 필요
- 001~003 구현 후에야 충분한 데이터가 쌓임 — 이 goal은 001~003 이후 착수
