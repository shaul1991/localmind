# Spec: Persona Runtime Delegation

상위: [goal](goal.md)

> 2026-07-03 페르소나 리뷰(인터뷰어·아키텍트·크리틱) + 사용자 결정 4건 반영:
> 검증 = 항상 + 추론 강도 강등 + 일일 상한 / 경고 기준 = 사실·수치·인용만 /
> 리포트 = recall 노출 / 응답 표시 = 문제만.

## Scope

localmind 런타임(brain 레이어)이 016 레지스트리의 페르소나 4개에게 자동 위임하는 기능:

| 지점 | 페르소나(고정 slug) | 자동 동작 (기본 켜짐) |
|---|---|---|
| ask_brain 합성 | `librarian` | 사서의 지침·모델로 답변 합성 |
| ask_brain 검증 | `critic` | 교차 백엔드로 사실 검증 → 문제 시 경고 부착 |
| capture 분류 | `curator` | 태그 제안 → 노트 frontmatter 자동 반영 |
| 품질 리포트 | `analyst` | 주기 실행 — 실패 질의 해석을 노트로 저장 |

지점↔페르소나 바인딩은 **고정 slug**다(위 표) — 다른 이름의 페르소나는 트리거되지
않으며, 이 사실을 문서에 명시한다(FR-10). 각 지점은 환경변수로 끌 수 있고, 실패·부재·
지연 시 본래 기능을 해치지 않는다.

## Context

- `askBrain()`은 검색 청크를 컨텍스트로 게이트웨이 `/v1/chat/completions`에 고정
  `ANSWER_MODEL`(기본 sonnet)·내장 시스템 프롬프트로 합성을 요청하고, 쿼리 로그는
  **합성 요청 전에** 기록된다. `capture()`는 frontmatter를 만들어 노트를 쓰고 색인한다
  (태그 없음). `make query-report`(004)는 실패 질의 로그를 집계해 터미널에 출력한다.
- 016의 `loadRegistry()`가 페르소나(name·프롬프트·대상별 모델)를 제공한다. 게이트웨이는
  **모델명으로 claude/codex 백엔드를 판별**한다 — 이 문서에서 "교차"란 합성과 검증이
  **서로 다른 게이트웨이 백엔드**(claude vs codex)를 쓰는 것을 뜻한다.

## Functional Requirements

- **FR-1 (페르소나 해석·무음 폴백)**: 런타임이 레지스트리에서 고정 slug로 페르소나를
  해석해 지침(본문)과 모델을 얻는다. 해석은 호출 시점에 이뤄져 **정의 변경이 재시작
  없이 다음 호출부터 반영**된다. 페르소나·레지스트리가 없거나 정의에 문제가 있으면
  해당 자동화만 생략하고 본래 동작을 유지하며, **응답에 어떤 표시도 남기지 않는다** —
  페르소나를 만들지 않은 사용자의 출력은 기존과 완전히 동일하다(goal의 약속).
  → goal: Constraints(본래 기능 우선), Success metrics(재시작 없이 반영)

- **FR-2 (사서 합성)**: `librarian`이 있으면 ask_brain 합성의 시스템 프롬프트와 모델을
  사서의 것으로 대체한다. 모델은 claude 대상 우선, 없으면 codex 대상을 쓴다. 런타임
  강제 규칙(노트 근거·출처 인용)은 페르소나 지침 **앞에** 붙어 항상 이긴다. 사서가
  없으면 기존 내장 프롬프트·ANSWER_MODEL로 동작한다.
  → goal: Objective, Success metrics(정의가 곧 동작)

- **FR-3 (크리틱 교차 검증 — 사실 대조)**: ask_brain이 답변을 만든 뒤 `critic`으로
  검증한다. 검증 범위는 **구체적 사실 주장·수치·날짜·인용이 출처 청크와 일치하는가**
  로 한정한다 — 일반 서술·종합·연결은 검사하지 않는다(경고 스팸 방지, 2026-07-03 결정).
  검증 모델은 **합성과 다른 백엔드를 우선**한다(크리틱의 codex 대상 우선). **교차가
  불가능하면(합성과 같은 백엔드의 대상만 있으면) 동종 검증으로 위장하지 않고 검증을
  생략**하며 사유를 표시한다. 문제 발견 시 답변 뒤에 경고 블록(무엇이 확인 안 되는지 +
  추정임을 명시 + 끄는 법 안내)을 부착한다. 답변 차단·재합성은 하지 않는다.
  → goal: Objective(교차 검증), Risks(검증 오판), Non-goals(경고 부착 확정)

- **FR-4 (검증 비용 통제)**: 자동 검증은 ① **추론 강도 자연 강등** — 게이트웨이 경유
  호출에는 페르소나의 reasoning_effort(high)가 전달되지 않아 codex CLI 기본 강도로
  돈다(프로필의 high는 `codex exec -p` 수동·오케스트레이션 경로 전용 — 자동 검증이
  high로 도는 경로가 존재하지 않음), ② **시간 상한** — 초과 시 포기, ③ **일일 횟수
  상한**(기본 50회) — 초과 시 생략.
  포기·생략 시 답변은 지연 없이 반환되고 사유가 표시된다. 검증 호출 실패(백엔드
  미기동 등)도 같은 방식이다. 합성이 실패했거나 검색 결과가 없어 답변이 없는 경우에는
  검증을 시도하지 않는다.
  → goal: Constraints(지연 통제), Risks(쿼터 잠식·지연 누적)

- **FR-5 (큐레이터 태깅)**: capture 시 `curator`로 태그를 제안받아 노트 frontmatter의
  `tags`에 자동 반영한다. 노트들에서 수집한 **기존 태그 어휘**를 제공해 재사용을
  우선시키고, 어휘가 비어 있으면(신규 설치 정상 경로) 새 태그를 보수적으로(소수만)
  달게 한다. 태깅은 **capture 시 1회만** 수행한다 — 재색인·이후 동작이 사용자가 수동
  수정한 태그를 덮어쓰지 않는다. 태깅 실패·지연 시 태그 없이 캡처를 완수한다.
  → goal: Objective, Risks(태그 오염)

- **FR-6 (분석가 리포트)**: 실패 질의 집계(004)에 `analyst`의 해석(패턴·가설·개선 제안)
  을 붙여 **노트 폴더의 리포트 노트로 저장**하는 실행 진입점을 제공한다. 리포트는 색인
  대상이라 recall/ask_brain으로 회수된다(2026-07-03 결정 — 두뇌가 자기 상태를 기억).
  집계에는 검증 결과 통계(경고율 등, FR-9의 로그 기반)를 포함한다. 로그 표본이
  부족하면 "데이터 부족"을 본문에 기록한 리포트를 생성한다(생성 자체는 함 — 안 나왔다는
  착각 방지). `analyst`가 없으면 해석 절 없이 집계만 담아 생성한다.
  → goal: Objective(자기 상태 기억), Expected outcome

- **FR-7 (주기 등록)**: 리포트 실행을 주기 등록하는 단계별 안내 진입점을 제공한다
  (기존 backup-cron 패턴). 등록은 사용자가 명시적으로 실행한다.
  → goal: Objective(주기 실행), Constraints(사람 주도 등록)

- **FR-8 (지점별 끄기)**: 사서 합성·크리틱 검증·큐레이터 태깅 3지점을 각각 환경변수로
  끌 수 있다(기본 켜짐 — 최대 자동화). 끄면 해당 지점은 기존 동작과 동일하다.
  (리포트는 실행형이라 토글이 아니라 실행 여부로 통제된다.)
  → goal: Success metrics(환경변수 하나로 끄기)

- **FR-9 (개입 관측 — 응답은 문제만, 로그는 전부)**: 응답 표시는 **문제와 능동적
  생략만** 한다 — 검증 경고, (페르소나가 존재하는데) 시간 초과·호출 실패·상한 초과·
  교차 불가로 인한 검증 생략, 그리고 **데이터를 변형한 개입(부여된 태그)**. 검증 통과·
  정상 합성·페르소나 부재는 **무음**이다(2026-07-03 결정 — 잡음 방지). 반면 쿼리
  로그에는 검증 결과가 통과·경고·생략 모두 기록되고, **합성에 쓴 모델·사서 개입
  여부도 함께 기록**된다(ask_brain 1회 = 로그 1레코드 유지) — 응답은 무음이어도
  로그로 개입을 감사할 수 있다. 환경변수로 끈 지점은 관련 필드를 기록하지 않는다.
  이 로그가 리포트(FR-6)의 입력이 된다.
  → goal: Constraints(관측 가능), §8 도그푸드, Risks(검증 오판 — 판단은 사용자)

- **FR-10 (문서)**: docs/agents.md(또는 usage.md)에 문서화한다 — 고정 slug 바인딩
  ("합성을 위임하려면 이름이 정확히 `librarian`이어야 한다"), 끄는 법(환경변수),
  태그를 바꾸는 법(노트 파일에서 직접 — 수동 수정은 보존됨), 리포트 노트에 검색 질의
  원문이 포함되어 백업 저장소에 커밋됨(주의), 검증 경고는 추정이며 판단은 사용자 몫.
  비개발자가 이해할 수 있는 평이한 표현으로.
  → goal: Risks(리포트 민감정보), Constraints(비개발자)

## Acceptance Criteria

- **AC-1 (사서 합성)**: Given `librarian` 페르소나가 레지스트리에 있을 때,
  When ask_brain을 호출하면,
  Then 합성 요청의 모델·시스템 프롬프트에 사서의 정의가 사용되고, 강제 규칙(노트
  근거·출처 인용)이 페르소나 지침 앞에 위치한다.

- **AC-2 (무음 폴백)**: Given 레지스트리가 없거나 4개 페르소나가 모두 없을 때,
  When ask_brain·capture를 호출하면,
  Then 기존과 동일한 동작·출력이며, 응답에 개입·생략 관련 표시가 **하나도 없다**.

- **AC-3 (검증 경고)**: Given 크리틱 검증이 경고 판정(확인 안 되는 사실 주장 목록)을
  반환할 때,
  When ask_brain이 응답을 구성하면,
  Then 답변 뒤에 해당 항목을 지목하는 경고 블록이 부착되고, 추정임과 끄는 법 안내가
  포함된다. (실제 hallucination 탐지 품질은 도그푸드에서 확인.)

- **AC-4 (검증 통과 = 무음, 로그만)**: Given 크리틱 검증이 통과 판정을 반환할 때,
  When ask_brain이 응답을 구성하면,
  Then 답변에 검증 관련 표시가 없고, 쿼리 로그에는 통과가 기록된다.

- **AC-5 (엣지 — 검증 실패·시간 초과)**: Given `critic`이 존재하고 검증 호출이
  실패하거나 시간 상한을 초과할 때,
  When ask_brain을 호출하면,
  Then 답변은 정상 반환되고 "검증 생략"과 사유가 표시된다.

- **AC-6 (교차 우선)**: Given 크리틱에 codex·claude 대상이 모두 있고 합성이 claude
  백엔드일 때,
  When 검증이 수행되면,
  Then codex 대상 모델로 검증 요청이 나가고, 요청에 페르소나의 reasoning_effort는
  포함되지 않는다(자연 강등 — FR-4①).

- **AC-7 (엣지 — 교차 불가 시 생략)**: Given 크리틱에 합성과 같은 백엔드의 대상만
  있을 때,
  When ask_brain을 호출하면,
  Then 동종 모델 검증을 하지 않고, "검증 생략(교차 모델 없음)"이 표시된다.

- **AC-8 (엣지 — 일일 상한)**: Given 오늘 검증 횟수가 상한에 도달했을 때,
  When ask_brain을 호출하면,
  Then 검증 없이 답변이 반환되고 "검증 생략(일일 상한)"이 표시되며, 로그에 생략이
  기록된다.

- **AC-9 (큐레이터 태깅)**: Given `curator` 페르소나가 있을 때,
  When capture로 노트를 저장하면,
  Then 노트 frontmatter에 `tags`가 기록되고, 응답에 부여된 태그가 표시된다.

- **AC-10 (엣지 — 태깅 실패·부재)**: Given 태깅 호출이 실패하거나 `curator`가 없을 때,
  When capture로 노트를 저장하면,
  Then 노트는 태그 없이 정상 저장·색인되고 캡처는 성공으로 보고된다(부재 시 표시도
  없음).

- **AC-11 (수동 태그 보존)**: Given 캡처된 노트의 태그를 사용자가 수동 수정한 뒤,
  When 재색인 또는 이후 localmind 동작이 수행되면,
  Then 수정된 태그가 그대로 보존된다.

- **AC-12 (분석가 리포트 생성)**: Given 실패 질의 로그에 분석할 데이터가 있을 때,
  When 리포트 진입점을 실행하면,
  Then 집계 + 검증 통계(경고율 등) + 분석가 해석이 담긴 리포트 노트가 노트 폴더에
  생성된다. (recall 회수는 실 임베딩이 필요하므로 도그푸드에서 확인.)

- **AC-13 (엣지 — 데이터 부족·분석가 부재)**: Given 로그 표본이 기준 미만이거나
  `analyst`가 없을 때,
  When 리포트 진입점을 실행하면,
  Then 각각 "데이터 부족" 본문 리포트 / 해석 절 없는 집계 리포트가 생성되며 실패로
  끝나지 않는다.

- **AC-14 (지점별 끄기)**: Given 검증(또는 태깅·사서 합성)을 환경변수로 껐을 때,
  When 해당 기능을 호출하면,
  Then 그 지점의 자동화·표시 없이 기존 동작과 동일하게 완수된다.

- **AC-15 (로그 단일 레코드)**: Given 검증이 수행된(또는 생략된) ask_brain 호출이
  있을 때,
  When 쿼리 로그를 확인하면,
  Then 그 호출은 **한 레코드**이고 검증 결과(통과/경고/생략)가 그 레코드에 있다 —
  이중 기록으로 리포트 집계를 오염시키지 않는다.

## Open questions

- 검증 시간 상한·태깅 시간 상한·일일 상한의 기본값(60s/30s/50회는 초기값) — 도그푸드
  실측 후 plan에 확정 기록.
- 태그 어휘 수집의 비용(매 capture 스캔 vs 캐시) — plan의 캐시 초안으로 시작, 실측 조정.
- capture 태깅의 동기/비동기 — 동기+상한으로 시작, 지연이 크면 비동기 전환을 후속 스펙.
- 비개발자용 설정 토글(MCP 도구) — goal Non-goals, 효용 확인 후 후속.
