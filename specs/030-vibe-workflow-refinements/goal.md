# Goal: 바이브 코딩 워크플로우 개정 — 계약 저장소 실전 테스트 발견 반영 (why)

spec: [spec.md](spec.md) · plan: [plan.md](plan.md)

## Background

029까지 세운 바이브 코딩 체계(도메인 페르소나 19종 + 도메인 가이드 + 프로젝트 계약
저장소)를 memo-api 놀이터에서 **3차 실전 테스트**(2026-07-05)로 검증했다:

- 1차(backend-dev): PATCH 구현 + api-contract.md 최초 작성(owner 쓰기)
- 2차(frontend-dev, 신규 세션): 세션 간 계약 소비 — **1차가 쓴 계약을 스스로 발견해
  정본으로 소비**(029의 존재 이유 실증). designer 위임 → ux-reviewer 점검 lane 왕복 포함.
- 3차(backend-dev, 신규 세션): **드리프트 자가 탐지→owner 수복**(rev.2) — 2차가 추가한
  정적 서빙 라우트의 계약 누락을 curl 실측으로 확인하고 계약을 갱신.

핵심 메커니즘(계약 소비·드리프트 수복·lane 핸드오프·완화 게이트 보고)은 전부 규율대로
동작했다. 동시에 **규약이 덜 조여진 지점 4건(F1~F4)** 이 실전에서 드러났다 — 029 Open
questions가 예약해 둔 "첫 실전 검증에서 개정 전제"의 실현이다.

## Problem (실전 발견 4건)

- **F1 — 계약 파일 위치 이탈**: backend-dev가 api-contract.md를 처음에 **프로젝트 루트**에
  생성했다(지시로 정정). 페르소나 본문의 `projects/<project>/` 표기가 "어디의 projects/인지"
  (노트 폴더)를 생략해 프로젝트 상대 경로로 오독됐다. 계약 저장소의 요점(백업·기기 동기화
  자동)이 무너지는 이탈이다.
- **F2 — design.md 게이트의 무대별 강도 미분화**: frontend-dev가 design.md 부재 시
  designer에게 정의를 위임해 만든 뒤 **사용자 확인 없이 즉시 구현**했다. 026 게이트("사용자
  확인 후 실행")의 문면과는 어긋나지만, 바이브 코딩의 저마찰 흐름에는 자연스럽다 —
  무대별 분기가 규약에 없어서 생긴 회색지대다. **사용자 결정(2026-07-05): 바이브 무대는
  정의→즉시 구현 허용(생성 사실을 보고 맨 앞에 명시 — 사후 검토 가능), SDD 무대(specs/)는
  기존대로 사용자 확인 필수.**
- **F3 — 바이브 프로젝트의 design.md 위치 미규정**: 026은 design.md 위치를
  `specs/{NNN}-{slug}/`(SDD 무대)로만 규정했다. 바이브 프로젝트엔 specs/가 없어 에이전트가
  `projects/memo-api/design.md`로 **즉흥 결정**했다 — 합리적 창발(계약 저장소와 같은 폴더 =
  백업·동기화·context-map 포인터 정합)이므로 규약으로 승격한다.
- **F4 — frontend-dev의 server.js 수정(lane 회색지대)**: same-origin 정적 서빙 훅을
  frontend-dev가 직접 추가했다. 계약 영향을 표면화했고 3차에서 수복됐으므로 실해는
  없었으나, 서버 코드 수정은 backend-dev lane이다. 규약 변경 없이 **관찰 유지**(Open q) —
  반복되면 경계 문구를 추가한다.

## Objective

F1~F3을 **문구 수준의 최소 개정**으로 반영한다(신규 코드 0 — 026~029와 동일한 콘텐츠
전용 스코프): ① 계약 위치 표기를 "노트 폴더의 projects/"로 조이고(페르소나 본문 +
템플릿 헤더), ② design.md 게이트를 무대별로 분기해 명문화하고(바이브: 정의→즉시 구현 +
생성 명시 / SDD: 사용자 확인 — AGENTS.md·UI dev 3종·designer), ③ 바이브 프로젝트의
design.md 표준 위치를 `projects/<project>/design.md`로 규약화한다(AGENTS.md·context-map
템플릿 포인터·관련 페르소나). F4는 Open question으로 기록만 한다.

## Success metrics

- 계약 경로를 언급하는 페르소나 본문(owner 3종 + 소비자 3종 + designer)과 계약 템플릿
  4종 헤더가 전부 "노트 폴더"를 명시한다(projects/ 포함 **줄 단위** grep AC — F1 재발 차단).
- AGENTS.md 디자인 절과 UI dev 3종·designer 본문이 무대별 게이트 분기(바이브=정의→즉시
  구현+명시 / SDD=사용자 확인)를 담는다(grep AC).
- AGENTS.md 계약 절·context-map 템플릿이 바이브 design.md 위치(`projects/<project>/design.md`)
  를 명시한다(grep AC).
- 파싱 19종 problems 0 불변, description 전부 불변(트리거 미도입 — 029 계승), 기존 테스트
  전체 green.

## Non-goals

- **신규 코드·신규 페르소나·신규 문서 종류** — 문구 개정만.
- **F4 경계 규약 변경** — 관찰 유지(Open q). 정적 서빙 같은 frontend-enabling 서버 훅의
  소유를 지금 못박지 않는다.
- **memo-api 놀이터의 미해결 드리프트 4건 정렬**(에러 포맷·UUID·spec.js·로그 배너) —
  테스트 산출물이지 localmind 규약이 아니다. 계약 원장에 추적된 상태로 둔다.
- **026 SDD 게이트 완화** — SDD 무대(specs/)의 "사용자 확인 후 실행"은 불변. 분기는
  바이브 무대에 한정.

## Constraints

- 오픈소스 위생(개인 절대경로 금지·평이한 한국어), 026~029 정본 스키마·description 불변
  원칙, 부분 문자열 함정 경계(신규 트리거 어휘 0) 전부 계승.
- 게이트 분기가 "SDD 게이트도 완화됐다"로 오독되지 않게 두 무대를 항상 병기한다.

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 특히 바이브 코딩으로 여러 도메인
페르소나를 쓰는 사용자 — 계약이 확실히 노트 폴더에 쌓이고, UI 사전 정의가 흐름을 끊지
않으면서도 항상 남는다.

## Risks

- **게이트 분기의 오독**(바이브 완화가 SDD까지 번짐) → 두 무대 병기 + AC로 SDD 문구
  불변 회귀 고정(앵커는 무대 고유 연속 문자열 "사용자 확인 후 실행"). AGENTS.md **바이브
  절의 기존 게이트 문구(026 게이트 우선 병기)도 함께 개정** — 빠뜨리면 바이브 에이전트가
  읽는 절에 옛 규칙이 남아 자기모순(리뷰 결함 2). 검증은 신설 연속 문자열("확인 규칙은
  무대별") 포함의 긍정 앵커로 — 원문은 라인랩 분절이라 부정 앵커는 공허 통과(잔존-A).
- **바이브 design.md의 품질 저하**(확인 생략으로 정의가 형식화) → 생성 사실 명시(맨 앞)로
  사후 검토 가능성 확보 + ux-reviewer가 design.md 대비 점검(기존 규율)으로 방어. 실전에서
  정의 품질이 떨어지면 확인 필수로 재론(Open q).
- **탁상 개정** → 이번 개정 자체가 실전 테스트 발견 기반이므로 위험 낮음. 다음 실전에서
  재관찰.
