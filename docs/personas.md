# 페르소나 에이전트 구성 (SSoT)

> **TL;DR** — localmind의 페르소나 위임 체계는 **기능 축 코어 5개 + 무대 확장 5개 + 도메인
> 스페셜리스트 9개, 총 19개**로 구성한다(2026-07-04 specs/026 디자인 무대 +2, 2026-07-05
> specs/028 바이브 코딩 무대 +9 — 도메인 구현 8종 + security-reviewer). 모델은 "실패 파장 × 난이도" 배치표(AGENTS.md)를 따르고, Codex(GPT 계열)는
> **크리틱의 교차 검증 백엔드**로만 먼저 도입한다. 런타임(지식 워크플로우) 개입은
> **최대 자동화**를 기본값으로 한다(비용 통제 장치는 specs/017에서 설계).
>
> 이 문서는 페르소나 **구성과 근거의 정본**이다. 스키마·배포는 [specs/016](../specs/016-persona-agent-registry/spec.md),
> 런타임 위임은 specs/017(예정), 오케스트레이션은 specs/018(예정)이 이 문서를 참조한다.
> 016 구현 후 이 구성을 레지스트리 파일(`agents/*.md`)로 실체화한다.

## 구성 원칙 (인터뷰 결정)

1. **코어 + 무대 확장 하이브리드**: 무대(지식 워크플로우 / SDD 워크플로우) 공통의 기능 축
   코어를 두고, 무대 전용 역할만 확장으로 더한다.
2. **소수 정예(4~6 코어)**: 넓은 역할·빠른 팬아웃. lane 침범은 프롬프트 규율(소유/비소유
   경계 명시)로 보완한다 — 기존 speckit 12개의 boundary preamble 패턴을 계승.
   *(2026-07-04 재조정 — specs/026: 코어 5는 불변으로 유지하고 **무대 확장만 3→5**로
   늘렸다(디자인 무대 designer·ux-reviewer). 단계×역할 매트릭스(10+) 기각 결정은
   "코어 증식 금지"의 의미로 유지된다 — 확장은 무대가 실제로 생길 때만, 삼권분립
   (정의/실행/검증)과 상호배타 lane을 지키는 조건으로.)*
   *(2026-07-05 재조정 — specs/028: 바이브 코딩 무대(명세 없이 가이드+대화로 진행하는 기술
   도메인 구현)에 **도메인 축** 9종을 추가했다. 기각된 매트릭스는 role×domain 곱(10×9=90)
   이었고, 이번 확장은 **역할 축 10종 불변 + worker의 구현 역할 하나에만 도메인 분화 8 +
   검증 lane에 보안 도메인 리뷰 1**이다 — 역할 축은 그대로다. 추가 도메인(게임·ML 등)의
   진입 문턱: 실제 반복 수요 + 서로소 어휘 확보 + 기존 lane 흡수 불가의 3조건.)*
3. **역할별 모델 지정**: 페르소나 정의가 대상 도구별 모델·추론 강도를 소유한다.
   표기는 별칭(opus/sonnet/haiku)을 기본으로 해 모델 버전업에 견디게 한다.
4. **구현-검증 독립성의 교차-모델 확장**: 크리틱은 Claude(Opus)와 Codex(GPT)를 함께 써서
   같은 모델 계열의 맹점 공유를 피한다.

## 무대별 개입 지도

### 지식 워크플로우 (localmind 런타임 — specs/017 범위)

기본값 = **최대 자동화** (2026-07-03 인터뷰 결정). 단, 017에서 비용 통제 장치(저비용 모델
우선, 예산·빈도 상한, 끌 수 있는 환경변수)를 함께 설계하는 것을 조건으로 한다.

| 지점 | 페르소나 | 개입 방식 |
|---|---|---|
| capture 직후 | curator | 자동 — 분류·태그·중복 후보 제안 |
| ask_brain 합성 | librarian | 자동(내장) — 현행 고정 ANSWER_MODEL을 페르소나로 교체 |
| ask_brain 답변 검증 | critic | 자동 — 출처 일치·hallucination 교차 검증(codex 포함) |
| 주기 리포트 | analyst | 자동(주기 실행) — 실패 질의(specs/004)·인덱스 품질 해석 |

### 디자인 무대 (specs/026 — 명시 위임만, 자동 개입 없음)

"정의 없이 구현 금지"의 디자인판: designer가 design.md(패턴·토큰·컴포넌트·프롬프트)를
완성·확인받은 뒤에만 worker가 구현, ux-reviewer가 design.md 대비 점검, critic이 최종 게이트.
트리거 어휘 배정(라우팅 tie 방지): designer="디자인 시스템·디자인 토큰·컴포넌트 정의·화면
정의", ux-reviewer="UX 리뷰·사용성·접근성·토큰 준수" — "설계"(architect)·"self-review"(스킬)·
"결함 검증/품질 게이트"(critic)는 이 둘의 description에 쓰지 않는다.

| 단계 | 페르소나 | 비고 |
|---|---|---|
| 디자인 사전 정의(design.md) | designer | 완성+사용자 확인 전 구현 금지 |
| UI 구현 | worker | design.md가 명세 — 없으면 반환 |
| 디자인 도메인 점검 | ux-reviewer | design.md 불일치 = 지적 대상 |
| 최종 게이트 | critic | 다운시프트 금지(기존 규율) |

### 바이브 코딩 무대 (specs/028 — 명시 위임만, 자동 라우팅 없음)

명세 없이 도메인 가이드(`guides/{domain}.md` — 완화 게이트: 있으면 따르고 없으면 일반
관례 + 명시)와 대화로 진행하는 기술 도메인 구현. worker(SDD 무대: spec/plan 확정 구현)와
구분된다. critic은 보안 도메인 트리거를 security-reviewer에 양도했다(최종 게이트에서
명백한 보안 결함은 여전히 잡음). 구현 8종은 opus — 바이브 코딩은 명세 부재가 기본값이라
"명세가 촘촘하지 않은 코딩 = Opus" 행에 해당(가이드 축적 후 다운시프트 재론).

| 단계 | 페르소나 | 비고 |
|---|---|---|
| 도메인 구현 | backend-dev·frontend-dev·ios-dev·android-dev·infra·data-platform·auth-dev·dba | 가이드 완화 게이트, UI는 design.md 우선 |
| 보안 도메인 리뷰 | security-reviewer | auth-dev 구현과 분리 lane |
| 최종 게이트 | critic | 다운시프트 금지(기존 규율) |

프로젝트 계약 저장소(specs/029 — `projects/<project>/`): context-map·용어집=architect,
api-contract=backend-dev, environments=infra(시크릿 값 금지). 소비는 UI dev 3종 + 전 도메인.
디자인 토큰은 design.md 정본에 포인터만(026 불변).

### SDD 워크플로우 (오케스트레이션 — specs/018은 self-review 행, 나머지 단계는 후속)

주 클라이언트 Claude Code가 오케스트레이터. Codex 위임은 `codex exec -p <persona>` 래핑
(+`--output-schema`/`-o` 파싱)이 기본 패턴.

| 단계 | 페르소나 | 비고 |
|---|---|---|
| goal/spec 인터뷰 | interviewer (+architect) | 크리틱이 spec 검토·Devil's Advocate 겸임 |
| plan 설계 | architect (+researcher) | |
| tasks·구현·테스트 | worker (+curator: 대량 기계작업) | Codex 병렬 워커는 **보류** — 크리틱 교차 운영 감 확보 후 재검토 |
| self-review·품질 게이트 | critic | Opus + Codex 교차. 다운시프트 금지 |
| 기술 조사 | researcher | 두괄식·출처 등급(T1~T4) 규율 적용 |

## 결정 이력 (2026-07-03 인터뷰)

- 기존 speckit 12개 체계: 불만은 적음 — 폐기가 아니라 새 무대·Codex까지 포괄하는 재도출.
  boundary preamble·모드 프로토콜 등 패턴은 계승 대상.
- 세트 구성: 무대별 별도 세트 ↔ 통합 단일 세트 중 **코어+확장 하이브리드** 채택.
- 입도: 단계×역할 매트릭스(10+) 대신 **소수 정예(4~6 코어)** 채택.
- 코어 구성: 기능 축 5개(A안) 채택 — 4코어 압축안(B안)은 한 페르소나가 여러 모델 티어를
  오가게 되어 기각.
- SDD 확장: 인터뷰어 1개 — 별도 QA리드는 두지 않고 크리틱이 AC 품질·테스트 전략 검토 겸임.
- Codex 범위: 교차 검증(크리틱)만 먼저. 병렬 구현 워커·조사 대조는 보류(조사 대조는
  기존 `ask` 도구로 이미 가능).
- 자동성: 명시 실행 기본(권장안) 대신 **최대 자동화** 채택 — 비용·예측가능성 트레이드오프를
  인지한 선택이며, 완화 장치 설계를 017의 요구사항으로 넘긴다.

## 후속

1. **specs/016** 구현 후 코어·확장 페르소나를 레지스트리 파일로 실체화(도그푸드 겸용) — 완료.
   specs/026이 templates/agents 동봉·시드로 신규 설치 부트스트랩까지 확장(026 당시 10종 → 028에서 19종).
2. **specs/017** (런타임 위임): 위 "지식 워크플로우 개입 지도" + 자동화 비용 통제가 범위.
3. **specs/018** (오케스트레이션): `codex exec` 크리틱 교차 검증 배관(self-review) —
   2026-07-03 인터뷰에서 1차 범위를 self-review로 좁힘. **SDD 전 단계 팬아웃은 후속
   스펙(019+)으로 이관**(018 goal Non-goals 참조).
4. 페르소나별 상세 프롬프트(소유/비소유 경계, 출력 형식)는 레지스트리 파일 본문에서
   관리한다 — 이 문서는 구성·근거만 소유한다.
