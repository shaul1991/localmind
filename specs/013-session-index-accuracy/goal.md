# Goal: Session & Index Accuracy (세션·색인 정확성)

## Background — 배경

2026-07-03 적대적 코드 리뷰(전수 평가)에서 **데이터 정확성**을 훼손하는 결함 묶음이
발견됐다. localmind의 두 핵심 약속 — "대화 맥락이 섞이지 않는다"(세션)와 "저장한 노트는
검색된다"(second-brain) — 을 조용히 깨뜨리는 것들로, 모두 에러 없이 틀린 결과를 내는
유형이라 사용자가 알아챌 방법이 없다:

- **세션 혼입(높음)**: explicit 세션 경로가 OpenAI `user` 필드를 세션 id로 받는데, 많은
  클라이언트가 모든 대화에 같은 `user` 값을 보낸다. resumable 판정이 메시지 **개수만**
  비교(내용 미검증)해서, 같은 id의 새 대화가 이전 대화의 CLI 세션에 접합돼 **이전 대화의
  맥락으로 새 대화에 답**할 수 있다. auto 모드의 prefix 해시 방어가 explicit 경로엔 없다.
- **색인 유실(높음)**: 청크 분할이 2000자 초과 문단을 분할하지 않고 **잘라 버려**, 긴
  문단(붙여넣은 로그·긴 산문·표)의 꼬리가 영구히 검색·RAG에서 누락된다.
- **주변 결함(중간)**: `delete_note`가 .md 외 임의 파일(인덱스 파일 포함)을 대상 삼을 수
  있음, 임베딩 모델 교체 시 차원 불일치가 조용한 쓰레기 검색 결과로 이어짐, 대화 중간에
  tools가 추가되면 함수호출이 침묵 실패, CLI가 세션 id를 안 준 경우의 무방어, 같은 초의
  동일 제목 캡처 덮어쓰기, 여러 MCP 프로세스(Claude Desktop + Claude Code + Cursor 동시
  사용) 간 인덱스 마지막-쓰기-승리 경합.

이 결함들은 전부 **테스트가 없는 지대**(HTTP 라우트·백엔드 파싱·brain의 청크/삭제 경로)에서
나왔다.

## Problem — 문제

- 세션과 색인은 localmind의 신뢰 기반인데, 위 결함들은 모두 **조용히** 틀린다 — 크래시가
  아니라 "맥락이 미묘하게 섞인 답", "있는데 안 찾아지는 노트"로 나타나 비개발자 사용자는
  원인을 특정할 수 없다.
- 같은 위협(내용 불일치 세션 접합)에 대해 auto 모드에는 방어가 있고 explicit 모드에는
  없는 비대칭이 존재한다.
- 다중 MCP 호스트 동시 사용은 README가 권장하는 사용 패턴("도구를 바꿔 써도 같은 두뇌")
  인데, 인덱스 계층이 이를 안전하게 지원하지 않는다.

## Objective — 목표

세션과 색인의 정확성 결함을 제거한다: (1) explicit 세션 경로에 auto 모드와 같은 수준의
내용(prefix) 검증을 추가해 대화 혼입을 차단하고, 세션 수명주기의 무방어 지점(빈 CLI 세션
id, tools 변경)을 방어한다. (2) 청크 분할의 내용 유실을 0으로 만들고, 색인의 자기 기술
정보(임베딩 모델·차원)를 기록해 모델 교체를 안전하게 하며, 다중 프로세스 동시 사용에서
색인 갱신이 유실되지 않게 한다. (3) `delete_note`·`capture_note`의 파일 조작 경계를
조인다.

## Expected outcome — 기대 결과

- 어떤 세션 모드에서도 내용이 다른 대화가 기존 CLI 세션에 접합되지 않는다.
- 노트에 적힌 모든 텍스트가 청크 어딘가에 포함된다 — "저장했는데 검색이 못 찾는" 구간이
  구조적으로 없다.
- 임베딩 모델을 바꿔도 검색이 조용히 망가지지 않는다(자동 재색인 또는 명확한 안내).
- `delete_note`는 마크다운 노트만 대상으로 한다 — 인덱스·설정 파일은 지목해도 거부된다.
- MCP 호스트 여러 개를 동시에 써도 각자의 색인 갱신이 서로를 덮어쓰지 않는다.

## Success metrics — 성공 지표

- 같은 explicit id로 내용이 다른 대화를 보내면 resume되지 않고 새 세션으로 처리된다
  (혼입 재현 시나리오가 실패 테스트 → 통과로 고정).
- 빈 줄 없는 5,000자 문단을 캡처한 뒤 문단 끝부분의 고유 문구가 `search_notes`로 검색된다.
- 비-.md·숨김 경로 삭제 시도가 거부되고 파일이 그대로 남는다.
- 프로세스 2개가 서로 다른 파일을 색인한 뒤 양쪽 임베딩이 모두 인덱스에 존재한다.

## Non-goals — 비목표

- HTTP 라우트(`chat.ts`/`messages.ts`) 전반의 테스트 커버리지 확충 자체는 목표가 아니다 —
  이번 AC 검증에 필요한 만큼만 추가한다(전면 커버리지는 별도 작업).
- `chat.ts`/`messages.ts`의 중복(~150줄) 통합 리팩터는 범위 밖 — 외과적 변경 원칙.
- 인덱스 성능 개선(mtime 1차 필터, 바이너리 벡터 등)은 범위 밖(BACKLOG B2 부수 항목).
- tool_calls 턴에서 auto resume이 안 되는 비효율(정확성은 유지됨)은 범위 밖.
- 세션 저장소의 프로세스 간 공유(인메모리 → 파일/DB)는 범위 밖 — 세션은 프로세스당
  하나뿐인 HTTP 서버(:8787)에만 있어 경합이 없다.

## Constraints — 제약

- 기존 문서화된 동작·시그니처 보존: `user`/`session_id`/`metadata.user_id`/
  `x-localmind-session`은 **계속 세션 id 소스로 인정**한다(제거가 아니라 검증 추가 —
  2026-07-03 사용자 결정). MCP 도구 입력 스키마도 유지.
- 청크 분할 변경은 인덱스 버전(bump)으로 1회 재색인을 유도한다 — 자동·안내 동반
  (임베딩 재계산 비용이 들므로 사용자에게 이유를 알린다).
- 다중 프로세스 안전장치는 **외부 의존성 없이**(node 내장 fs 원자 연산) 구현한다.
- 에러·안내 메시지는 비개발자가 이해할 수 있는 평이한 한국어로.

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함) — 특히 LibreChat 등 `user` 필드를
  보내는 클라이언트 사용자, 긴 문서를 붙여넣어 캡처하는 사용자, MCP 호스트를 여러 개
  동시에 쓰는 사용자

## Risks — 리스크

- prefix 검증 추가로, 히스토리를 편집해 보내는 클라이언트의 explicit 세션이 fresh로
  떨어질 수 있음 — 정확성이 우선(auto 모드와 동일한 트레이드오프)이며 틀린 답이 아니라
  토큰 절약 감소일 뿐임을 문서에 명시해 완화.
- 파일 락 도입 시 락 보유 프로세스가 죽으면 다른 프로세스가 영원히 대기할 수 있음 —
  stale 타임아웃(락 파일 나이 기준 강제 해제)으로 완화. 데드락 시나리오를 AC로 고정.
- 인덱스 버전 bump로 기존 사용자 전원이 1회 전체 재색인을 겪음(CPU 임베딩이면 수 분) —
  기존 v1→v2 마이그레이션과 동일한 기제이며 안내 메시지로 완화.
- 청크 분할 방식 변경이 검색 품질(청크 경계)에 미묘한 영향을 줄 수 있음 — 유실 0이
  우선이고, 품질 튜닝은 후속.
