# 024 — 라벨↔경로 바인딩 (what)

goal: [goal.md](goal.md) · plan: [plan.md](plan.md)

## 용어

- **바인딩(binding)**: 라벨 → 정규화된 원본 폴더 경로의 기록. `BrainIndex.bindings?`
  (선택 필드 — 색인 버전 불변, 구버전 코드는 무시).
- **재바인딩(rebind)**: 대상 라벨의 기록 바인딩 경로 ≠ 이번 실행의 현재 경로인 상태.
- **대상/부재 라벨·후퇴(fallback)**: 020 정의 계승 — 대상 = readdir 성공 라벨, 부재 =
  등록됐으나 readdir 실패, 후퇴 = `REINDEX_FALLBACK=1` 또는 NOTES_DIR 완전 부재.
  후퇴의 프루닝·재바인딩 판정은 020 FR-3대로 **전면 보류**가 우선한다.

## FR

- **FR-1 (바인딩 기록)** *(goal: O1)* — 재색인은 **대상 라벨**의 현재 폴더 경로를
  `bindings[label]`에 기록한다. 경로 정규화는 존재 시 `fs.realpathSync`(심링크 해소, 셸
  `canon_path`의 `pwd -P`와 동일 규칙), 부재 시 lexical(`path.resolve` + `~` 확장).
  **부재 라벨의 기존 바인딩은 덮어쓰지 않는다**(보존). **후퇴(REINDEX_FALLBACK) 중에는
  bindings를 기록·갱신하지 않는다** — 후퇴 시 FOLDERS의 경로는 폴백(백업 미러 등)이라,
  기록하면 다음 정상 재색인이 거짓 재바인딩으로 오판된다(판정뿐 아니라 기록도 후퇴 범위
  밖 — 020 "구성 신뢰 불가" 원칙). 기록 바인딩이 없는 라벨(첫 실행·구버전 색인)은
  재바인딩이 아니며 현재 경로를 기록하고 기존 프루닝(020 FR-1)을 그대로 적용한다.
- **FR-2 (재바인딩 구분·보존)** *(goal: O1, 020 알려진 한계 해소)* — 대상 라벨의 기록
  바인딩 경로가 현재 경로와 다르면 **재바인딩**으로 판정한다. 재바인딩 라벨의 미스캔(seen
  아님) 기존 항목은 "폴더 내 삭제"로 **프루닝하지 않고 보존**한다. 현재 경로의 파일은 같은
  라벨로 정상 색인·추가한다 — 옛/새 경로에 **같은 relpath**가 있으면 키가 같아 새 내용으로
  갱신(재임베딩)된다: 보존이 보장되는 것은 "옛 경로에만 있던 relpath" 항목이다(색인 키가
  `label/relpath`라 물리 경로를 구분하지 못하는 데이터모델의 성질 — 정직 표기). 명시적
  재색인 요약에 라벨·기록 경로·현재 경로·보존 건수·수락 방법(FR-3)을 평이한 한국어로
  안내한다. 후퇴 상태에서는 재바인딩 판정을 하지 않고 020 FR-3(전면 보류)이 우선하며,
  재바인딩 안내 대신 020의 보류 안내를 낸다.
- **FR-3 (재바인딩 수락 탈출구)** *(goal: O1)* — `REINDEX_ADOPT_REBIND=<라벨,...>` 지정
  시, 그 라벨에서 **이번 스캔에 포함되지 않은(seen 아님) 기존 항목을 제거**하고
  `bindings[label]`을 현재 경로로 갱신한다(재임베딩 비용 수락). 제거 판정은 seen 여부다 —
  색인 항목에는 물리 경로 정보가 없어 "옛 경로 소속"을 경로로 선별할 수 없으며, 재바인딩
  상태에서 seen 아님 = 옛 경로에만 있던 relpath와 동치다. **수락은 그 라벨의 현재 경로
  스캔(readdir)이 성공한 경우에만 적용한다** — 새 경로가 미마운트 등으로 안 읽히면 seen이
  비어 "seen 아님 제거"가 라벨 전량 삭제가 되므로(020이 없애려던 missing≠deleted의 재도입),
  이때는 보존하고 사유를 안내한다(대상 라벨 판정 `scanned` 안에서만 처리). 규칙: 항목별
  공백 트림, 빈 값은 조용한 no-op, 재바인딩 상태가 아닌 라벨 지정은 무시하고 사유 안내,
  후퇴 중에는 전부 무시. `make reindex`가 이 변수를 **명시적으로** 전달한다(020 AC-11의
  `REINDEX_PRUNE_LABELS`와 동일 배선 규율 — make 암묵 export 의존 금지).
- **FR-4 (additive 하위호환)** *(goal: Non-goals, Risks)* — `bindings`는 선택 필드다:
  색인 버전을 올리지 않고, 필드가 없는 색인(구버전·첫 실행)은 "바인딩 미기록"으로
  동작한다(FR-1의 첫 실행 규칙). reload-merge는 `ours.bindings[l] ??= disk` 규칙으로
  병합한다(내 기록 우선, 없는 라벨만 디스크에서 채움). 구버전 코드가 저장하며 필드를
  탈락시켜도 다음 신버전 재색인이 재기록한다(자가 수렴 — 색인 항목 유실 아님).

## Acceptance Criteria

각 AC는 테스트와 1:1 매핑한다. 하니스는 기존 관례(자식 프로세스 격리 + HTTP 임베딩 스텁
+ `calls()` 계측 + `BRAIN_INDEX`/`NOTES_DIR` 주입 + `indexKeys`)를 쓴다. 경로 예시는
플레이스홀더(`/home/<user>/...`)를 쓴다.

- **AC-1 (FR-1·2)** Given 라벨 `a`가 경로 `/a`로 바인딩된 색인이 있고(옛/새 경로의
  relpath는 겹치지 않게 구성), When `a`를 `/b`로 재바인딩해(NOTES_DIR에서 `a`의 경로 변경)
  재색인하면, Then 옛 경로에만 있던 `a` 항목이 **1건도 제거되지 않고** 보존되고, `/b`의
  파일이 `a` 라벨로 추가되며, 요약에 재바인딩·기록 경로·현재 경로·보존 건수·수락
  명령(`REINDEX_ADOPT_REBIND=a`)이 안내된다.
- **AC-2 (FR-2 회귀 고정)** Given 바인딩 경로가 불변인 대상 라벨에서 파일 1개를 삭제하고,
  When 재색인하면, Then 그 키만 프루닝된다(재바인딩 오판 없음 — 020 AC-4 회귀 고정).
- **AC-3 (FR-1 첫 기록)** Given 바인딩이 없는 기존 색인(구버전·첫 실행)에서, When 같은
  구성으로 재색인하면, Then 재바인딩으로 오판되지 않아 어떤 항목도 보존 오판·프루닝되지
  않고, 대상 라벨의 바인딩이 현재 경로로 기록된다.
- **AC-4 (FR-3 수락)** Given AC-1의 재바인딩 상태에서, When `REINDEX_ADOPT_REBIND=a`로
  재색인하면, Then 이번 스캔에 포함되지 않은 `a` 라벨 기존 항목(= 옛 경로에만 있던
  relpath 집합)이 제거되고 `bindings[a]`가 `/b`로 갱신되며, 이후 재색인의 요약에 `a`
  재바인딩 안내가 더는 나오지 않는다.
- **AC-5 (FR-3 경계)** Given 재바인딩 상태가 아닌 라벨 `a`와 색인에 없는 라벨 `x`를,
  When `REINDEX_ADOPT_REBIND=" a , x ,"`(공백·빈 항목 포함)로 재색인하면, Then 아무것도
  제거되지 않고 각각 무시 사유가 안내되며, 빈 값(`REINDEX_ADOPT_REBIND=`)은 안내 없이
  no-op이다.
- **AC-6 (FR-3 미마운트 가드)** Given AC-1의 재바인딩 상태에서 새 경로 `/b`의 디렉토리를
  제거(readdir 실패)하고, When `REINDEX_ADOPT_REBIND=a`로 재색인하면, Then `a` 라벨의
  어떤 항목도 제거되지 않고 `bindings[a]`도 갱신되지 않으며, "폴더를 열 수 없어 수락을
  보류했어요" 류의 사유가 안내된다.
- **AC-7 (FR-1 후퇴 우선)** Given 재바인딩이 발생할 구성이라도 후퇴(`REINDEX_FALLBACK=1`)
  이면, When 재색인하면, Then 020 FR-3대로 삭제·프루닝이 전면 보류되어 어떤 라벨도
  제거되지 않고, 재바인딩 수락(FR-3)도 무시되며, **`bindings`가 1건도 기록·변경되지
  않고**(FR-1 후퇴 제외 — 폴백 경로 오염 방지), 020의 "삭제 반영 보류" 안내가 나온다.
- **AC-8 (FR-4 병합)** Given 두 프로세스가 같은 색인을 로드한 뒤 각자 다른 라벨을
  색인·저장하면, Then 최종 색인의 `bindings`에 두 라벨의 경로가 모두 남는다(reload-merge
  `??=` 규칙 — 013 다중 프로세스 패턴의 바인딩판).
- **AC-9 (배선)** Given `make reindex REINDEX_ADOPT_REBIND=<라벨>`을 실행하면, Then
  reindex.sh가 변수를 재색인 프로세스에 명시적으로 전달한다(020 AC-11 규율).

## Open questions

1. **재바인딩 기본 정책** — 보존(현 결정: 파괴 금지)이 기본이지만, 라벨을 자주 옮기는
   사용자에겐 매번 `REINDEX_ADOPT_REBIND` 요구가 번거로울 수 있다. "자동 수락 + 시끄러운
   1회 안내"로 뒤집을지 사용 관찰 후 재검토(마운트 경로 변동 오판 위험이 근거).
2. **doctor 연계** — 재바인딩 상태(보존 중인 라벨)를 022의 doctor 라벨 안내에 함께 표시할지.
3. **023과의 순서** — additive라 순서 무관(023 spec Open Q7과 동일 내용). 동시 구현만
   회피(같은 저장 경로·요약 채널).
