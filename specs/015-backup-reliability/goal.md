# Goal: Backup & Recovery Reliability (백업·복구 신뢰성)

## Background — 배경

localmind의 핵심 약속 중 하나는 "노트·기억이 안전하게 쌓이고, 새 기기에서 통째로
복구된다"이다(README 백업·복구 절, ROADMAP "파일이 정본"). 2026-07-03 인프라 리뷰에서
이 약속을 **조용히** 깨는 결함들이 발견됐다:

- **백업의 인질 구조(중간)**: `make backup`의 첫 단계인 memory-export가 실패하면(스택
  꺼짐이 가장 흔한 원인) 이후의 노트 커밋·push·개인 설정(extras) 백업까지 전부 중단된다.
  cron 자동 백업 사용자는 Docker를 꺼둔 몇 주간 **노트 백업도 함께 무력화**되는데 로그
  파일 외엔 알 방법이 없다. 같은 저장소의 `backup-init`은 export 실패를 경고로 넘기고
  노트를 백업한다 — 동작이 갈라져 있다.
- **복구의 누락(중간)**: README가 "통째 복구"로 안내하는 `make recover`가
  `BACKUP_EXTRA_FILES`로 백업한 개인 설정 파일을 복원하지 않는다(`make restore`에만
  배선됨). 사용자는 복구 완료 메시지를 믿고 진행한 뒤에야 설정 누락을 발견한다.
- **파괴적 명령의 가드 공백(중간)**: `make purge`의 노트 삭제 가드가 빈값·`/`·`$HOME`만
  거부해 `BACKUP_DIR=/tmp` 같은 값은 `rm -rf`가 통과한다. Docker 데몬이 꺼져 있으면
  볼륨 삭제는 조용히 실패한 채 노트만 지워지고 "완전 제거 완료"가 출력된다.
- **주변 결함(중간~낮음)**: MCP 등록이 선-삭제 후-추가라 add 실패 시 기존 등록까지 소실,
  backup-cron이 등록 시점의 커스텀 변수(`BACKUP_DIR`·`BACKUP_EXTRA_FILES`)를 크론
  라인에 싣지 않음 + 두 기기 사용 시 non-ff push 영구 실패의 무알림, `make up` 헬스
  대기가 정작 :8787(chat)을 확인하지 않고 curl 타임아웃도 없음(BACKLOG B4), `.env`가
  기본 umask(644)로 생성된 뒤 OAuth 토큰이 들어감.

## Problem — 문제

- 백업·복구는 실패가 **한참 뒤에**(디스크 사고, 새 기기) 드러나는 영역이라, "조용한
  부분 실패"가 가장 위험하다 — 성공 메시지와 실제 상태가 다르면 안전망이 안전망이
  아니다.
- 파괴적 명령(purge)은 실패 허용 설계(경고 후 계속)가 비가역 단계(rm -rf) 앞에서
  게이트 없이 이어진다 — 절반만 지워진 상태를 사용자가 알 수 없다.
- 같은 목적의 코드(backup vs backup-init의 export 처리)가 서로 다른 실패 정책을 갖고
  있어 "한 번은 됐는데 왜 지금은 안 되지"를 만든다.

## Objective — 목표

백업·복구·제거의 실제 동작을 약속(문서·출력 메시지)과 일치시킨다: (1) `make backup`이
부분 실패에서도 가능한 것(노트·extras)은 백업하고, 안 된 것(메모리)을 명확히 알린다
(2026-07-03 사용자 결정: 경고 후 계속 + 비정상 종료 코드). (2) `make recover`가 extras
복원까지 포함해 "통째 복구"가 되게 한다. (3) purge의 비가역 단계에 실질 가드를 세우고
부분 완료를 정직하게 보고한다. (4) MCP 등록·cron 등록·기동 헬스 대기·`.env` 권한의
신뢰성 구멍을 메운다.

## Expected outcome — 기대 결과

- 스택이 꺼진 상태의 `make backup`(수동·cron 공통)에서도 노트·extras는 백업되고,
  "메모리는 백업 안 됨(스택 꺼짐)"이 출력·종료 코드로 드러난다.
- 새 기기 `make recover` 한 번으로 노트·기억·개인 설정 파일이 모두 복원된다.
- 노트 폴더로 지정될 수 없는 경로(`/tmp` 등 홈 밖)에 대한 purge가 거부되고, 볼륨 삭제가
  실패했으면 "완전 제거"가 아니라 남은 것을 알려준다.
- MCP 재등록이 실패해도 기존 등록이 사라지지 않는다.
- cron 백업이 등록 시점의 백업 설정 그대로 돈다.
- `make up`이 "준비 완료"라고 말하면 chat(:8787)이 실제로 응답하는 상태다.

## Success metrics — 성공 지표

- 스택 다운 상태에서 `make backup` 실행 → 노트 커밋 생성 + 비0 종료 코드 + 메모리 실패
  안내(셸 테스트로 고정).
- extras를 포함한 백업 repo에서 `make recover` → extras 파일이 원위치에 복원됨.
- `BACKUP_DIR=/tmp NOTES=1 FORCE=1 make purge`가 삭제 없이 거부됨(셸 테스트로 고정).
- add가 실패하는 조건에서 mcp-install 실행 → 기존 등록이 그대로 남아 있음.

## Non-goals — 비목표

- 백업 대상 확장(인덱스·DB 백업 등)은 범위 밖 — "파일이 정본, 파생은 재생성" 철학 유지.
- non-ff push의 자동 병합·충돌 해결은 범위 밖 — 알림(로그에 원인·해결 안내)까지만.
  두 기기 동시 운용의 동기화 전략은 별도 주제.
- OS 키체인 연동(BACKLOG B1의 본체)은 범위 밖 — 이번엔 `.env` 파일 권한(600)까지만.
- 알림 채널(메일·데스크톱 알림) 추가는 범위 밖 — 종료 코드·로그 메시지 층위까지만.
- `scripts/up.sh`의 빌드 출력 스트리밍 개선은 다루지 않되, 빌드 실패 시 사용자가 실제
  에러를 볼 방법 안내는 포함한다.

## Constraints — 제약

- 모든 신규·수정 셸 로직은 기존 관례대로 `scripts/*.test.sh`(임시 디렉토리, 라이브
  스택·네트워크 불필요)로 검증 가능해야 한다 — CI에서 돈다.
- 기존 make 인터페이스(타깃 이름·변수)는 유지 — 동작의 신뢰성만 고친다.
- 비대화(cron·CI) 환경과 대화 환경 모두에서 동작 — 기존 `FORCE=1`/확인 토큰 관례 유지.
- 안내 메시지는 비개발자가 이해할 수 있는 평이한 한국어로.
- backup의 부분 실패 정책은 `backup-init`과 **통일**한다(같은 일은 같은 방식으로).

## Stakeholders — 이해관계자

- 단일 사용자(설치한 개인 누구나 — 비개발자 포함) — 특히 cron 자동 백업에 의존하는
  사용자, 기기를 옮기는 사용자, 스택을 지웠다 다시 까는 사용자

## Risks — 리스크

- "경고 후 계속" 정책이 실패를 덜 눈에 띄게 만들 수 있음 — 비0 종료 코드 + 실행 말미의
  실패 요약(무엇이 안 됐고 어떻게 하는지)으로 완화. cron 로그에서 grep 가능한 고정
  문구를 유지.
- purge 가드 강화가 정당한 커스텀 경로(홈 밖에 노트를 두는 사용자)를 오차단할 수 있음 —
  홈 밖 경로는 일괄 거부가 아니라 추가 확인(명시적 재확인 토큰)으로 완화할지 spec에서
  확정.
- recover에 extras 복원을 배선하면 복원 대상 파일이 기존 파일을 교체함 — 기존
  restore-extras의 `.bak-<타임스탬프>` 보존 규칙이 recover 경로에서도 동일하게
  작동해야 한다(specs/006 AC 재사용).
- up.sh에 :8787 대기를 추가하면 첫 기동(이미지 빌드·모델 pull)에서 대기 시간이 길어질
  수 있음 — 타임아웃과 진행 안내(`make logs`)로 완화.
