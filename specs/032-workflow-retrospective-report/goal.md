# Goal: 워크플로우 회고 리포트 — 작업 방식의 지속 개선 루프 (why)

spec: [spec.md](spec.md) · plan: [plan.md](plan.md)

## Background

004가 실패 질의 분석(query-report)을, 017이 analyst 페르소나의 주기 리포트 위임을
(make report → reports/ 노트), 016·026·028이 페르소나 레지스트리·무대 확장을 세웠다.
이들은 모두 **결과물(노트·검색·코드)의 품질**을 관측한다 — 그러나 그 결과물을 만드는
**작업 방식 자체**(SDD 흐름에서 사용자 개입이 어디에·왜 드는가)를 관측하는 루프는 없다.

2026-07-05, 019~030 스프린트 회고를 **수동 1회** 수행했다. 발견:

- 사용자 개입 약 36회 중 **60%가 기계적 반복**(승인 루프 — "커밋해도 될까?" 류).
- **가치의 80%는 방향·결정에서** 나왔다(어떤 스펙을 열까, 어떤 트레이드오프를 택할까).
- 이 한 번의 회고에서 규약 3건(clean→커밋 완료 정의 · CI 감시 상시화 · 결정 로그)과
  031 스펙(기기 동기화)이 **즉시** 산출됐다 — AGENTS.md 규약 7·결정 로그 규약이 그 결과다.

즉, 회고는 실측된 고수익 활동이다. 사용자 승인(2026-07-05): 이 수동 회고를
**지속 루프**로 만든다.

## Problem

- 회고가 수동 1회성이라 재현·주기화되지 않는다 — 다음 스프린트의 개선 신호가 휘발한다.
- 반복되는 기계적 개입(승인 루프 등)이 어디에 몰리는지 매번 사람이 기억으로 재구성한다 —
  자동화 후보를 놓치거나, 근거 없이 성급히 규약을 바꾼다.
- 스펙마다 흩어진 미해결 Open questions에 "재론 조건"이 적혀 있지만, 그 조건의 도래를
  아무도 주기적으로 되짚지 않는다 — OQ가 잊힌다.
- 결정 로그 노트(AGENTS.md 규약으로 second-brain에 쌓이기 시작)를 한자리에 모아
  "왜 그렇게 정했더라"를 회고 관점으로 되짚을 도구가 없다.

## Objective

작업 방식을 주기적으로 되짚는 **회고 리포트**를 만든다 — 004/017 리포트와 동형인
2단 구조(기계 집계 스크립트 + analyst 해석 위임)로, `make retro`(명시 실행)와
선택적 주기 옵션(`make retro-cron`)로 구동한다. 1차 입력은 **프록시 신호만** 쓴다:
git log(구간)·specs OQ 전수·결정 로그 노트·query-report 집계·guides/·projects/ 인벤토리.
출력은 second-brain 회고 노트(reports/ 관례 계승) — 작업 패턴 관측 · 자동화 후보(반복
임계 통과분) · OQ 대시보드 · **사용자 결정 대기 액션 리스트**를 담는다.

핵심 안전 원칙: **회고는 제안까지만 한다.** AGENTS.md·페르소나·규약의 실제 개정은
사용자 결정 + SDD 스펙을 경유한다(자기 개정 루프 금지). 이 게이트를 FR·AC로 고정한다.

### 신규 페르소나를 만들지 않는 근거 (028 진입 3조건)

028은 새 도메인 페르소나의 진입 문턱을 **실제 반복 수요 + 서로소 어휘 + 기존 lane
흡수 불가**의 3조건으로 못박았다. 회고는 이 중 **"기존 lane 흡수 불가"를 충족하지
못한다** — analyst의 lane이 정확히 "로그·지표 해석과 개선 제안"이고, 회고는 그 해석
대상을 검색 로그에서 **작업 방식 프록시 신호로 넓힐 뿐**이다. 따라서 신규 페르소나가
아니라 **analyst 주기 리포트의 확장**으로 구현한다(analyst 정본 소유에 회고 해석을 추가).

## Success metrics

- `make retro`가 픽스처(가짜 git log·fixture specs·fixture 노트 폴더)에서 회고 노트를
  결정적으로 생성한다 — 기계 집계 4파트(commit/spec cadence · OQ · 결정 노트 · 검색
  집계)가 fixture 기대치와 1:1 일치(AC로 고정).
- 자동화 후보 승격은 **동일 패턴 3회 이상 관찰**분에만 붙는다(2회는 "관찰 중" — 오늘
  회고의 판정 기준 계승. AC로 고정).
- 회고 스크립트는 `<노트 폴더>/reports/` **바깥의 어떤 파일도 쓰거나 고치지 않는다**
  (AGENTS.md·agents/*·specs/* 불변). 강제는 단일 가드 함수(경로 prefix 검사 + 유일 쓰기
  지점) + 결정적 테스트 단언이다 — Node 프로세스의 진짜 샌드박스가 아님을 정직하게
  명시한다(리뷰 D4 — "물리 제한" 과장 제거).
- analyst 페르소나가 없어도 기계 집계 노트가 정상 생성된다(017 위임 패턴 — 해석만 생략).
- 결정 로그 노트가 frontmatter **tags의 `decision`** 관례로 결정적으로 수집된다 —
  capture_note에 선택적 tags 파라미터를 추가해 생성 경로부터 실현(리뷰 D2: 기존
  capture는 type: 필드를 낼 수 없음). AGENTS.md 결정 로그 규약에 식별 관례 보강.

## Non-goals

- **세션 트랜스크립트 마이닝**(1차 제외). 개입 하나하나의 원문·승인 루프 카운트는
  트랜스크립트에 있으나 크고·민감하고·비싸다. 1차는 프록시 신호만 쓰고, 프록시 부족이
  판명되면 2차에서 검토한다(Open question). 이 때문에 "기계적 개입(승인 루프) 정밀
  카운트"는 1차 노트의 능력 밖임을 노트가 정직하게 명시한다.
- **자기 개정 자동화** — 회고가 AGENTS.md·페르소나·규약을 직접 고치는 것. 제안까지만.
- **OQ 재론 조건의 자동 도래 판정** — 각 OQ에 적힌 조건 문구를 표면화할 뿐, "조건이
  도래했다"는 판정은 사람/analyst의 몫이다(자동 판정 과장 금지).
- **회고 → 스펙 자동 초안 생성** — 액션 리스트를 goal/spec 초안으로 자동 전개하는 것은
  범위 밖(Open question).
- **결정 노트 소급 태깅** — 기존에 쌓인 결정 노트에 `decision` 태그를 자동으로 붙이지
  않는다(관례는 이번 이후 신규 노트부터 — 정착 관찰).
- **상주 프로세스** — 회고는 명시 실행 + 선택적 cron이지, 백그라운드 데몬이 아니다.

## Constraints

- 오픈소스 대상 — 특정 개인·특정 저장소를 사용자로 특정하지 않는다. 예시·AC에 실제 개인
  절대경로 금지(플레이스홀더). 비개발자가 이해할 평이한 한국어(에러·노트 문구).
- 2단 구조: 순수 집계 모듈(결정적·테스트 가능) + 얇은 진입점(IO·git spawn·페르소나
  위임) — query-report.ts / query-analysis.ts / report-note.ts 관례를 그대로 계승한다.
- ⚠️ **회고 노트는 reports/에 쌓이고 검색·백업에 잡힌다**(docs/agents.md §4 계열 주의 — 재검 R3 정정).
  노트에는 커밋 해시·스펙 번호·결정 노트 제목·검색 질의 요약이 담긴다 — 민감 신호이므로
  로컬/백업 저장소 커밋 사실을 노트 문구로 고지한다.
- 모델 배치(AGENTS.md 실패 파장 × 난이도): 기계 집계·OQ 파싱·결정적 테스트 = **sonnet**
  (통과 기준이 명확한 잘 명세된 루틴). 해석 = **analyst 페르소나(sonnet)**. 스펙·이 설계
  판단 = opus 🔖. 최종 게이트(critic)는 다운시프트 금지.
- Simplicity First: 형태소 분석·트랜스크립트 파서 같은 무거운 의존성 도입 금지(004 계승).
  guides/·projects/ "변경 이력"은 백업 repo git archaeology(백업 커밋 노이즈)를 파지 않고
  **인벤토리 + 최종 수정 시각**으로 축약한다(부족 판명 시 확장 — Open question).

## Stakeholders

단일 사용자(설치한 개인 누구나 — 비개발자 포함). 특히 SDD/바이브 흐름을 반복하며 "내
작업 방식의 어디가 반복 노동이고 어디가 진짜 결정인가"를 주기적으로 되짚고 싶은 사용자.

## Risks

- **프록시-신호 한계** → 1차 입력은 개입의 *결과물*(커밋·스펙·결정 노트)을 보지, 개입의
  *과정*(승인 루프 왕복)을 직접 세지 못한다. 오늘 회고가 잡아낸 "60% 기계적 개입"은
  트랜스크립트에서 나온 수치다. 완화: 노트가 이 한계를 명시하고, 승인 루프 카운트를
  트랜스크립트 마이닝 2차의 트리거 조건으로 못박는다(Open q).
- **자기 개정 폭주** → 회고가 규약을 바로 고치면 검토 없는 변화가 축적된다. 완화: 단일 가드
  함수 + 결정적 테스트 단언(진짜 샌드박스 아님 — FR-7 정직화) + 액션 리스트를 "제안·
  사용자 결정 대기"로만 렌더.
- **성급한 자동화** → 1~2회 관찰로 자동화 후보를 올리면 소음이다. 완화: 3회 임계(FR-4).
- **결정 노트 식별 관례 미정착** → tags `decision`은 이번에 신설하는 관례라 초기엔
  수집량이 0에 가깝다(소급 태깅 안 함). 초기 수집량이 낮았을 진짜 원인(생성 경로가
  식별 필드를 낼 수 없었음)은 capture tags 파라미터로 해소(리뷰 D2). 완화: 표본 부족을
  노트가 명시(analyst 해석 생략), 정착을 관찰(Open q).
- **탁상 설계** → 회고 노트 스키마·임계는 첫 실전 회고에서 개정 전제. analyst 해석 품질은
  CI로 검증 불가(페르소나 규율 의존 — 026~031 정직한 한계 계승).
- **cadence 부정합** → report(주 1회)와 retro(스프린트 경계 — 불규칙)를 한 cron에 묶으면
  대부분의 주에 "데이터 부족" 회고를 양산한다. 완화: 별도 cron·별도 관심사(FR-8), 자동
  주기 기본값은 Open question으로 유보.
