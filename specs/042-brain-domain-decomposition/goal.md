# Goal: Brain 도메인 책임 분해

## Background - 배경

localmind의 핵심 second-brain 기능은 폴더 설정, 색인 영속화, 증분 색인, 검색, 캡처,
답변 합성·검증, 질의 이벤트 기록까지 하나의 큰 모듈에 모여 있다. 이 모듈은 오랜 기간
v4→v5 무재임베딩 이행, 다중 프로세스 병합, 안전한 프루닝·재바인딩, 소프트 삭제 같은
중요한 데이터 보호 규칙을 축적했다.

각 규칙은 현재 동작하지만 상태와 정책의 소유자가 코드 구조상 분명하지 않다. 한 영역을
수정할 때 다른 영역의 캐시·락·색인 실행·로그 동작까지 함께 이해해야 하므로, 변경 검토와
회귀 원인 추적 비용이 계속 커진다.

## Problem - 문제

- 디스크 색인 상태와 색인 실행 정책이 같은 범위에 섞여 있어 캐시, 락, 마이그레이션,
  single-flight의 책임을 구분하기 어렵다.
- 검색·캡처·RAG가 공통 인프라와 서로의 내부 상태에 가까이 붙어 있어 독립 테스트와 교체가
  어렵다.
- 공개 함수가 많은 단일 모듈이 사실상 API이면서 구현체 역할까지 맡아, 내부 구조를 바꾸는
  작업이 사용자 데이터와 MCP 동작에 영향을 줄 위험이 있다.
- 기존의 데이터 보호 규칙은 여러 과거 스펙과 테스트에 분산되어 있어, 분해 작업자가 반드시
  보존해야 할 계약을 한곳에서 추적하기 어렵다.

## Objective - 목표

Brain 기능을 명확한 책임과 단방향 의존성을 가진 도메인 경계로 분해하되, 설치한 사용자가
관찰하는 동작과 저장 데이터는 그대로 유지한다. 기존 공개 모듈은 호환 facade이자 조립 지점으로
남겨 호출자 변경 없이 내부 책임만 이동한다.

## Expected outcome - 기대 결과

- 색인 저장 상태와 색인 실행 정책의 소유자가 분명해진다.
- 검색, 캡처, RAG, 질의 기록을 각각 독립적으로 테스트할 수 있다.
- 공개 API와 색인 포맷은 유지되며 기존 노트나 벡터를 다시 만들 필요가 없다.
- 후속 기능 변경은 관련 경계 안에서 검토할 수 있고, 데이터 보호 회귀는 특성 테스트에서 즉시
  드러난다.

## Success metrics - 성공 지표

- [ ] 기존 Brain 공개 export와 함수 signature가 100% 유지된다.
- [ ] v4→v5 이행, v5 sidecar, 다중 프로세스 병합, 프루닝·재바인딩, 캡처 충돌,
  watcher·삭제, 질의 로그 계약이 모두 회귀 테스트로 고정되고 green이다.
- [ ] 색인 영속 상태와 색인 실행 상태의 소유자가 중복되지 않고, 신규 내부 모듈 간 순환
  의존성이 0개다.
- [ ] 분해 전후 동일 fixture에서 색인 JSON·sidecar 의미, 검색 결과, 캡처 결과, RAG 결과,
  MCP에 전달되는 출력이 동일하다.
- [ ] 분해 과정에서 전체 재색인 또는 임베딩 재생성 호출이 0회임을 테스트로 확인한다.
- [ ] AGENTS.md의 적대적 self-review에서 치명·중대 결함 0, 전체 테스트 green을 확인한다.

## Non-goals - 비목표

- 검색 점수 계산, 임베딩, 정렬, 필터링 또는 RAG 프롬프트를 개선하지 않는다.
- 색인 v5 JSON·sidecar 형식, 마이그레이션 정책 또는 락 프로토콜을 변경하지 않는다.
- 환경변수 이름·기본값·평가 시점, MCP/API 응답, 사용자 메시지·로그 문구를 변경하지 않는다.
- 캡처 검증, 태그 추천, 노트 삭제·휴지통 정책을 변경하지 않는다.
- 성능 최적화, 신규 캐시, 신규 외부 의존성 또는 데이터 마이그레이션을 도입하지 않는다.
- 041의 검색 품질·로그 의미론을 재설계하지 않는다. 041에서 확정된 계약을 소비만 한다.

## Constraints - 제약

- `specs/041-retrieval-quality-contract` 구현과 self-review가 clean으로 닫힌 뒤 착수한다.
- 동작 보존 리팩터다. 분해 전후의 관찰 가능한 차이는 결함으로 취급한다.
- 기존 색인과 sidecar를 그대로 읽고 써야 하며 재임베딩을 요구해서는 안 된다.
- 공개 facade를 우회하도록 기존 호출자를 일괄 수정하지 않는다.
- 테스트를 먼저 보강하고 작은 책임 단위로 이동하며, 각 단계에서 전체 회귀를 green으로 유지한다.
- 사용자 노트와 질의 원문을 fixture나 저장소에 복사하지 않는다.

## Stakeholders - 이해관계자

- localmind를 설치해 쓰는 단일 사용자 누구나(비개발자 포함)
- Brain 기능을 호출하는 MCP·API·웹 경계의 유지보수자
- 검색·캡처·RAG·색인 기능을 후속 개발하는 기여자

## Risks - 리스크

- 모듈 이동 중 초기화 순서가 바뀌어 환경변수 또는 singleton 상태의 의미가 달라질 수 있다.
- 캐시와 디스크 스냅샷의 소유권을 잘못 나누면 다중 프로세스 업데이트를 덮어쓸 수 있다.
- sidecar·마이그레이션 플래그를 분리하면서 무의도 재임베딩이나 데이터 손실이 발생할 수 있다.
- watcher와 명시적 재색인이 서로 다른 인스턴스를 사용하면 single-flight가 깨질 수 있다.
- facade가 내부 구현을 중복 보유하면 분해 후에도 두 개의 상태 정본이 생길 수 있다.
- 구조 테스트만 통과하고 사용자 시나리오 회귀를 놓칠 수 있으므로 기존 특성 테스트와 통합
  테스트를 함께 유지해야 한다.
