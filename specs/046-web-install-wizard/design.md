# Design: 웹 설치 위저드 (디자인 사전 정의)

> 디자인 게이트: 이 문서 확인 전 UI 구현 착수 금지(AGENTS.md "디자인·UI/UX 작업").
> **비개발자용**이라 시각·토큰뿐 아니라 **화면 문구·상태 전이·에러 카피**까지 정본으로 못 박는다.
> 토큰 정본은 034 `public/ui/style.css`(다크모드 포함) — 서버가 인라인으로 재사용한다.
> 이 위저드는 040 부트스트랩 가이드(scripts/bootstrap-guide.mjs)의 **능동 승격판**이다.
>
> **최우선 원칙 못 박기(AGENTS.md)**: 미적 완성도와 충돌하면 **상태 가시성·추적성이 이긴다.**
> 모든 단계·액션의 로딩/성공/실패를 명시적으로 표면화하고, 장시간 docker 대기는 절대 무음이 되지 않는다.

## 0. 자기완결 제약이 디자인을 어떻게 좌우하는가 (선결 정본)

이 위저드는 **설치 전에** Node 내장 http로 뜨고 강한 CSP 하에 동작한다. 그래서 아래가
디자인 토큰·컴포넌트 선택의 **하드 제약**이다(구현이 이걸 어기면 화면이 아예 안 뜬다):

- **외부 리소스 전면 금지**: CDN·웹폰트·원격 이미지·외부 스크립트 없음. CSP는
  `default-src 'none'; style-src 'unsafe-inline'; script-src 'unsafe-inline'; connect-src 'self'`
  수준을 목표(정확한 값은 plan에서 확정 — connect-src는 SSE/폴링 위해 self 필요).
- **모든 CSS/JS 인라인**: `<style>`·`<script>` 태그 내부. 외부 `.css`/`.js` 파일 링크 금지.
  (040은 `/style.css`를 서빙했지만 046은 CSP 강화를 위해 **인라인 권장** — plan에서 nonce vs 인라인 확정.)
- **아이콘·이미지는 텍스트/이모지/인라인 SVG data URI만**. 아이콘 폰트·PNG 스프라이트 금지.
- **프레임워크 없음(바닐라 JS)**. 상태관리 라이브러리·번들러 없음. `fetch`+`EventSource`(또는 청크 폴링)만.
- **폰트는 시스템 폰트 스택**(`system-ui`). 별도 폰트 로드 없음 → 034 `--font-*` 그대로.

→ 결과: **토큰은 CSS 변수 인라인 선언 1곳**, 컴포넌트는 **의미 클래스 + 바닐라 상태 토글**로 구성한다.

## 1. 화면 구성 (위저드 전체)

**단일 페이지 + 순차 스텝 진행**(SPA식 화면 전환, 페이지 리로드 없음). 상단에 스텝 진행 표시,
본문에 현재 스텝 카드, 각 스텝은 **확인 게이트 → 실행 → 진행 스트리밍 → 결과**로 닫힌다.

## 2. 디자인 시스템 패턴

- **패턴**: **단일 페이지 순차 위저드**(상단 StepProgress + 현재 스텝 카드 + 이전/다음). 각 스텝은
  자기완결(확인 게이트 → 실행 → 스트리밍 → 결과 pill). **다음 버튼은 현재 스텝이 성공으로 닫혀야 활성**.
- **근거**:
  - 능동 실행 위저드는 "한 번에 하나씩, 확인하고 실행"이 본질 — 순차 스텝이 이 멘탈모델과 1:1.
  - 상단 StepProgress가 **전체 위치 가시성**(내가 6개 중 어디)을 준다 — 비개발자 이탈 방지.
  - 각 스텝의 확인 게이트가 goal 제약("각 단계 사용자 확인")을 **UI 구조로 강제** — 실수로 풀오토 실행 불가.
  - 040(세로 스크롤 가이드)과 달리 046은 **긴 대기·실행 결과**가 스텝마다 있어, 스크롤 페이지면 진행 중인
    단계가 스크롤 밖으로 밀려 상태 가시성이 떨어진다 → 한 번에 한 스텝 focus가 유리.
- **기각한 대안**:
  - *멀티 화면(페이지 이동)* — CSP·무의존 하에 라우팅·상태 보존이 번거롭고, 새로고침 시 진행 유실.
    단일 페이지 + 클라이언트 스텝 전환이 더 견고.
  - *040식 세로 스크롤 전체 노출* — 실행·대기가 있는 능동 위저드엔 부적합(위 근거). 040은 안내판이라 맞았음.
  - *가로 탭 자유 이동* — 순서 의존(스택이 떠야 MCP 등록 가능)을 깨서 실패 경로가 늘어남. 순차 강제가 안전.

### 3.2 상태색 매핑 못 박기 (상태 가시성 핵심 — 정본)

이 위저드의 **모든 진행 상태**는 아래 5색 팔레트로만 표현한다(임의 색 금지). 색은 항상 **텍스트·아이콘과 병기**.

| 논리 상태 | 토큰 | pill 문구 예 | 아이콘(텍스트/이모지) |
|---|---|---|---|
| idle(초기·미실행) | `--color-idle` | "대기" · "확인 중…" | ○ / ◌ |
| loading(진행 중) | `--color-primary` | "진행 중" · "켜는 중…" | ◐ (회전 애니 — reduced-motion 시 정지) |
| success(성공) | `--color-ok` | "완료" · "됨" · "등록됨" | ✓ |
| warn(주의·대기·건너뜀) | `--color-warn` | "실행 필요" · "아직" · "건너뜀" | ! |
| error(실패) | `--color-error` | "실패" | ✗ |

> loading을 `--color-primary`로 둔 이유: idle(회색)과 명확히 구분되어 "지금 뭔가 돌고 있다"가 한눈에 보임.
> 무음 대기 방지의 시각적 핵심 — 진행 중은 회색이 아니라 **파란 살아있는 색 + 회전 아이콘 + 갱신되는 숫자**.

## 4. 컴포넌트 정의

> 각 컴포넌트: 목적 · 변형 · 상태 · **화면 상태 전이**(입력→상태→표시). API 선택·엔드포인트 계약은
> plan/아키텍트 소유 — 여기는 그 **결과의 표시와 전이**만 정의한다.

### 4.1 StepProgress (스텝 진행 표시)
- **목적**: 6개 스텝 중 현재 위치와 완료/대기를 상단에 항상 노출(전체 상태 가시성).
- **변형**: `wide`(데스크톱, 라벨+점) · `compact`(<480px, 점만 + "3/6" 텍스트).
- **스텝별 상태 표시**: done(✓·`--color-ok`) · current(●·`--color-primary`, 굵게) · upcoming(○·`--color-idle`) ·
  failed(✗·`--color-error`, 현재 스텝이 실패로 멈춤).
- **상태 전이**: 스텝 성공으로 닫힘 → 해당 점 done + 다음 점 current. 실패 → 현재 점 failed 유지(전진 불가).
- **접근성**: `aria-current="step"`, 각 점 `aria-label="3단계 스택 기동 — 진행 중"`.

### 4.2 StepCard (스텝 카드)
- **목적**: 현재 스텝의 제목·용어 설명·게이트·진행·결과를 담는 컨테이너.
- **구조**: 번호+제목(`--font-title`) → 용어 설명(TermHint) → 본문(게이트/입력/진행/결과) → StepNav.
- **변형**: 없음(단일). 카드 = `--color-surface` + border + `--radius-card` + `--shadow-card` + 패딩 `--space-lg`.
- **상태**: 스텝별 서브상태(아래 §5 전이도)를 자식 컴포넌트가 표현. 카드 자체는 컨테이너.

### 4.3 TermHint (용어 툴팁/설명)
- **목적**: Docker·MCP·토큰·임베딩 등 용어에 짧은 설명 병기(비개발자 필수).
- **변형**: `inline`(ⓘ 아이콘 + 한 줄 dim 설명, 기본) · `tooltip`(용어 옆 ⓘ 버튼 → 클릭/포커스 시 팝오버).
  → **기본은 inline**(호버 툴팁은 모바일·키보드에서 취약). tooltip은 보조.
- **상태**: tooltip 변형만 idle↔open. `<details>`/`<button aria-expanded>` 네이티브로 키보드 지원.
- **문구 정본**: §7 콘텐츠 표 참조.

### 4.4 PrereqBadge (준비물 배지 — 040 계승·확장)
- **목적**: Docker(설치/실행 구분)·Node·임베딩 엔진의 라이브 상태를 배지로.
- **변형**: 필수(missing 시 warn) · 선택(missing 시 idle "선택 · 없음", 예: ollama).
- **상태·표시**(`.pill` 재사용):
  | 상태 | pill | 문구 | 비고 |
  |---|---|---|---|
  | loading | idle | "확인 중…" | `/api/checks` 응답 전 |
  | 설치됨+실행중 | ok | "실행 중 ✓" | Docker: 설치+info 성공 |
  | 설치됨+미실행 | warn | "설치됨 · 앱 실행 필요" | **핵심 구분** — 조치: "Docker Desktop 열기" |
  | 미설치 | error/warn | "없음" | 조치: "받기" 링크 |
  | 선택·미설치 | idle | "선택 · 없음" | 진행 가능 |
  | unknown | idle | "확인 불가" | 점검 실패 → 수동 확인 안내 |
- **상태 전이**: 진입/새로고침 → 전부 "확인 중…" → 응답 도착 시 확정. "앱 실행 필요"·"없음"이면 **조치 버튼/링크 노출**.
  Docker "설치됨·앱 실행 필요"에서 사용자가 앱을 켠 뒤 [↻ 다시 확인] → 재폴링 → "실행 중 ✓"로 전이하면 다음 게이트 활성.

### 4.5 ConfirmGate (확인 게이트)
- **목적**: 능동 실행 전 사용자 확인 강제(goal 제약). 경고 카피 + 실행 버튼.
- **구조**: ⚠ 경고 카드(무엇을 실행·예상 소요·비파괴 안내) + 주 실행 버튼(`--color-primary`).
- **변형**: `standard`(스택 기동·MCP 등록) · `optional`(선택 단계 — "건너뛰기" 링크 병기, 예: 부 백엔드).
- **상태·전이**:
  | 상태 | 표시 | 전이 |
  |---|---|---|
  | idle | 경고 + [지금 켜기 ▶] 활성 | 준비물 충족 시 활성, 미충족 시 disabled + "먼저 준비물을 갖춰주세요" |
  | 확인됨(실행 시작) | 버튼 disabled(`cursor:wait`) + "켜는 중…" | 클릭 → ProgressStream 노출, 버튼 잠금(중복 실행 방지) |
  | 실행 완료 | 게이트 접힘/비활성, ResultPill 노출 | 성공/실패는 ProgressStream 결과가 결정 |
- **접근성**: 버튼 `<button>` + 명확한 라벨. 실행 중 `aria-busy="true"`.

### 4.6 ProgressStream (진행/로그 스트림 뷰) — 상태 가시성의 심장
- **목적**: 능동 실행의 표준 출력·진행률을 **실시간 스트리밍**으로 노출. 무음 대기 절대 금지.
- **구조 2단**:
  1. **요약 라인**(항상 보임): loading pill(◐ 회전) + 사람 말 진행 문구 + **진행 지표**.
     - 스택 기동: `준비 중… 34/120 (채팅✓ 메모리… AI엔진…)` — up.sh 헬스 폴링(seq 1..120)을 파싱해 표시.
     - 모델 내려받기 감지 시: `AI 모델 받는 중… (처음 한 번만 오래 걸려요)` — 정확한 % 없어도 "살아있음" 신호 유지.
  2. **자세한 로그**(`<details>` 접힘 기본): raw stdout `pre.mono`(`--log-max-height`, 자동 하단 스크롤).
- **변형**: `streaming`(실행 중) · `settled`(완료 후 로그 보존 — 디버깅/추적성).
- **상태·전이**:
  | 상태 | 요약 라인 | 로그 | 다음 버튼 |
  |---|---|---|---|
  | idle | 숨김 | 숨김 | 비활성 |
  | loading | ◐ "켜는 중… N/120" (갱신) | 스트리밍 추가 | 비활성 |
  | success | ✓ "완료 — 채팅·메모리·AI엔진 준비됨" | 접힘 보존 | **활성** |
  | error | ✗ "실패 — {사람말 원인}" + [다시 시도] | 자동 펼침(원인 노출) | 비활성, [다시 시도] 노출 |
- **무음 방지 규칙(못 박음)**: 스트림이 N초(예: 20s) 무갱신이면 요약 라인에 "아직 받는 중이에요 —
  멈춘 게 아니에요" 안내를 유지(heartbeat). 진행 지표는 최소 5초 주기로 시각적으로 변한다.
- **추적성**: 실패 시 로그는 지우지 않고 보존(재시도해도 이전 로그 접근 가능) — 디버깅 용이성 원칙.

### 4.7 SecretInput (토큰/키 입력 — 마스킹)
- **목적**: Gemini 등 API 키 입력. **원문 화면 노출·클라이언트 왕복 금지**(goal 제약).
- **구조**: 라벨 + TermHint + `type="password"` 입력 + [저장] + 상태 pill. 발급 안내(claude 토큰)는 별도 버튼.
- **변형**: `paste`(키 붙여넣기 — Gemini) · `oauth-guide`(claude 토큰 — 브라우저 흐름 안내, 입력 없음).
- **상태·전이**:
  | 상태 | 표시 | 전이 |
  |---|---|---|
  | empty | 빈 입력 + "설정 안 됨"(idle pill) | 기본 |
  | typing | 마스킹된 점(●●●) | `type=password` — 원문 미표시 |
  | 저장 중 | [저장] disabled + "저장 중…"(loading pill) | 클릭 → 서버로 전송(POST), 응답 대기 |
  | 저장됨 | 입력 비우고 "설정됨 ✓"(ok pill) + "다시 입력" 링크 | 서버가 .env 기록 성공 → **입력값 즉시 클리어, 원문 절대 되돌려받지 않음** |
  | 오류 | 입력 유지(값은 마스킹) + inline-error "형식이 올바르지 않아요 …" | 검증 실패 |
- **보안 못 박음**: 저장 성공 후 GET/응답에 원문 금지 — 서버는 **설정됨 여부(boolean)와 마스킹 힌트(예: 끝 4자)만** 반환.
  화면 어디에도 원문 표시 안 함. "다시 입력"은 새 값 받기이지 기존 값 조회 아님.
- **선택 스킵**: `optional` 항목은 [건너뛰기] → "건너뜀"(warn pill), 다음 진행 허용.

### 4.8 ResultPill (상태 pill — 로딩/성공/실패)
- **목적**: 각 액션 결과를 한 조각으로 압축 표시(§3.2 팔레트).
- **변형**: `inline`(문구 옆) · `block`(스텝 결과 강조).
- **상태**: idle/loading/success/warn/error(§3.2 표). **색+아이콘+텍스트 3중**(색만으로 전달 금지).
- **전이**: 부모 액션(게이트 실행·저장·폴링)의 상태를 그대로 반영.

### 4.9 McpVerifyPoller (MCP 등록 확인 — 폴링)
- **목적**: mcp-install 실행 후 "Claude Code 다시 켜기" 안내 + 등록 여부 자동 폴링(FR-4/AC-5).
- **구조**: 안내 카드("Claude Code를 **완전히 종료**했다 다시 여세요 — 왜: 새 도구는 재시작 후 인식돼요")
  + 폴링 상태 pill + [지금 확인] 수동 버튼.
- **상태·전이**:
  | 상태 | 표시 | 전이 |
  |---|---|---|
  | 등록 실행 중 | loading "등록 중…" | mcp-install.sh 스트리밍(ProgressStream 재사용) |
  | 대기(폴링) | warn "아직 — Claude Code를 다시 켜주세요" (주기 폴링) | 등록 감지 전, 안내 유지 |
  | 등록됨 | ok "등록됨 ✓ — 이제 Claude Code에서 localmind를 써요" | 폴링이 등록 확인 → 다음 활성 |
  | 실패 | error "등록 실패 — 기존 설정은 그대로예요" + [다시 시도] | 비파괴 재시도(기존 등록 보존) |
- **폴링 못 박음**: 무한 폴링 금지 — 시각적 heartbeat 유지하되, 일정 시도 후 "안 되면 이렇게" 수동 안내로 저하.

### 4.10 CompletionCard (완료 카드)
- **목적**: 설치 완료 축하 + "이제 이렇게 쓰세요" 다음 행동 안내.
- **구조**: 🎉 완료 헤드 + 준비된 것 요약(채팅·메모리·MCP 각 ok pill) + 다음 행동
  (첫 사용 튜토리얼 링크·`make health` 등 복사 chip) + [사용 시작] 주 버튼.
- **변형**: `full`(전 단계 성공) · `partial`(선택 단계 건너뜀 — 건너뛴 것과 "나중에 하는 법" 명시).
- **상태**: 정적(도달 시점에 확정). 링크·명령은 CopyChip(040 `.copy-btn` 계승) 재사용.

### 4.11 StepNav (이전/다음)
- **목적**: 스텝 이동. 다음은 **현재 스텝 성공 시에만 활성**(순차 강제).
- **상태·전이**: 다음 = 현재 스텝 success면 활성, 그 외 disabled + 이유 툴팁("먼저 이 단계를 완료해주세요").
  이전 = 항상 활성(되돌아보기 허용, 단 실행 중 액션은 잠금).

## 5. 화면 상태 전이도 (최소 흐름 6스텝)

각 스텝의 공통 서브상태: **초기 → (로딩) → 성공 | 실패 → 재시도**. 다음 활성 조건을 못 박는다.

```
① 환영
   [초기] 환영·무엇을 하는지 3줄 → [다음] 항상 활성 → ②

② 준비물 점검
   [로딩] 전부 "확인 중…"
     → /api/checks 응답
   [결과] Docker(설치/실행 구분)·Node·임베딩 배지 + 결핍별 조치
     · Docker 미실행 → "앱 실행 필요"(warn) + [Docker 열기]/[↻ 다시 확인]
     · 필수 미충족 → [다음] 비활성("필수 준비물을 먼저 갖춰주세요")
     · 필수 충족(선택 미충족 허용) → [다음] 활성 → ③
   [재시도] [↻ 다시 확인] → 로딩 재진입

③ 스택 기동 (능동 실행 + 스트리밍)
   [초기] ConfirmGate: ⚠경고("5~15분 걸릴 수 있어요·정상") + [지금 켜기 ▶]
     → 확인(클릭)
   [로딩] ProgressStream: ◐ "준비 중… N/120 (채팅 메모리 AI엔진)" + 접힌 로그
     (무음 방지 heartbeat, 모델 다운로드 문구)
   [성공] ✓ "완료 — 3개 준비됨" → [다음] 활성 → ④
   [실패] ✗ "실패 — {사람말 원인}" + 로그 자동 펼침 + [다시 시도]
     → 재시도는 비파괴(기존 상태 보존), ③ 초기로

④ 백엔드/토큰 (선택 — 스킵 가능)
   [초기] SecretInput들: claude 토큰(oauth-guide) · Gemini 키(paste, optional)
     · claude: [발급 안내] → 브라우저 흐름 안내 + 서버가 .env 기록 → "설정됨 ✓"
     · Gemini: 입력 → [저장] → 서버 기록 → "설정됨 ✓"(원문 미노출) / [건너뛰기]→"건너뜀"
   [성공/스킵] → [다음] 활성 → ⑤   (선택 단계라 미설정도 다음 허용)
   [오류] 입력 형식 오류 → inline-error, 값 유지(마스킹)

⑤ MCP 등록·확인 (능동 실행 + 폴링)
   [초기] ConfirmGate: "Claude Code에 localmind를 연결합니다" + [연결하기 ▶]
     → 확인
   [로딩] mcp-install 스트리밍(ProgressStream) → 등록 실행
   [대기] McpVerifyPoller: warn "아직 — Claude Code를 완전히 종료 후 다시 켜세요" + 폴링
   [성공] ok "등록됨 ✓" → [다음] 활성 → ⑥
   [실패] error "실패 — 기존 설정 그대로" + [다시 시도](비파괴)

⑥ 완료
   [최종] CompletionCard: 🎉 + 준비된 것 요약 + 다음 행동(튜토리얼·명령 복사) + [사용 시작]
     · partial: 건너뛴 항목과 "나중에 하는 법" 명시
```

**공통 전이 규칙(못 박음)**:
- 실행 중(로딩)에는 [다음]·[이전]의 파괴적 이동 잠금(중복 실행·경합 방지), 진행은 계속 스트리밍.
- 모든 실패는 **원인(사람 말) + 다음 조치(재시도/문서 링크)** 를 함께 표시 — 로그 코드만 던지지 않는다.
- 재시도는 항상 **비파괴**(기존 .env·데이터·MCP 등록 보존) — goal Constraint·AC-8.

## 6. 접근성 기본

- **색만으로 상태 전달 금지**: 모든 pill/배지는 **색 + 아이콘(✓/✗/!/◐/○) + 텍스트** 3중.
- **키보드 접근**: 모든 버튼·링크·입력·`<details>`는 네이티브 요소로 탭 순서·엔터/스페이스 기본 지원.
  포커스는 `outline: 2px solid --color-primary`(034 계승). 스텝 전환 시 새 스텝 제목에 포커스 이동.
- **상태 변화 알림**: 진행 요약 라인·pill 변화는 `aria-live="polite"`, 실패 카피는 `aria-live="assertive"`.
  실행 중 버튼 `aria-busy`, ConfirmGate 실행 버튼 `aria-disabled` 이유 제공.
- **대비**: 034 토큰이 라이트/다크 각각 WCAG AA 목표. loading 파란색 위 흰 글자 대비 확인(구현 시 검증).
- **모션**: `prefers-reduced-motion: reduce` 시 spin(◐ 회전)·전환 애니 정지 — 진행은 숫자 갱신으로 대체 표현.
- **StepProgress**: `aria-current="step"`, 각 스텝 상태를 `aria-label`에 텍스트로("3단계 스택 기동 — 진행 중").
